# 要件定義書: Harness Engineering 書籍

## Intent Analysis

### User Request
Harness Engineering（AIエージェントをポリシーベースで駆動させるための環境設計・ガードレール・フィードバックループの方法論）に関する書籍の企画書、目次、参照文献リストを作成する。

### Request Type
New Project（新規プロジェクト - 書籍制作の企画・設計フェーズ）

### Scope Estimate
Multiple Components（企画書、詳細目次、参照文献リスト、および関連資料）

### Complexity Estimate
Complex（急速に発展するAIエージェント分野の包括的な書籍企画。多数の一次ソース調査、体系的な構成設計、商業出版を見据えた品質が必要）

---

## 1. 機能要件

### FR-1: 書籍コンセプト・スコープ

#### FR-1.1: 主題定義
- **主題**: Harness Engineering - AIエージェントを自律的かつ信頼性高く動作させるための環境設計・ポリシー・ガードレール・フィードバックループを包括する方法論
- **スコープ**: Prompt Engineering → Context Engineering → Harness Engineering の進化系全体を俯瞰し、Harness Engineering を中心に据える包括的アプローチ
- **時代背景**: 2025年のAIエージェント元年から2026年のAgent Harness時代への転換点を捉える

#### FR-1.2: ターゲット読者（複層構造）
- **プライマリ**: AIエンジニア・MLエンジニア（エージェント開発の実務者）
- **セカンダリ**: ソフトウェアエンジニア全般（AI活用開発に移行したい開発者）
- **ターシャリ**: テックリード・アーキテクト（組織にAIエージェント基盤を導入したい意思決定者）

#### FR-1.3: 言語・出版形態
- **言語**: 日本語（英語の専門用語・原典は適宜併記）
- **出版形態**: 商業出版（出版社から刊行）
- **ページ数**: 内容に応じて柔軟に（包括的スコープ＋全テーマ網羅を考慮すると、300〜500ページ規模が想定される）

### FR-2: 主要テーマ（全網羅）

#### FR-2.1: エージェントハーネスのアーキテクチャ設計
- 6つの標準コンポーネント: Reasoning Engine, Planning & Orchestration, Tool Registry, Memory & Context, State & Persistence, Structured I/O
- Framework / Runtime / Harness の3層スタック構造
- モデル非依存設計の原則

#### FR-2.2: ポリシー・ガードレール設計
- AGENTS.md / CLAUDE.md 等のマップファイル設計
- Golden Principles（不変ルール）の策定手法
- ルールファイル体系（二層構造: ベースレイヤー + 拡張レイヤー）
- 品質ドキュメント・設計ドキュメントの検証ステータス管理

#### FR-2.3: 長期実行エージェントの運用パターン
- セッション継続性と進捗ブリッジング
- コンテキストウィンドウの限界とメモリ管理戦略
- 自動リファクタリング・継続的クリーンアップ
- 増分開発（Incremental Development）パターン

#### FR-2.4: 組織導入・チーム運用
- Agent-first開発文化への組織移行
- PRフロー変革（短命PR、最小限のブロッキングゲート）
- 人間とエージェントの役割分担（「Humans Steer, Agents Execute」）
- チーム生産性指標の再定義

### FR-3: プラットフォーム・ツール（マルチプラットフォーム横断）

#### FR-3.1: カバーすべきエコシステム
- **OpenAI**: Codex, ChatGPT, Agents SDK, App Server
- **Anthropic**: Claude Code, Claude Agent SDK, Model Context Protocol (MCP)
- **LangChain**: LangGraph, DeepAgents
- **Salesforce**: Agentforce
- **Google**: ADK (Agent Development Kit)
- **その他**: CrewAI, Vercel AI SDK 等

#### FR-3.2: 比較・分析の観点
- 各プラットフォームのハーネスアーキテクチャ比較
- 強み・弱み・適用シーン
- 相互運用性とベンダーロックイン回避戦略

### FR-4: 差別化・独自価値

#### FR-4.1: AI-DLCフレームワークの体系化
- 本リポジトリで実践しているAI-DLC（AI-Driven Development Life Cycle）をHarness Engineeringの実践例として体系的に紹介
- INCEPTION → CONSTRUCTION → OPERATIONS の3フェーズワークフロー
- ルール詳細の二層構造（ベースレイヤー＋拡張レイヤー）の設計パターン

#### FR-4.2: 差別化提案（AI分析に基づく）
以下の差別化要素を提案する:
- **日本語圏初の体系的Harness Engineering解説書**: 英語圏のブログ・記事は増加中だが、日本語での包括的書籍は皆無
- **実践フレームワーク付き**: 理論だけでなく、読者がすぐに導入できるテンプレート・チェックリスト・設計パターンを提供
- **進化の文脈化**: Prompt Engineering → Context Engineering → Harness Engineeringの進化を歴史的・技術的に体系化し、「なぜ今Harnessなのか」を明確に示す
- **複数プラットフォーム横断比較**: 特定ベンダーに偏らない中立的な比較分析

### FR-5: 成果物要件

#### FR-5.1: 企画書
- 書籍概要（仮タイトル、サブタイトル）
- 企画趣旨・市場背景
- ターゲット読者とペルソナ
- 差別化ポイント
- 章構成概要
- 想定ページ数・スケジュール感
- 著者プロフィール欄（テンプレート）
- 類書分析

#### FR-5.2: 詳細目次
- パート構成（大分類）
- 章構成（中分類）
- 節・項構成（小分類）
- 各章の概要説明（2〜3行）
- 各章の想定ページ数配分

#### FR-5.3: 参照文献リスト
- 一次ソース（OpenAI, Anthropic, LangChain等の公式ブログ・ドキュメント）
- 学術論文・研究ペーパー（arXiv等）
- 実践事例・業界記事（Medium, Substack, 技術ブログ等）
- 関連書籍
- 各文献にカテゴリ・関連章・重要度を付記

## 2. 非機能要件

### NFR-1: 品質基準
- 商業出版に耐えうる正確性・信頼性
- 一次ソースに基づく情報の裏付け
- 技術的な正確性と最新性（2026年2月時点）

### NFR-2: 成果物フォーマット
- すべてMarkdown形式で作成
- 図表はMermaid記法またはASCIIダイアグラムで表現
- 出版社提出時にWord/PDF変換可能な構造

### NFR-3: 活用目的
- 執筆開始前の全体設計・ロードマップとして使用
- 著者自身が執筆の指針として参照する完成度

## 3. Key Stakeholders

| ステークホルダー | 役割 | 関心事項 |
|---|---|---|
| 著者（ユーザー） | 書籍の著者・企画者 | 書籍の方向性、差別化、市場性 |

## 4. Success Criteria

- [ ] 企画書が商業出版の提案に使える完成度である
- [ ] 目次が書籍全体の構成を明確に示し、執筆の指針となる
- [ ] 参照文献リストが主要な一次ソースを網羅している
- [ ] Harness Engineeringの概念が正確かつ体系的に整理されている
- [ ] AI-DLCフレームワークが独自の差別化要素として位置付けられている

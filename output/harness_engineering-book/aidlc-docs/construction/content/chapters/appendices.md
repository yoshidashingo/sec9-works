## 付録A: AGENTS.md テンプレート集

> **本付録の目的**: 読者がすぐに自分のプロジェクトに適用できるAGENTS.mdテンプレートを提供する。AGENTS.md仕様（STD-02）およびOpenAI Codexのガイドライン（OAI-09）に準拠し、プロジェクトタイプやチーム規模に応じたバリエーションを揃えた。各テンプレートはそのままコピーして使用できる。

### A.1 基本テンプレート（すべてのプロジェクト共通）

すべてのプロジェクトに適用可能な最小構成のテンプレートである。AGENTS.md仕様（STD-02）が推奨するセクション構成に従い、OpenAI Codexのガイドライン（OAI-09）で示された「プロジェクト概要」「ビルド・テストコマンド」「コードスタイル」の3つの必須セクションを含む。

```markdown
# AGENTS.md

## Project Overview
<!-- プロジェクトの概要を1-3文で記述する -->
このプロジェクトは[プロジェクト名]である。[主要な機能・目的を記述する]。

## Tech Stack
- Language: [言語名] [バージョン]
- Framework: [フレームワーク名] [バージョン]
- Database: [DB名] [バージョン]（該当する場合）
- Package Manager: [パッケージマネージャ名]

## Directory Structure
```
src/           # アプリケーションコード
tests/         # テストコード
docs/          # ドキュメント
scripts/       # ビルド・デプロイスクリプト
```

## Build & Test Commands
- Install dependencies: `[インストールコマンド]`
- Build: `[ビルドコマンド]`
- Run tests: `[テストコマンド]`
- Run single test: `[単一テスト実行コマンド]`
- Lint: `[リントコマンド]`
- Format: `[フォーマットコマンド]`

## Code Style & Conventions
- [コーディング規約1: 例）関数名はcamelCaseを使用する]
- [コーディング規約2: 例）エラーハンドリングはtry-catchで統一する]
- [コーディング規約3: 例）コメントは英語で記述する]

## Golden Rules
<!-- 絶対に守るべきルール。機械的で意見のあるルールを記述する -->
1. テストなしのコードはコミットしない
2. 既存のコードスタイルを尊重する
3. 破壊的変更を加える前に確認を求める
4. `.env` ファイルやシークレットをコミットしない

## Context & References
- Architecture docs: `docs/architecture.md`
- API specification: `docs/api-spec.md`
- Contributing guide: `docs/CONTRIBUTING.md`
```

**使用上のポイント**:
- `Project Overview` は短く簡潔に。エージェントが最初に読むセクションであるため、プロジェクトの本質を1-3文で伝えることが重要である
- `Build & Test Commands` は正確なコマンドを記述する。エージェントはこのコマンドを文字通り実行するため、曖昧な記述は避ける
- `Golden Rules` は5つ以下に絞る。ルールが多すぎるとエージェントのコンテキストを圧迫し、遵守率が低下する

### A.2 プロジェクトタイプ別テンプレート

#### A.2.1 Webアプリケーション用テンプレート

フロントエンド・バックエンド・データベースを含むWebアプリケーションプロジェクト向けである。

```markdown
# AGENTS.md

## Project Overview
[プロジェクト名]は[目的]のためのWebアプリケーションである。
フロントエンドはReact/Next.js、バックエンドはNode.js/Express、
データベースはPostgreSQLを使用する。

## Tech Stack
- Frontend: React 19 + Next.js 15 (TypeScript)
- Backend: Node.js 22 + Express 5
- Database: PostgreSQL 17
- ORM: Prisma 6
- Authentication: NextAuth.js v5
- Package Manager: pnpm

## Directory Structure
```
app/               # Next.js App Router ページ・レイアウト
components/        # 再利用可能なUIコンポーネント
  ui/              # 基本UIコンポーネント（Button, Input等）
  features/        # 機能固有のコンポーネント
lib/               # ユーティリティ・ヘルパー関数
  db/              # データベース接続・クエリ
  auth/            # 認証ロジック
  api/             # API クライアント
server/            # バックエンドAPI
  routes/          # APIルート定義
  middleware/      # Express ミドルウェア
  services/        # ビジネスロジック
prisma/            # Prisma スキーマ・マイグレーション
tests/             # テストコード
  unit/            # ユニットテスト
  integration/     # 統合テスト
  e2e/             # E2Eテスト（Playwright）
public/            # 静的アセット
docs/              # ドキュメント
```

## Build & Test Commands
- Install: `pnpm install`
- Dev server: `pnpm dev`
- Build: `pnpm build`
- Unit tests: `pnpm test`
- Single test: `pnpm test -- --testPathPattern="[ファイル名]"`
- E2E tests: `pnpm test:e2e`
- Lint: `pnpm lint`
- Format: `pnpm format`
- Type check: `pnpm typecheck`
- DB migrate: `pnpm prisma migrate dev`
- DB seed: `pnpm prisma db seed`

## Code Style & Conventions
- コンポーネントは関数コンポーネント + React Hooks で実装する
- Server Components をデフォルトとし、インタラクションが必要な場合のみ
  Client Components（'use client'）を使用する
- API レスポンスは必ず型定義（TypeScript interface）を作成する
- エラーハンドリングは Error Boundary + try-catch で統一する
- CSS は Tailwind CSS のユーティリティクラスを使用する
- ファイル名はkebab-case、コンポーネント名はPascalCaseとする

## Golden Rules
1. Server Components / Client Components の境界を意識する
2. データベーススキーマ変更時は必ずマイグレーションファイルを作成する
3. APIエンドポイントには必ずバリデーション（zod）を適用する
4. 環境変数は `.env.local` に定義し、コミットしない
5. 新しいページ追加時は対応するE2Eテストも作成する

## API Design Guidelines
- RESTful 規約に従う（GET/POST/PUT/DELETE）
- エンドポイントは `/api/v1/[resource]` 形式
- レスポンスは `{ data, error, meta }` 形式で統一する
- ページネーションは cursor-based を標準とする

## Context & References
- Architecture: `docs/architecture.md`
- API Spec: `docs/api-spec.md`
- DB Schema: `prisma/schema.prisma`
- Component Library: `docs/components.md`
```

#### A.2.2 モバイルアプリケーション用テンプレート

React NativeまたはFlutterベースのモバイルアプリケーション向けである。

```markdown
# AGENTS.md

## Project Overview
[プロジェクト名]は[目的]のためのモバイルアプリケーションである。
React Nativeで実装し、iOS/Androidの両プラットフォームに対応する。

## Tech Stack
- Framework: React Native 0.76 (TypeScript)
- Navigation: React Navigation v7
- State Management: Zustand v5
- API Client: TanStack Query v5
- Testing: Jest + React Native Testing Library
- E2E: Detox
- Package Manager: yarn

## Directory Structure
```
src/
  screens/         # 画面コンポーネント
  components/      # 再利用可能コンポーネント
    shared/        # 共通コンポーネント
    [feature]/     # 機能別コンポーネント
  navigation/      # ナビゲーション定義
  hooks/           # カスタムフック
  stores/          # Zustand ストア
  services/        # API通信・外部サービス連携
  utils/           # ユーティリティ関数
  types/           # 型定義
  assets/          # 画像・フォント等
  constants/       # 定数定義
ios/               # iOS ネイティブコード
android/           # Android ネイティブコード
__tests__/         # テストコード
e2e/               # E2Eテスト（Detox）
docs/              # ドキュメント
```

## Build & Test Commands
- Install: `yarn install`
- iOS dev: `yarn ios`
- Android dev: `yarn android`
- Unit tests: `yarn test`
- Single test: `yarn test --testPathPattern="[ファイル名]"`
- E2E (iOS): `yarn e2e:ios`
- E2E (Android): `yarn e2e:android`
- Lint: `yarn lint`
- Type check: `yarn typecheck`
- Pod install: `cd ios && pod install`

## Code Style & Conventions
- 画面コンポーネントは `[ScreenName]Screen.tsx` 形式で命名する
- ビジネスロジックはカスタムフック（`use[Feature].ts`）に分離する
- プラットフォーム固有コードは `*.ios.tsx` / `*.android.tsx` で分離する
- スタイルは StyleSheet.create で定義し、インラインスタイルは避ける
- ナビゲーションパラメータには必ず型を定義する

## Golden Rules
1. ネイティブモジュールの変更後は必ずビルドを確認する
2. 新しい画面追加時はナビゲーション型定義を更新する
3. iOS/Android両方でのレンダリングを確認してからPRを出す
4. パフォーマンスに影響するレンダリングには `React.memo` を適用する
5. サードパーティライブラリ追加前にバンドルサイズへの影響を確認する

## Platform-Specific Notes
- iOS: 最小対応バージョン iOS 16.0
- Android: minSdkVersion 26 (Android 8.0)
- ProGuardルール変更時は `android/app/proguard-rules.pro` を更新

## Context & References
- Architecture: `docs/architecture.md`
- Navigation Map: `docs/navigation.md`
- Design System: `docs/design-system.md`
```

#### A.2.3 データパイプライン用テンプレート

ETL/データ処理パイプラインプロジェクト向けである。

```markdown
# AGENTS.md

## Project Overview
[プロジェクト名]は[データソース]から[目的]のためにデータを収集・変換・格納する
データパイプラインである。日次バッチ処理とリアルタイムストリーム処理の
両方をサポートする。

## Tech Stack
- Language: Python 3.12
- Orchestration: Apache Airflow 2.10 / Prefect 3
- Processing: Apache Spark 3.5 / Polars 1.x
- Streaming: Apache Kafka 3.8
- Storage: Amazon S3 + Delta Lake
- Database: Amazon Redshift / BigQuery
- Infrastructure: Terraform + Docker
- Package Manager: uv

## Directory Structure
```
dags/              # Airflow DAG定義
pipelines/         # パイプラインコアロジック
  extractors/      # データ抽出モジュール
  transformers/    # データ変換モジュール
  loaders/         # データ格納モジュール
  validators/      # データ品質検証
models/            # データモデル定義（Pydantic）
schemas/           # スキーマ定義（Avro/JSON Schema）
config/            # 環境別設定ファイル
scripts/           # ユーティリティスクリプト
tests/             # テストコード
  unit/            # ユニットテスト
  integration/     # 統合テスト
  fixtures/        # テストデータ
infra/             # Terraform / Docker 定義
docs/              # ドキュメント
  data-dictionary/ # データ辞書
  lineage/         # データリネージ図
```

## Build & Test Commands
- Install: `uv sync`
- Unit tests: `uv run pytest tests/unit/`
- Integration tests: `uv run pytest tests/integration/`
- Single test: `uv run pytest tests/unit/[ファイル名] -v`
- Lint: `uv run ruff check .`
- Format: `uv run ruff format .`
- Type check: `uv run mypy pipelines/`
- DAG validation: `uv run airflow dags test [dag_id] [date]`
- Local Airflow: `docker compose up airflow`

## Code Style & Conventions
- ETLの各ステージ（Extract/Transform/Load）は独立したモジュールとして実装する
- データスキーマは Pydantic モデルで厳密に定義する
- SQL クエリは `queries/` ディレクトリに `.sql` ファイルとして管理する
- パイプラインの冪等性を保証する（再実行しても同じ結果になること）
- ログは構造化ログ（structlog）で出力する

## Golden Rules
1. 本番データに対する書き込み操作は必ずドライランモードで先に検証する
2. スキーマ変更時はマイグレーションスクリプトとロールバック手順を必ず用意する
3. 新しいデータソース追加時はデータ辞書（`docs/data-dictionary/`）を更新する
4. パイプライン失敗時のリカバリ手順を DAG のドキュメントに記載する
5. PII（個人識別情報）を含むカラムには必ずマスキング処理を適用する

## Data Quality Rules
- Null許容カラムを明示的に定義する
- 数値カラムには妥当な範囲チェックを設定する
- 重複チェックはパイプラインの最終段階で実施する
- データ品質メトリクスは Great Expectations で管理する

## Context & References
- Architecture: `docs/architecture.md`
- Data Dictionary: `docs/data-dictionary/`
- Data Lineage: `docs/lineage/`
- Runbook: `docs/runbook.md`
```

```markdown
# AGENTS.md

## Project Overview
[1-2文でプロジェクトの目的を記述する]

## Quick Reference
- Language: [言語] [バージョン]
- Build: `[ビルドコマンド]`
- Test: `[テストコマンド]`
- Lint: `[リントコマンド]`

## Structure
[主要ディレクトリのみ3-5行で記述]

## Rules
1. テストを書いてからコミットする
2. 既存のパターンに従う
3. 不明点があれば質問する

## Notes
- [プロジェクト固有の注意事項を1-3個記述する]
```

**設計意図**: 個人開発では、チームの合意形成やスタイル統一のためのルールは不要である。エージェントに必要なのは「何を作っているか」「どうビルド・テストするか」「最低限のルール」のみである。テンプレートを肥大化させるよりも、プロジェクトの実態に合わせて漸進的に追加していく方が効果的である（TLDV-03）。

#### A.3.3 大規模組織（10名以上）向けテンプレート

複数チームが関わるモノレポや大規模プロジェクトにおいて、AGENTS.mdの階層構造を活用するテンプレートである。AGENTS.md仕様（STD-02）の階層マージ機能を前提とする。

**ルートAGENTS.md（リポジトリ全体のポリシー）**:

```markdown
# AGENTS.md (Root)

## Organization Policy
このリポジトリは[組織名]のモノレポであり、複数チームが共同で開発を行う。
すべてのエージェントは以下の組織ポリシーに従うこと。

## Monorepo Structure
```
packages/
  frontend/        # フロントエンドアプリ（Team A担当）
  backend/         # バックエンドAPI（Team B担当）
  shared/          # 共有ライブラリ（全チーム共通）
  infra/           # インフラ定義（Platform Team担当）
docs/              # 組織横断ドキュメント
tools/             # 共有ツール・スクリプト
```

## Global Commands
- Install (all): `pnpm install`
- Build (all): `pnpm -r build`
- Test (all): `pnpm -r test`
- Lint (all): `pnpm -r lint`

## Organization-Wide Rules
1. パッケージ間の依存関係は `shared/` を経由する。直接の相互依存を作らない
2. セキュリティ脆弱性のあるパッケージは48時間以内に更新する
3. PII（個人識別情報）を含むログ出力を禁止する
4. すべてのAPIエンドポイントに認証を適用する
5. 他チームのパッケージを変更する場合は該当チームのレビューを必須とする

## Security & Compliance
- シークレット管理: AWS Secrets Manager / HashiCorp Vault を使用
- コミットに含めてはならないファイル: `.env*`, `*credentials*`, `*.pem`, `*.key`
- 依存関係の脆弱性スキャン: CI/CDで自動実行
- ライセンスチェック: 新規依存追加時に自動検証

## Cross-Team Communication
- アーキテクチャ変更提案: `docs/rfc/` にRFCを作成
- 共有ライブラリ変更: `#shared-lib` チャンネルで事前通知
- インフラ変更: Platform Teamのレビュー必須

## Per-Package Configuration
各パッケージのディレクトリに配置された AGENTS.md が、
そのパッケージ固有のルールを定義する。
ルートのポリシーは全パッケージに適用される。
```

**パッケージ単位のAGENTS.md（例: `packages/backend/AGENTS.md`）**:

```markdown
# AGENTS.md (packages/backend)

## Package Overview
バックエンドAPIサービス。RESTful APIとGraphQLの両方を提供する。
Team Bが担当する。

## Tech Stack
- Runtime: Node.js 22
- Framework: Fastify 5
- GraphQL: Mercurius
- ORM: Drizzle ORM
- Test: Vitest

## Commands (Package-Specific)
- Dev: `pnpm dev`
- Test: `pnpm test`
- Test (watch): `pnpm test:watch`
- DB migrate: `pnpm db:migrate`
- Generate types: `pnpm codegen`

## Package-Specific Rules
1. APIレスポンスは必ず `ApiResponse<T>` 型でラップする
2. DB操作はすべてトランザクション内で行う
3. 外部API呼び出しにはリトライロジック（指数バックオフ）を適用する
4. ログレベルは `debug/info/warn/error` の4段階で統一する

## Team Contact
- Team Lead: [名前]
- Slack: #team-backend
- On-call: [ローテーション情報]
```

**設計意図**: 大規模組織では、ルートのAGENTS.mdで組織共通ポリシーを定義し、各パッケージのAGENTS.mdでチーム固有のルールを追加する。エージェントは、対象ファイルのパスに基づいてルートとパッケージの両方のAGENTS.mdを読み込み、統合して適用する。これは本書第4章で解説した「ルールファイル体系の二層構造」（4.3節）と同じ設計原則に基づく。

## 付録B: ハーネス設計チェックリスト

> **本付録の目的**: ハーネスの設計・導入・評価の各段階で漏れなく確認すべき項目を、チェックリスト形式で提供する。新規プロジェクト、既存プロジェクトへの導入、品質評価の3つのシナリオに対応する。

### B.1 新規プロジェクト用チェックリスト

新規プロジェクトでエージェントハーネスを構築する際の3フェーズチェックリストである。

#### フェーズ1: ハーネス設計

| # | チェック項目 | 確認 |
|---|------------|------|
| H-01 | プロジェクトの目的・スコープを定義したか | [ ] |
| H-02 | 使用するエージェントプラットフォーム（Codex / Claude Code / DeepAgents等）を選定したか | [ ] |
| H-03 | エージェントハーネスの6コンポーネント（第3章）のうち、必要なものを特定したか | [ ] |
| H-04 | AGENTS.md または CLAUDE.md を作成し、プロジェクト概要・ビルドコマンド・コード規約を記述したか | [ ] |
| H-05 | ディレクトリ構造をドキュメント化したか | [ ] |
| H-06 | エージェントが使用するツール群を定義したか | [ ] |

            approve_actions=["file_write", "terminal"]

        SummarizationMiddleware(
            trigger_threshold=0.85  # コンテキスト使用率85%で発動

        LoopDetectionMiddleware(
            max_same_file_edits=5   # 同一ファイル5回編集で警告

    ],
)

## 付録D: 用語集

> **本付録の目的**: 本書で使用する主要な技術用語を50音順（日本語）で定義する。各項目は「用語名（英語名）」「定義」「関連章」の形式で記載する。

---

**AGENTS.md**
AIコーディングエージェントに対してプロジェクト固有の指示を与えるためのオープンフォーマット仕様ファイルである。プロジェクト概要、ビルドコマンド、コードスタイル規約、ガードレールなどを記述する。Agentic AI Foundation（AAIF）の設立プロジェクトのひとつとして標準化され、6万以上のOSSプロジェクトが採用している（STD-02, OAI-09）。
**関連章**: 第4章, 付録A

**CLASSic次元** (CLASSic Dimensions)
エージェンティックAIの評価フレームワークであり、Cost（コスト）、Latency（レイテンシ）、Accuracy（正確性）、Security（セキュリティ）、Stability（安定性）の5次元で構成される。アーキテクチャ選択と具体的な障害モードの対応付けを可能にする体系的な評価手法である（ACD-04）。
**関連章**: 第7章, 付録B

**CLAUDE.md**
Anthropicが提供するClaude Codeにおいて、プロジェクト固有のコンテキストとルールを定義するための設定ファイルである。AGENTS.mdと同等の役割を果たし、プロジェクト概要、ビルドコマンド、コーディング規約、ガードレールなどを記述する。Claude Code起動時に自動的に読み込まれ、エージェントの動作を規定する（ANT-13）。
**関連章**: 第4章, 付録A, 付録C

**Deep Agents** (ディープエージェンツ)
LangChainが開発したエージェントハーネス製品であり、Framework（LangChain）→ Runtime（LangGraph）→ Harness（DeepAgents）の3層スタックの最上位に位置する。計画ツール、サブエージェント、ファイルシステムアクセス、自動要約・コンパクションの4つの組み込み機能を提供する「バッテリー込み」のハーネスである（LC-01, LC-02, TLDV-01）。
**関連章**: 第1章, 第3章, 第5章

**Framework / Runtime / Harness** (フレームワーク / ランタイム / ハーネス)
Harrison Chase氏（LangChain CEO）が提唱したエージェント開発スタックの3層分類である。Frameworkはモデルとの対話・ツール定義・外部サービス統合の抽象化層、Runtimeは状態管理・耐久実行・チェックポイントの実行基盤層、Harnessは計画・サブエージェント・ファイルシステム・コンパクションを備えた最上位層である（LC-01）。
**関連章**: 第1章, 第3章

**Human-in-the-Loop** (ヒューマン・イン・ザ・ループ)
エージェントの実行フローの中に人間の確認・承認ポイントを組み込む設計パターンである。高リスクな操作（ファイル書き込み、外部API呼び出し、デプロイ等）の前に人間の判断を介在させることで、エージェントの自律性と安全性のバランスを取る。Middlewareパターンで実装されることが多い（TLDV-02）。
**関連章**: 第3章, 第4章, 第6章

**LoopDetectionMiddleware** (ループ検出ミドルウェア)
エージェントが同一ファイルをN回以上編集するなどのループ状態を検出し、アプローチの変更を促すコンテキストを注入するミドルウェアである。破滅ループ（Doom Loop）を防止するための実装パターンとして、LangChainのTerminal Bench 2.0実験で有効性が実証された（LC-08）。
**関連章**: 第6章

**MCP** (Model Context Protocol)
Anthropicが開発し、オープンソースとして公開したAIアプリケーションの標準接続プロトコルである。JSON-RPCベースのクライアント・サーバー通信により、AIモデルと外部データソース・ツール・サービスを標準化された方法で接続する。「AIアプリケーションのUSB-C」と形容される（ANT-15, STD-03）。
**関連章**: 第5章

**SKILL.md** (スキルファイル)
DeepAgentsおよびClaude Codeで使用される、エージェントの能力を段階的に開示するための設定ファイルである。YAML前置き（メタデータ）と本文（詳細手順）の2層構造を持ち、デフォルトではYAML前置きのみがロードされ、エージェントが必要と判断した場合に完全な内容が読み込まれる。プログレッシブ・ディスクロージャーの原則に基づく設計である（LC-15, ANT-10）。
**関連章**: 第4章, 第5章, 付録C

**Terminal Bench** (ターミナルベンチ)
LangChainチームが開発したコーディングエージェントのベンチマークである。バージョン2.0は、ソフトウェアエンジニアリング、バイオロジー、セキュリティ、ゲーミングの4領域にまたがる89タスクで構成され、各タスクは数分から10分程度の作業時間と100回以上のツールコールを要求する実践的な評価基盤である（LC-16）。
**関連章**: 第1章, 第7章

**エージェントエンジニアリング** (Agent Engineering)
非決定論的なLLMシステムを信頼性の高い本番体験へと洗練させる反復的プロセスである。Product thinking（プロダクト思考）、Engineering（エンジニアリング）、Data science（データサイエンス）の3つの必須スキルセットが求められる。build → test → ship → observe → refine のサイクルを回すことで、エージェントの品質を漸進的に向上させる（LC-13）。
**関連章**: 第1章, 第8章

**エージェントハーネス** (Agent Harness)
AIモデルを中核に据えながら、そのモデルが長期実行タスクを信頼性高く遂行するために必要な周辺インフラストラクチャの総称である。プロンプトプリセット、ツールコールの管理、ライフサイクルフック、計画機能、サブエージェント管理など、モデルの「外側」で提供される仕組みの集合体を指す。Harrison Chase氏の3層分類ではFrameworkとRuntimeの上に位置する最上位層である（LC-01, BLG-01）。
**関連章**: 第1章, 第3章

**ガードレール** (Guardrails)
エージェントの動作を安全な範囲内に制限するための機構の総称である。入力ガードレール（ユーザー入力の検証）、出力ガードレール（生成結果の検証）、実行ガードレール（ファイルシステム分離、ネットワーク分離）の3種類に大別される。Constitutional AI（ANT-17）の思想を実装に落とし込んだものである。
**関連章**: 第4章, 第7章

**コンテキストエンジニアリング** (Context Engineering)
エージェントのコンテキストウィンドウに何を・いつ・どのように配置するかを体系的に設計する技術分野である。プロンプトエンジニアリングが「言語による問いの最適化」であるのに対し、コンテキストエンジニアリングは「推論に必要な全体設計の最適化」を対象とする。Dex Horthy氏が2025年4月にこの概念を命名した。Write（外部永続化）、Select（必要時取得）、Compress（要約・トリミング）、Isolate（コンテキスト分離）の4つの操作戦略がある（LC-11, ANT-03）。
**関連章**: 第2章, 付録F

**コンテキストロット** (Context Rot)
長期実行エージェントにおいて、コンテキストウィンドウ内の情報が時間とともに劣化し、エージェントの判断精度が低下する現象である。古い情報が蓄積して新しい情報と矛盾する、重要な指示が他の情報に埋もれる、コンテキストの使用率が高まることで推論品質が劣化する、といった症状を呈する（ANT-03）。
**関連章**: 第2章, 第6章

**コンパクション** (Compaction)
長時間実行中に肥大化したコンテキストを要約・圧縮する技術の総称である。会話履歴の要約、大規模ツール結果のエビクション（退避）、不要メッセージの削除などを含む。OpenAIのGPT-5.1-Codex-Maxで初めてネイティブサポートされ、複数コンテキストウィンドウにまたがる作業を可能にした（OAI-05, LC-04）。
**関連章**: 第6章

**推論サンドイッチ** (Reasoning Sandwich)
エージェントの実行フェーズごとに推論コストの配分を最適化するパターンである。計画フェーズで高い推論コスト（extra-high）、実装フェーズで中程度の推論コスト（high）、検証フェーズで再び高い推論コスト（extra-high）を配分する。LangChainのTerminal Bench 2.0実験で、全フェーズで最大推論を使用するアプローチ（53.9%）よりもバランス型配分（63.6%）が優れた結果を示した（LC-08）。
**関連章**: 第1章, 第6章

**ハーネスエンジニアリング** (Harness Engineering)
モデルの周囲にシステムを構築し、タスクパフォーマンス、トークン効率、レイテンシ等の目標を最適化する技術分野である。プロンプトエンジニアリングとコンテキストエンジニアリングの上位概念として位置づけられ、ツール連携、状態管理、ガードレール、サブエージェント管理、評価基盤まで含む包括的なシステム設計を対象とする。2026年2月にOpenAIのRyan Lopopolo氏が発表した記事により広く普及した（OAI-01, LC-08）。
**関連章**: 第1章, 第2章, 全章

**プログレッシブ・ディスクロージャー** (Progressive Disclosure)
エージェントに対して情報を段階的に開示する設計原則である。初期ロード時にはメタデータ（YAML前置き）のみを読み込み、エージェントが必要と判断した時点で詳細情報をロードする。トークン効率を向上させると同時に、エージェントの認知負荷を低減する。SKILL.mdの設計で具体化されている（LC-15, ANT-10）。
**関連章**: 第4章, 第5章

**プロンプトエンジニアリング** (Prompt Engineering)
LLMに対する入力（プロンプト）を最適化し、望ましい出力を引き出す技術である。ハーネスエンジニアリングの基盤をなす技術であるが、単一の推論呼び出しの最適化に焦点を当てるため、長期実行タスクや複数セッションにまたがるエージェントの制御には限界がある。コンテキストエンジニアリング、ハーネスエンジニアリングへと発展的に拡張された（BLG-04）。
**関連章**: 第2章

**ミドルウェア** (Middleware)
エージェントの実行ループにおいて、モデルへのリクエストを送信する前後に割り込み、コンテキストや動作を修正するフックの仕組みである。modify model requestフックによるコンテキストエンジニアリングの中核をなす。変更可能な項目には、モデル、ツール、システムプロンプト、メッセージ履歴、モデル設定が含まれる。DeepAgentsにおけるcreate_react_agentからcreate_agent + middlewareへの進化を象徴する概念である（TLDV-02）。
**関連章**: 第3章, 第5章, 第6章

## 付録F: コンテキストエンジニアリングの学術的基盤

> **本付録の目的**: 第2章で概観したコンテキストエンジニアリングの学術的な裏付けを詳述する。本書の実務的な内容を支える理論的基盤として、主要な学術論文と研究成果を体系的に整理する。

### F.1 サーベイ論文: コンテキストエンジニアリングの3基盤コンポーネント

Mei, Yao et al.（2025年7月）は、コンテキストエンジニアリングを正式な学問分野として確立するサーベイ論文を発表した（ACD-02）。この論文は、コンテキストエンジニアリングを3つの基盤コンポーネントに体系化している。

**1. Context Retrieval and Generation（コンテキストの検索と生成）**

エージェントが推論に必要な情報を、外部ソースから取得・生成するプロセスである。RAG（Retrieval-Augmented Generation）が代表的な手法であるが、エージェントの文脈では、タスクの進行に伴って動的に検索クエリを生成し、必要な情報を段階的に取得する「エージェンティックRAG」へと進化している。

この領域の主要な技術的課題は以下の通りである。

- **検索精度と再現率のトレードオフ**: 多くの情報を取得すればコンテキストが肥大化し、少なすぎれば重要な情報を見落とす。エージェントの文脈では、タスクの進行段階に応じて検索戦略を動的に調整する必要がある
- **情報の鮮度管理**: 長期実行タスクでは、セッション開始時に取得した情報が陳腐化する可能性がある。外部ソースとの同期頻度の設計が重要になる
- **マルチモーダル情報の統合**: テキスト、コード、画像、構造化データなど、異なる形式の情報をコンテキストウィンドウ内で効率的に表現する手法

**2. Context Processing（コンテキストの処理）**

取得した情報を、LLMが効率的に推論できる形式に変換するプロセスである。トークン効率を最大化しつつ、情報の本質を保持することが目標となる。

主要な処理技術は以下の通りである。

- **要約・圧縮**: 長大なドキュメントやツール出力を、重要な情報を保持したまま短縮する。本書で「コンパクション」として扱っている技術の理論的基盤である
- **構造化**: 非構造化テキストをJSON、表、リストなどの構造化形式に変換し、LLMが解析しやすい形にする
- **フィルタリング**: タスクに無関係な情報をコンテキストから除外し、コンテキスト汚染（Context Poisoning）を防止する

**3. Context Management（コンテキストの管理）**

コンテキストのライフサイクル全体を管理するプロセスである。LangChainのLC-11で示された4つの操作戦略（Write, Select, Compress, Isolate）は、この領域の実務的な実装指針に相当する。

- **ライフサイクル管理**: コンテキスト情報の作成・更新・アーカイブ・削除を体系的に管理する。長期実行エージェントでは特に、古い情報の退避と新しい情報の統合のバランスが重要である
- **優先度管理**: コンテキストウィンドウの容量制限の中で、どの情報を優先的に保持し、どの情報を退避するかを決定するアルゴリズム
- **一貫性管理**: 複数のソースから取得した情報間の矛盾を検出・解決する仕組み

このサーベイ論文は、それまで実務的なノウハウの集積にとどまっていたコンテキストエンジニアリングに、学問的な体系を与えたという点で画期的である。本書の第2章で述べたコンテキストエンジニアリングの定義と技術分類は、この論文の体系に基づいている。

### F.2 ACEフレームワーク: コンテキスト進化プレイブック

Zhang, Hu, Upasani et al.（2025年10月発表、2026年1月改訂）は、ACE（Agentic Context Engineering）フレームワークを提案した（ACD-01）。ACEの核心的なアイデアは、コンテキストを「静的なドキュメント」ではなく「進化するプレイブック（Evolving Playbook）」として扱うことにある。

ACEフレームワークの3つのフェーズは以下の通りである。

**1. Generation（生成）フェーズ**

エージェントがタスクを実行する過程で、成功パターンや失敗パターンを構造化された形式で記録する。単なるログ出力ではなく、将来の同様のタスクで再利用可能な「プレイブック」として体系化する点が重要である。

具体的には、エージェントが特定のタスクタイプ（例: APIの統合テスト作成）を実行する際に、「このフレームワークではモックオブジェクトの設定に注意が必要」「テスト実行前にDBのシード処理が必須」といった知見をプレイブックとして蓄積する。

**2. Reflection（リフレクション）フェーズ**

蓄積されたプレイブックを、エージェント自身が評価・分析する。成功率の高い手法と低い手法を区別し、プレイブックの品質を向上させる。

このフェーズは、本書第6章で述べた自己検証ループ（PreCompletionChecklistMiddleware）の概念的な基盤でもある。エージェントが自身の過去の行動を振り返り、改善に活かすという考え方は、ハーネスエンジニアリングの反復的改善サイクルと密接に関連する。

**3. Curation（キュレーション）フェーズ**

リフレクションの結果に基づき、プレイブックの内容を精緻化・最適化する。冗長な情報の削除、一般化可能なパターンの抽出、例外ケースの追記などを行う。

ACEフレームワークの実験では、このプレイブック進化のアプローチにより、ベースラインに対して**+10.6%の性能改善**が報告されている。注目すべきは、この改善がモデルの変更やファインチューニングなしに達成されたことである。コンテキスト（プレイブック）の品質を高めるだけで、エージェントの性能を有意に向上させられることを実証している。

この研究は、本書が繰り返し強調してきた「モデルはコモディティ、ハーネス（コンテキスト設計を含む）こそが競争優位」という命題の学術的な裏付けを提供するものである。

### F.3 マルチエージェントコードアシスタントのためのコンテキストエンジニアリング

Haseeb（2025年8月）は、マルチエージェントLLMコードアシスタントにおけるコンテキストエンジニアリング手法を提案した（ACD-03）。この研究は、単一エージェントではなく複数エージェントが協調する環境でのコンテキスト設計に焦点を当てている。

提案手法は4つのステージで構成される。

**1. Intent Clarification（意図明確化）**

ユーザーの要求を分析し、曖昧な部分を解消してから処理を開始するステージである。ハーネスエンジニアリングの文脈では、Initializerエージェント（ANT-01）の役割に相当する。

**2. Semantic Literature Retrieval（セマンティック文献検索）**

コードベース、ドキュメント、外部ライブラリの情報を意味的類似性に基づいて検索するステージである。キーワードマッチではなく、コードの意図やパターンの類似性に基づく検索を行う。

**3. Document Synthesis（ドキュメント合成）**

検索した複数の情報ソースを統合し、タスクの実行に最適なコンテキストを構築するステージである。情報間の冗長性を排除し、矛盾を解決した上で、コンテキストウィンドウの容量内に収まる形で合成する。

**4. Multi-Agent Code Generation（マルチエージェントコード生成）**

合成されたコンテキストを基に、複数のエージェントが協調してコードを生成するステージである。各エージェントは異なる責務（設計、実装、テスト、レビュー等）を持ち、それぞれに最適化されたコンテキストが配信される。

この研究は、第6章で解説したマルチエージェント並列実行パターンにおけるコンテキスト分離戦略の理論的裏付けを提供する。各エージェントに「必要な情報だけを」渡すことで、コンテキスト汚染を防ぎつつ協調を実現するという設計原則は、実務でも有効性が確認されている。

### F.4 外部実行時メモリとポインタ方式

本書第3章で紹介した「外部実行時メモリ + ポインタ方式」（TLDV-06）の理論的基盤について述べる。この概念は、IBM研究チームが提案したコンテキストウィンドウ溢れ問題の解決策に基づいている。

**問題の定式化**:

コンテキストウィンドウの容量を $W$ トークン、現在のコンテキスト使用量を $C$ トークンとすると、長期実行エージェントでは以下の不等式が容易に破れる。

$$C \leq W$$

特にツール出力が大量のデータを返す場合（例: データベースクエリ結果、ファイル一覧、APIレスポンス）、$C$ は急激に増大する。

**ポインタ方式の解決策**:

ツール出力をコンテキストウィンドウ内に直接格納する代わりに、外部メモリ（ファイルシステム、データベース等）に保存し、コンテキストウィンドウ内にはその参照（ポインタ）のみを保持する方式である。

```
従来方式:
  コンテキストウィンドウ: [指示] + [ツール出力全体(大量)] + [応答]
  → コンテキスト溢れのリスク

ポインタ方式:
  コンテキストウィンドウ: [指示] + [ツール出力のポインタ(小量)] + [応答]
  外部メモリ:             [ツール出力全体]
  → エージェントが必要時にポインタ経由で参照
```

この方式の利点は以下の3点である。

1. **コンテキストウィンドウの効率的利用**: ポインタは数トークンで表現できるため、大量のツール出力を扱っても $C$ の増大を抑制できる
2. **選択的読み込み**: エージェントが必要な部分のみを読み込むことで、不要な情報によるコンテキスト汚染を防ぐ
3. **情報の永続化**: 外部メモリに保存された情報はセッション間で共有可能であり、長期実行タスクの状態管理に活用できる

Anthropicのコード実行とMCPの統合（ANT-08）で報告されたトークン85%削減（約134kから約5k）は、このポインタ方式の実践的な実装例とみなせる。オンデマンドツールロードの仕組みも、必要な情報のみをコンテキストに読み込むという同じ原理に基づいている。

LangChainのDeepAgentsにおける大規模ツール結果のエビクション（LC-03）も、この研究の延長線上にある実装である。コンテキストウィンドウの使用率がmax_input_tokensの85%に達した時点で自動的に圧縮を発動する仕組み（LC-04）は、ポインタ方式の閾値ベースの運用モデルと言える。

### F.5 ガードレールエージェント群アーキテクチャ

NVIDIA研究チームは、エージェントセキュリティのためのガードレールエージェント群アーキテクチャを提案した（TLDV-06）。この研究は、単一のガードレールモデルではなく、複数の専門化されたエージェントによる協調的な安全保障を実現する設計である。

アーキテクチャは4つのガードレールエージェントで構成される。

**1. グローバルセーフティエージェント（Global Safety Agent）**

システム全体の安全ポリシーを管理し、すべてのエージェント行動が組織のルール・方針に準拠しているかを監視する。Constitutional AI（ANT-17）の「憲法」に相当するグローバルルールの執行者である。

このエージェントは、個別のタスクやドメインに依存しない普遍的な安全基準を適用する。例えば、PII（個人識別情報）の出力禁止、危険な操作（プロダクション環境の直接変更等）の阻止、コンプライアンス違反の検出などを担当する。

**2. 攻撃エージェント（Attack Agent）**

レッドチーミングの自動化を担当するエージェントである。システムの弱点を探索し、ガードレールの回避（ジェイルブレイク）を試みることで、防御の脆弱性を事前に特定する。

本番環境ではなく、テスト環境やシミュレーション環境で動作する。発見された脆弱性は防御エージェントにフィードバックされ、ガードレールの強化に活用される。

**3. 防御エージェント（Defense Agent）**

ツール使用と外部情報のチェックを担当するエージェントである。エージェントが実行しようとするアクション（ファイル書き込み、API呼び出し、コマンド実行等）を事前に検証し、安全ポリシーへの準拠を確認する。

具体的な防御メカニズムは以下の通りである。

- **入力検証**: エージェントへの入力に悪意のあるプロンプトインジェクションが含まれていないかを検査する
- **出力検証**: エージェントの出力に機密情報やポリシー違反がないかを検査する
- **ツール呼び出し検証**: エージェントが要求するツール呼び出しが許可された範囲内であるかを検証する

**4. 評価エージェント（Evaluation Agent）**

安全値・成功率を数値化し、ガードレールシステム全体の効果を定量的に評価するエージェントである。CLASSic次元（ACD-04）のSecurityとStabilityの自動計測を担当する。

**リスクベースの防御配置原則**:

この研究の重要な知見は、すべてのエージェント行動に対して均一にガードレールを適用するのではなく、**リスクが大きい部分に限定して防御を集中配置する**という原則である。

```
高リスク操作（重点防御）:
  - 外部システムへの書き込み
  - ファイルの削除・上書き
  - 環境変数の変更
  - 本番環境へのデプロイ

低リスク操作（軽量チェック）:
  - ファイルの読み取り
  - 情報の検索・分析
  - テスト環境での実行
```

この設計原則は、第4章で述べたガードレール設計の実務的な指針（4.4節）の理論的根拠を提供するものである。コスト対効果の観点からも、全操作に重厚なガードレールを適用するよりも、リスクプロファイルに応じた防御の傾斜配分が合理的である。

### F.6 コンテキスト失敗モードの体系

LangChainのブログ記事「Context Engineering for Agents」（LC-11）は、Drew Breunig氏の分析に基づき、コンテキストの失敗モードを4つの類型に体系化した。これらの失敗モードは、ハーネス設計における防御的設計の出発点となる。

**1. Context Poisoning（コンテキスト汚染）**

コンテキストウィンドウに誤った情報や悪意のある情報が混入し、エージェントの判断を歪める失敗モードである。外部ソースから取得した情報に誤りが含まれていた場合や、プロンプトインジェクション攻撃により意図しない指示がコンテキストに注入された場合に発生する。

対策として、情報ソースの信頼性検証、入力検証によるインジェクション検出、コンテキスト内の矛盾検出メカニズムが有効である。ガードレールエージェント群（F.5節）の防御エージェントは、この失敗モードへの体系的な対策を提供する。

**2. Context Distraction（コンテキスト散漫）**

コンテキストウィンドウに大量の情報が蓄積され、エージェントが本来集中すべきタスクの情報に注意を向けられなくなる失敗モードである。「情報過多」とも言える状態であり、長期実行エージェントにおけるコンテキストロット（ANT-03）の主要な原因のひとつである。

対策として、プログレッシブ・ディスクロージャー（必要な情報のみを段階的に開示）、コンパクション（不要な情報の圧縮・退避）、コンテキスト分離（サブエージェントへの委譲）が有効である。

**3. Context Confusion（コンテキスト混乱）**

コンテキストウィンドウ内の情報が曖昧であるか、複数の解釈が可能であるため、エージェントが正しい行動を選択できなくなる失敗モードである。指示が抽象的すぎる場合や、異なるルール間で矛盾がある場合に発生する。

対策として、AGENTS.md / CLAUDE.md における明確で具体的なルール記述、ルール間の優先順位の明示、テスト可能な判断基準の提供が有効である。第4章で述べた「機械的で意見のあるルール」（4.2節）は、この失敗モードへの直接的な対策である。

**4. Context Clash（コンテキスト衝突）**

コンテキストウィンドウ内の異なる情報ソースが互いに矛盾している状態である。例えば、システムプロンプトのルールと取得したドキュメントの記述が矛盾する場合や、古い会話履歴の情報と最新のツール出力が矛盾する場合に発生する。

対策として、情報ソースの優先順位の明確化、タイムスタンプに基づく新旧判定、矛盾検出時の人間へのエスカレーションが有効である。本書第4章で述べた「ルールファイル体系の二層構造」（4.3節）における競合解決ルールは、この失敗モードを構造的に防止する設計である。

### F.7 OSメタファーによるコンテキスト設計理論

LangChainのブログ記事（LC-11）は、LLMベースのエージェントシステムをオペレーティングシステム（OS）のアナロジーで説明する理論的フレームワークを提示している。このメタファーは、コンテキスト設計の根本的な制約と戦略を直感的に理解するための強力なツールである。

**LLM = CPU（中央処理装置）**

LLMは、コンテキストウィンドウに配置された情報を「処理」する演算装置に相当する。CPUがレジスタやキャッシュに載ったデータしか直接処理できないのと同様に、LLMはコンテキストウィンドウ内の情報しか直接利用できない。モデルの推論能力は「クロック数」に、パラメータ数は「演算ユニットの数」に対応する。

**コンテキストウィンドウ = RAM（ランダムアクセスメモリ）**

コンテキストウィンドウは、LLMが現在アクセス可能な情報を保持する作業メモリに相当する。RAMには物理的な容量制限があるのと同様に、コンテキストウィンドウにもトークン数の上限がある。RAMの容量を超えるとスワッピングが発生して性能が低下するように、コンテキストウィンドウの容量に近づくと推論品質が低下する。

このメタファーから導出される設計原則は以下の通りである。

| OSの概念 | エージェントシステムの対応 | 設計への示唆 |
|---------|----------------------|------------|
| 仮想メモリ | 外部メモリ（ファイルシステム、DB） | コンテキストウィンドウに収まりきらない情報を外部に退避し、必要時に読み込む |
| ページング | コンパクション・エビクション | 使用頻度の低い情報を外部に退避し、使用頻度の高い情報をコンテキストに保持する |
| キャッシュ | プロンプトキャッシュ | 頻繁にアクセスする情報（システムプロンプト等）を高速にアクセスできる領域に保持する |
| プロセス分離 | サブエージェント分離 | 各エージェントに独立したコンテキスト空間を割り当て、相互干渉を防ぐ |
| IPC（プロセス間通信） | エージェント間メッセージング | 分離されたエージェント間で必要最小限の情報を交換する仕組み |
| ファイルシステム | 永続化メモリ（進捗ファイル等） | セッションをまたいで情報を保持する不揮発性ストレージ |

このOSメタファーは、ハーネス設計者に対して3つの重要な洞察を提供する。

第一に、**メモリ管理の重要性**である。OSが高度なメモリ管理を行うように、ハーネスもコンテキストウィンドウの管理を高度に行う必要がある。コンパクション、エビクション、外部メモリへのスワッピングは、OS設計の知見を直接応用したものである。

第二に、**プロセス分離の原則**である。マルチエージェントシステムにおける各エージェントのコンテキスト分離は、OSにおけるプロセス分離と同じ理由で重要である。あるエージェントのコンテキスト汚染が他のエージェントに波及することを防ぎ、システム全体の安定性を確保する。

第三に、**階層的メモリアーキテクチャ**である。レジスタ→キャッシュ→RAM→ディスクという階層構造と同様に、エージェントシステムにおいても、システムプロンプト（高速アクセス）→コンテキストウィンドウ（作業メモリ）→外部メモリ（永続ストレージ）という階層を意識した設計が有効である。

### F.8 学術研究とハーネスエンジニアリングの接点

本付録で取り上げた学術研究とハーネスエンジニアリングの実務は、以下の点で密接に関連している。

| 学術研究 | ハーネスエンジニアリングへの適用 |
|---------|--------------------------|
| 3基盤コンポーネント（ACD-02） | ハーネスの6コンポーネントのうち「Memory & Context」「Tool Registry」の設計理論 |
| ACEフレームワーク（ACD-01） | AGENTS.md / CLAUDE.md の反復的改善、プレイブック的なルール蓄積 |
| マルチエージェントコンテキスト（ACD-03） | サブエージェントのコンテキスト分離戦略、オーケストレーター・ワーカーパターン |
| 外部実行時メモリ + ポインタ（TLDV-06） | コンパクション、ファイルシステムによるコンテキスト管理、MCP統合 |
| ガードレールエージェント群（TLDV-06） | 多層ガードレール設計、リスクベースの防御配置 |
| コンテキスト失敗モード（LC-11） | 防御的コンテキスト設計、ルールファイルの競合解決 |
| OSメタファー（LC-11） | コンテキストウィンドウのメモリ管理戦略、プロセス分離 |

ハーネスエンジニアリングは2026年時点で急速に発展している実務領域であり、学術的な体系化はまだ初期段階にある。しかし、ここで紹介した研究が示すように、コンテキストエンジニアリングを中心とした理論的基盤は着実に構築されつつある。今後、より多くの実証研究とベンチマーク結果が蓄積されることで、ハーネス設計の「ベストプラクティス」が「エンジニアリング原則」へと昇華していくことが期待される。

実務者は、学術論文の知見をそのまま適用するのではなく、自身のプロジェクトの制約条件（コンテキストウィンドウのサイズ、タスクの複雑性、チームの規模等）に合わせて適応的に取り入れることが重要である。学術研究が提供するのは「なぜそうするべきか」の理論的根拠であり、「具体的にどうするか」は本書の各章（特に第3章〜第7章）の実装パターンとチェックリスト（付録B）を参照されたい。

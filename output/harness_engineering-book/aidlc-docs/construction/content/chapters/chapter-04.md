# 第4章: ポリシーとガードレールの設計

> **読者レベル**: ★★☆（中級 — AI/LLMの基礎知識がある読者向け）
> **想定ページ数**: 約35ページ

エージェントハーネスの6つのコンポーネントを第3章で概観した。本章では、そのなかでも最も「人間の意図」に近い領域——ポリシーとガードレール——に焦点を当てる。

AIエージェントがどれほど高度な推論能力を備えていても、「何をすべきか」「何をしてはならないか」を明示的に伝えなければ、人間の期待通りに動く保証はない。モデルの非決定論的な性質を前提として受け入れたうえで、エージェントの行動を予測可能な範囲に収めるための「設計された制約」——それがポリシーとガードレールである。

本章では、`AGENTS.md`や`CLAUDE.md`に代表されるマップファイルの設計哲学から始め、ルールファイル体系の二層構造、Constitutional AIに根ざすガードレールの理論と実装、オープン標準の動向、そしてスキルによるプログレッシブ・ディスクロージャー（Progressive Disclosure）まで、ポリシー設計の全体像を体系的に解説する。章末のHands-onでは、読者が自身のプロジェクトに対して実際に`AGENTS.md`/`CLAUDE.md`を作成し、その効果を検証する演習を行う。

## 4.1 マップファイルの設計哲学

### 「マニュアルではなくマップ」

エージェントがリポジトリ上で作業を開始するとき、最初に必要なものは何か。それは、リポジトリ全体の「地図」である。

OpenAIは`AGENTS.md`の公式ガイドにおいて、このファイルの設計思想を端的に表現している。「マニュアルではなくマップ」——すなわち、エージェントに対して逐一の手順を指示するのではなく、リポジトリの構造・規約・ツール・テスト手順を俯瞰的に示す「地図」として設計せよ、ということである（OAI-09）。

`AGENTS.md`（エージェンツ・エムディー）は、AIコーディングエージェントのためのオープンフォーマットであり、2026年2月時点で60,000以上のオープンソースプロジェクトに採用されている（STD-02）。Codex、Cursor、Devin、GitHub Copilotなど主要なAIコーディングツールがサポートしており、事実上の標準としての地位を確立しつつある。

`AGENTS.md`の典型的な構成は以下の通りである。

```markdown
# AGENTS.md

## Project Overview
TypeScript + React のフロントエンド。Monorepo構成。

## Build & Test
- `npm run build` でビルド
- `npm test` でテスト実行（Jest）
- `npm run lint` でLint実行（ESLint + Prettier）

## Code Style
- TypeScript strict mode を使用
- コンポーネントは関数コンポーネントで記述
- CSS-in-JS は使わず、CSS Modules を使用

## Architecture
docs/architecture.md を参照

## Directory Structure
- src/components/ — UIコンポーネント
- src/hooks/ — カスタムフック
- src/lib/ — ユーティリティ関数
- src/types/ — 型定義
```

[図4.1: AGENTS.mdの基本構成]

このファイルが「マップ」として機能するために重要なのは、**網羅性よりも導線**である。すべての詳細を`AGENTS.md`に書き込むのではなく、プロジェクト概要、ビルド・テストコマンド、コーディング規約といった最小限のコンテキストを提供し、詳細が必要な場合は`docs/`ディレクトリへ誘導する設計が推奨される（OAI-09）。

### CLAUDE.mdとその設計パターン

Anthropicのエコシステムでは、`CLAUDE.md`が同様の役割を果たす。AnthropicのClaude Code Best Practicesでは、`CLAUDE.md`をプロジェクトのルートに配置し、エージェントが最初に読み込むコンテキストとして活用する手法が詳述されている（ANT-13）。

`CLAUDE.md`の特徴は、Claude Codeのセッション開始時に自動的にコンテキストへ注入される点にある。つまり、エンジニアがセッションごとに同じ指示を繰り返す必要がない。プロジェクトの規約、ディレクトリ構造、テスト手順、ドメイン知識といった情報を一度`CLAUDE.md`に記述すれば、それ以降のすべてのセッションでエージェントはその情報を踏まえて動作する。

`CLAUDE.md`のベストプラクティスとして、Anthropicは以下の設計パターンを推奨している（ANT-13）。

1. **バッシュコマンドの明示**: ビルド、テスト、Lint実行のコマンドを正確に記述する。エージェントが自力で発見できるものでも、明示することで試行錯誤のトークンを節約できる
2. **コードスタイルの宣言**: インポートの順序、命名規則、型定義の方針など、コードレビューで指摘しがちな規約を列挙する
3. **ワークフロー指示**: TDD（テスト駆動開発）で進めるべきか、既存テストをまず実行してから修正すべきか、といった作業の進め方を指定する
4. **ドメイン知識の要約**: プロジェクト固有の用語、ビジネスロジックの要点、データモデルの概要を簡潔に記述する

### 約100行のテーブル・オブ・コンテンツ

マップファイルの設計において重要な原則のひとつが、**「ファイル自体を小さく保ち、目次としての機能を最大化する」**ことである。

実務で効果的なパターンは、`AGENTS.md`または`CLAUDE.md`を約100行前後のテーブル・オブ・コンテンツ（Table of Contents）として設計し、詳細な情報は`docs/`ディレクトリ配下の個別ファイルへ誘導する手法である。

```markdown
# CLAUDE.md

## このファイルについて
このファイルはプロジェクトのマップである。
詳細は各セクションからリンクされたドキュメントを参照すること。

## プロジェクト概要
ECサイトのバックエンドAPI。Go + PostgreSQL。
→ 詳細: docs/architecture.md

## ビルド・テスト
- `make build`
- `make test`
- `make lint`
→ 詳細: docs/development-guide.md

## APIエンドポイント一覧
→ docs/api-reference.md

## データモデル
→ docs/data-model.md

## コーディング規約
- Go標準のフォーマッタ（gofmt）を使用
- エラーハンドリングは errors.Wrap を使用
- テストは Table-Driven Tests パターンで記述
→ 詳細: docs/coding-standards.md

## デプロイメント
→ docs/deployment.md
```

この設計の利点は二重である。第一に、マップファイル自体のトークン消費を最小化できる。`CLAUDE.md`はセッション開始時に自動でコンテキストへ注入されるため、ファイルが肥大化するとそれだけでコンテキストウィンドウの貴重なスペースを圧迫する。第二に、エージェントは必要な情報にだけアクセスすればよく、不要な情報によるContext Distraction（コンテキスト注意散漫）を回避できる（LC-11）。

### マップファイル肥大化問題

理論的には美しい「約100行のテーブル・オブ・コンテンツ」設計であるが、実務では`CLAUDE.md`が際限なく肥大化していくという問題が頻発する。

2026年1月に開催されたAIコーディング道場の議論では、この問題が参加者の実体験として生々しく報告された（TLDV-03）。ある参加者は、開発指示を出した際に`CLAUDE.md`の設定が期待通りに機能しなかった経験を次のように語っている。

> 「タスクをちゃんと整理して書き出してからファイルベースで進めてほしかったんですけど、じゃんじゃん進みようと始めたので、ちょっとCLAUDE.mdも大きいのかなと思って分けたり」（TLDV-03）

この発言は、`CLAUDE.md`の肥大化がもたらす2つの問題を端的に示している。

**問題1: コンテキスト圧迫**

`CLAUDE.md`が数百行に膨らむと、セッション開始時にコンテキストウィンドウの大きな割合がマップファイルだけで占有される。その結果、実際の作業に使えるコンテキスト空間が圧縮され、エージェントの推論品質が低下する。

**問題2: 指示の優先順位曖昧化**

マップファイルに多数のルールが混在すると、エージェントがどのルールを優先すべきか判断できず、結果として指示が無視されたり、意図しない順序で作業が進められたりする。

この問題に対する実践的な対策として、以下のアプローチが報告されている。

1. **ファイル分割**: `CLAUDE.md`をトップレベルのマップファイルと、`docs/`配下の詳細ファイルに分割する。前述のテーブル・オブ・コンテンツ設計はこの対策の基本形である
2. **階層的なCLAUDE.md**: ルートの`CLAUDE.md`に加えて、サブディレクトリごとに`CLAUDE.md`を配置する。Claude Codeは作業ディレクトリのコンテキストに応じて該当する`CLAUDE.md`を読み込むため、関連するルールだけがコンテキストに注入される
3. **SQLite + MCPによる動的管理**: Claude Memプラグインのように、SQLiteをローカルに持ち、MCPサーバー経由でコンテキストを管理する手法が報告されている。コンテキスト圧縮を常時適用し、コンテキストウィンドウの使用量を50%以下に維持することで、「通常は50ツールコールで制限に達するところを、劇的に延長できる」とされる（TLDV-03）

第三のアプローチは、マップファイルの静的な配置という従来の枠組みを超え、コンテキスト管理そのものを動的に最適化する発想である。第6章で扱うコンパクションの技術と密接に関連する。

### MCP厳選の知見

マップファイルの肥大化と並んで、実務で直面するもうひとつの問題が、MCP（Model Context Protocol）サーバーの追加によるコンテキスト消費である。

2025年11月のAIコーディング道場で、参加者のhabuchin氏は以下のような知見を共有している（TLDV-04）。

> 「コード量が増えてきたりするとだんだん思考が散見するみたいな感じで精度が落ちてくるタイミングが多分あるんですね。そうならないようにドキュメントでこういう作業の時はここだけを読めとかそういう工夫がだんだん必要になる」（TLDV-04）

MCPサーバーを追加するたびに、そのサーバーが提供するツール定義がコンテキストへ注入される。10個のMCPサーバーを接続すれば、各サーバーのツール定義だけでコンテキストウィンドウの相当な割合を占有することになる。その結果、実際の作業タスクに使えるコンテキスト空間が圧縮され、エージェントの推論精度が低下する。

この問題に対する実践的な対策は「厳選」である。作業の種類に応じて、必要なMCPサーバーだけを有効化し、不要なものは無効化する。たとえば、コーディングタスクにはファイルシステムとgit関連のMCPのみを有効化し、ドキュメント生成タスクには別のMCPセットに切り替える、といった運用が推奨される。

この知見は、第4.6節で扱うプログレッシブ・ディスクロージャーの設計原則と直結する。汎用エージェントが少数のツールで高い効果を発揮する背景には、「必要最小限のコンテキストで最大限の成果を引き出す」というトークン効率の原則がある。

## 4.2 Golden Principles（不変ルール）

### 機械的で意見のあるルール

マップファイルが「何がどこにあるか」を示す地図であるのに対し、Golden Principles（ゴールデンプリンシプル）は「何をすべきか、何をしてはならないか」を定める不変ルールである。

12 Factor Agentsの方法論では、本番対応のAIエージェントを構築するために、12個の明確な設計原則が提唱されている（BLG-11）。この方法論の核心は、ルールは**「機械的で意見のある」**（mechanical and opinionated）ものであるべきだという主張にある。

「機械的」とは、ルールの適用に判断の余地がないことを意味する。「テストを書くべき」ではなく「すべてのパブリック関数にユニットテストを書け」と記述する。「コードを読みやすくすべき」ではなく「1関数あたり最大30行、ネスト深さ最大3」と記述する。このような機械的なルールは、非決定論的なLLMに対しても一貫した行動を引き出しやすい。

「意見のある」とは、複数の妥当な選択肢がある場合にひとつを明確に選ぶことを意味する。たとえば、TypeScriptのインポートスタイルとして相対パスと絶対パスのどちらが「正しい」かには議論の余地がある。しかし、ルールとして「絶対パスを使用する」と宣言してしまえば、エージェントは迷わずそのスタイルに従う。意見の是非よりも、一貫性の方がはるかに重要なのである。

### Golden Principlesの具体例

以下は、Golden Principlesとしてマップファイルに記述するルールの実例である。

```markdown
# Golden Principles（不変ルール）

## コード品質
- すべてのパブリック関数にドキュメントコメントを記述する
- 1関数の行数は最大50行とする
- ネストの深さは最大3レベルとする
- マジックナンバーは禁止。定数として定義する

## テスト
- すべてのパブリックAPIにユニットテストを記述する
- テストは Table-Driven Tests パターンで記述する
- モックは最小限にし、可能な限り実際の依存を使用する
- テストファイルは対象ファイルと同一ディレクトリに配置する

## セキュリティ
- シークレット情報（APIキー、パスワード）をコードにハードコードしない
- 環境変数または Secrets Manager を使用する
- SQLクエリは必ずパラメータバインディングを使用する

## Git
- コミットメッセージは Conventional Commits 形式とする
- 1PRあたりの変更は300行以内を目標とする
- main ブランチへの直接pushは禁止する

## 禁止事項
- force push しない
- .env ファイルをコミットしない
- 既存のテストを削除しない（修正のみ可）
- 外部APIの認証情報をログに出力しない
```

[図4.2: Golden Principlesの記述例]

これらのルールには共通する特徴がある。いずれも**検証可能**であるということである。「コードを読みやすくする」は検証が困難であるが、「1関数の行数は最大50行」はLintツールで機械的に検証できる。エージェントが従うルールは、理想的には自動検証が可能なレベルの具体性を持つべきである。

### コードベースの可読性と一貫性の維持

Golden Principlesの最大の価値は、**コードベース全体の一貫性を維持する**ことにある。

人間の開発者がチームでコードを書く場合、コードレビューという社会的なプロセスを通じて一貫性が保たれる。しかし、AIエージェントがコードを生成する場合、セッション間で状態が保持されないため、異なるセッションで異なるスタイルのコードが生成されるリスクがある。

たとえば、あるセッションではエラーハンドリングに`try-catch`を使い、別のセッションでは`Result`型を使うといった不一致が生じ得る。Golden Principlesとして「エラーハンドリングは`Result`型を使用する」と明示しておけば、どのセッションでも同じパターンが適用される。

この一貫性は、長期実行エージェント（第6章）において特に重要である。複数のコンテキストウィンドウにまたがる作業では、各セッションが異なるコンテキストで動作する。Golden Principlesは、セッション間をつなぐ「共通言語」として機能し、コードベース全体の整合性を担保するのである。

### エージェント実行間の予測可能性

Golden Principlesのもうひとつの重要な機能は、**エージェントの行動を予測可能にする**ことである。

LLMの出力は本質的に非決定論的である。同じプロンプトに対して、実行するたびに異なる回答が返される可能性がある。この非決定論性は、自由形式のテキスト生成においては長所であるが、コードベースの一貫性が求められるソフトウェア開発においてはリスクである。

Golden Principlesは、非決定論的な出力空間を制約することで、予測可能性を高める。エージェントに「好きなようにコードを書け」と指示するのではなく、「以下のルールに従ってコードを書け」と指示することで、出力のばらつきを大幅に抑制できる。

この原則は、OpenAIの7人チーム実験（OAI-01）においても確認されている。エンジニアたちが発見した教訓のひとつは、エージェントに自由度を与えるよりも、明確なルールで行動を制約した方が、結果として高品質なコードが生成されるということであった。「人間が舵を取り、エージェントが実行する」というハーネスエンジニアリングの基本原則は、Golden Principlesの設計にそのまま当てはまる。

## 4.3 ルールファイル体系の二層構造

### ベースレイヤーとカスタマイズレイヤー

Golden Principlesの数が増え、複数のプロジェクトに跨ってルールを管理するようになると、ルールファイルの体系的な管理が課題となる。この課題に対する有効な設計パターンが、**二層構造**（Two-Layer Architecture）である。

二層構造とは、ルールファイルを以下の2つのレイヤーに分離する設計パターンである。

```
ルールファイル体系
├── ベースレイヤー（Base Layer）
│   └── 標準ルール（変更不可）
│       プラットフォーム共通の設計原則、
│       業界標準のコーディング規約、
│       セキュリティのベースライン
│
└── カスタマイズレイヤー（Customization Layer）
    └── 拡張ルール（プロジェクト固有）
        業界・業務に特化したルール、
        組織独自の品質基準、
        プロジェクト固有の規約
```

[図4.3: ルールファイル体系の二層構造]

**ベースレイヤー**は、変更不可の標準ルールを格納する。これは、チームやプロジェクトを超えて共通に適用される基本原則であり、個別のカスタマイズの対象とはしない。ベースレイヤーの変更は、組織全体の合意に基づいて慎重に行われるべきである。

**カスタマイズレイヤー**は、ベースレイヤーを**上書きではなく補完・拡張**するプロジェクト固有のルールを格納する。カスタマイズレイヤーのルールは、ベースレイヤーのルールと矛盾しない限りにおいて自由に追加・変更できる。

この設計パターンには、ソフトウェアエンジニアリングにおけるオブジェクト指向プログラミングの「継承」に通ずる考え方がある。ベースレイヤーが「基底クラス」、カスタマイズレイヤーが「派生クラス」に相当し、派生クラスは基底クラスの機能を継承しつつ、独自の機能を追加するのである。

### AI-DLC事例: 標準ルール + 拡張ルールの競合解決

二層構造の具体例として、本書の執筆プロセス自体で使用しているAI-DLC（AI-Driven Development Life Cycle）のルールファイル体系を紹介する。

AI-DLCでは、以下のディレクトリ構造でルールを管理している。

```
.steering/
├── aws-aidlc-rules/
│   └── core-workflow.md          # マスターワークフロー（変更不可）
├── aws-aidlc-rule-details/       # ベースレイヤー（変更不可）
│   ├── common/
│   ├── inception/
│   ├── construction/
│   └── operations/
└── ga-aidlc-rule-details/        # カスタマイズレイヤー
    ├── introduction.md           # 会社紹介
    ├── company-profile.md        # コアコンピタンス
    ├── service-catalog.md        # サービス体系
    └── services/                 # サービス別実施要領
        ├── training.md
        ├── agent-introduction.md
        ├── advisory.md
        └── talent.md
```

この構造において、`aws-aidlc-rule-details/`がベースレイヤーに、`ga-aidlc-rule-details/`がカスタマイズレイヤーに該当する。ベースレイヤーにはAI-DLCフレームワーク標準のルール詳細が格納されており、このディレクトリ配下のファイルは一切変更しない。カスタマイズレイヤーには、組織独自の知見・ノウハウ・サービスメニュー・実施要領が格納される。

各ステージを実行する際のルール読み込み手順は以下の通りである。

1. `aws-aidlc-rule-details`から該当ステージのルールファイルを読み込む
2. `ga-aidlc-rule-details`に同名または関連するファイルが存在する場合、追加で読み込む
3. 両方のルールを統合して適用する

重要なのは、ステップ3における**競合解決の優先順位**が明確に定義されている点である。

### 優先順位設計の原則

二層構造において避けられないのが、ベースレイヤーとカスタマイズレイヤーの間で生じるルールの競合である。この競合を解決するための優先順位設計は、二層構造の成否を左右する重要な設計判断である。

AI-DLCでは、以下の3段階の優先順位を定義している。

```
優先順位（高 → 低）

1. core-workflow.md のプロセスフロー  ← 最優先（変更不可）
   ワークフローの基本的な順序・構造は不可侵

2. カスタマイズレイヤーの業界・業務固有ルール  ← 内容面で優先
   業界固有の規制要件、組織の品質基準など

3. ベースレイヤーの標準ルール  ← ベースライン
   プラットフォーム共通の基本原則
```

[図4.4: 三段階の優先順位設計]

この優先順位設計にはいくつかの重要な設計判断が含まれている。

**第一に、「プロセスの骨格」は不可侵とする。**AI-DLCにおける`core-workflow.md`は、INCEPTION → CONSTRUCTION → OPERATIONSの3フェーズで進行するワークフローの基本構造を定義している。この構造自体をカスタマイズレイヤーで変更することは許されない。これは、ワークフローの予測可能性を担保するための設計判断である。

**第二に、「内容」の優先権はカスタマイズレイヤーにある。**プロセスの骨格に従いつつ、具体的な成果物の内容・品質基準・用語定義などはカスタマイズレイヤーが優先される。たとえば、ベースレイヤーが「設計レビューを実施する」と規定しているのに対し、カスタマイズレイヤーが「金融庁ガイドラインに基づくリスク評価を含めた設計レビューを実施する」と規定している場合、後者が適用される。

**第三に、競合が不明確な場合のフォールバック先を定義する。**どちらのレイヤーのルールを適用すべきか判断できない場合、ベースレイヤーのルールがフォールバックとして適用される。これにより、ルール適用の不確実性を最小化している。

この優先順位設計は、一般的なソフトウェアプロジェクトにも応用可能である。以下に、Web開発プロジェクトでの二層構造の例を示す。

```markdown
# ベースレイヤー: organization-rules.md（組織全体共通）

## セキュリティ
- すべてのAPIエンドポイントに認証を必須とする
- SQLインジェクション対策としてパラメータバインディングを使用する
- OWASP Top 10 のすべての脆弱性に対策する

## テスト
- カバレッジ80%以上を維持する
- E2Eテストを主要ユースケースに実装する
```

```markdown
# カスタマイズレイヤー: project-rules.md（プロジェクト固有）

## セキュリティ（拡張）
- PCI DSS準拠のため、カード情報は暗号化して保存する
- 決済処理は専用のマイクロサービスで分離する
（→ ベースレイヤーのセキュリティルールに加えて適用）

## テスト（拡張）
- 決済フローのテストは実際のサンドボックス環境で実行する
- 性能テストで99パーセンタイルのレスポンスタイムを500ms以下に維持する
（→ ベースレイヤーのテストルールに加えて適用）
```

二層構造の設計パターンは、組織が成長し、プロジェクトが増加するにつれてその価値が増大する。最初は単一の`AGENTS.md`で十分であっても、組織全体でのルール標準化が進むと、共通ルール（ベースレイヤー）とプロジェクト固有ルール（カスタマイズレイヤー）の分離が不可避となる。早い段階から二層構造を意識した設計を行うことで、将来のスケールに備えることができる。

## 4.4 安全性とガードレール

### Constitutional AIの理論的基盤

ガードレール設計の理論的基盤を理解するためには、Constitutional AI（コンスティテューショナルAI、略称CAI）に遡る必要がある。

2022年12月にAnthropicが発表したConstitutional AIの論文は、AIシステムの安全性確保に対する画期的なアプローチを提案した（ANT-17）。CAIの核心的なアイデアは、AIシステムに「憲法」（Constitution）——すなわち、従うべき原則の集合——を与え、AIシステム自身にその原則に基づく自己批判と修正を行わせるというものである。

CAIの訓練プロセスは2段階で構成される。

**第1段階: 自己批判による教師あり学習**

AIモデルに応答を生成させた後、「憲法」の原則に照らして自身の応答を批判させ、修正させる。この自己批判と修正のプロセスを繰り返すことで、原則に沿った応答を生成する能力が向上する。

**第2段階: RLAIF（Reinforcement Learning from AI Feedback）**

人間のフィードバックの代わりに、AIによるフィードバックを用いた強化学習を行う。これにより、人間のアノテーターへの依存を軽減しつつ、大規模な安全性訓練が可能となる。

CAIのアプローチがハーネスエンジニアリングにとって重要なのは、「自然言語で記述されたルール」がAIの行動を制御できるという実証を提供した点にある。`AGENTS.md`や`CLAUDE.md`に記述するルールは、まさにCAIの「憲法」のハーネス版であると捉えることができる。

### Constitutional Classifiers: ジェイルブレイク対策

CAIの概念を実用的なセーフガードとして発展させたのが、Constitutional Classifiers（コンスティテューショナル・クラシファイア）である（ANT-18）。

2025年2月にAnthropicが発表したConstitutional Classifiersは、自然言語で記述された「憲法」からセーフガードを自動生成する技術である。具体的なプロセスは以下の通りである。

1. **自然言語ルールの定義**: 禁止すべき入出力のカテゴリを自然言語で記述する
2. **合成データの生成**: 定義されたルールに基づいて、許可される入出力と禁止される入出力のデータセットを自動生成する
3. **分類器の訓練**: 生成された合成データで入力分類器と出力分類器を訓練する
4. **ランタイムでの適用**: 訓練された分類器が、ユーザー入力とモデル出力をリアルタイムで検査する

この技術の最も注目すべき成果は、**ジェイルブレイク（Jailbreak）成功率を4.4%にまで低減した**ことである（ANT-18）。ジェイルブレイクとは、AIシステムの安全機構を回避して、禁止された出力を生成させる攻撃手法である。従来のガードレールではジェイルブレイク耐性に限界があったが、Constitutional Classifiersはこれを大幅に改善した。

```
ユーザー入力
    │
    ▼
┌──────────────────┐
│  入力分類器        │ ← 「憲法」に基づいて訓練
│  (Input Classifier) │
└──────────────────┘
    │ 許可
    ▼
┌──────────────────┐
│  LLMモデル         │
│  (推論実行)         │
└──────────────────┘
    │
    ▼
┌──────────────────┐
│  出力分類器        │ ← 「憲法」に基づいて訓練
│  (Output Classifier)│
└──────────────────┘
    │ 許可
    ▼
ユーザーへの応答
```

[図4.5: Constitutional Classifiersのアーキテクチャ]

エージェントハーネスの設計者にとって、Constitutional Classifiersの設計思想から得られる教訓は明確である。ガードレールは、**モデルの内部に組み込む**のではなく、**モデルの外側に配置する独立した検査機構**として設計すべきである。この「外側に配置する」という設計判断は、ハーネスエンジニアリングの基本原則——モデルの外側にシステムを構築する——と完全に一致する。

### Policy-as-Prompt: 自然言語ポリシーからの自動ガードレール生成

Constitutional Classifiersがモデルの訓練段階でガードレールを構築するアプローチであるのに対し、Policy-as-Prompt（ポリシー・アズ・プロンプト）は、ランタイム段階でポリシーを動的に適用するアプローチである（ACD-09）。

Policy-as-Promptの基本的なアイデアは、自然言語で記述された組織のポリシー文書を、動的で強制可能なガードレールに自動変換するというものである。具体的には以下のプロセスで実現される。

1. **ポリシー文書の入力**: 組織のセキュリティポリシー、コンプライアンス規定、利用規約などを自然言語で記述する
2. **ガードレールプロンプトの自動生成**: ポリシー文書からプロンプトベースの分類器を自動生成する
3. **ランタイムでの最小権限適用**: 生成された分類器がランタイム時に最小権限ポリシーを強制する

このアプローチの利点は、ポリシーの変更に迅速に対応できることである。Constitutional Classifiersでは分類器の再訓練が必要であるが、Policy-as-Promptではプロンプトの変更だけでポリシーの更新が可能である。

```python
# Policy-as-Prompt の概念的な実装例
class PolicyGuardrail:
    def __init__(self, policy_document: str):
        self.policy_prompt = self._generate_guardrail_prompt(policy_document)

    def _generate_guardrail_prompt(self, policy: str) -> str:
        return f"""
        以下の組織ポリシーに基づいて、
        ユーザーのリクエストが許可されるか判定してください。

        === ポリシー ===
        {policy}
        === ポリシー終了 ===

        判定: ALLOW / DENY / ESCALATE
        理由: （簡潔に）
        """

    def check(self, user_request: str) -> dict:
        # LLMにポリシー準拠を判定させる
        result = llm.invoke(
            system=self.policy_prompt,
            user=user_request
        )
        return parse_decision(result)
```

Policy-as-Promptは、マップファイルのGolden Principlesとガードレールを統合する可能性を示唆している。`AGENTS.md`に記述されたルールを自動的にガードレールとして適用するアーキテクチャは、ポリシー管理の一元化と実行時の自動強制を両立する理想的なアプローチとなり得る。

### Middlewareによるガードレール実装

理論的な基盤から実装レベルへ移行しよう。LangChainのエコシステムでは、ガードレールをMiddleware（ミドルウェア）として実装するパターンが確立されている（TLDV-02）。

第3章で解説したMiddlewareの概念を復習すると、Middlewareはエージェントのモデルリクエストを修正するフックである。`modify model request`フックを通じて、モデルへの入力（システムプロンプト、メッセージ履歴、ツール定義など）を動的に変更できる。

LangChain VIPコミュニティコールでは、Colin McNamara氏がガードレールをMiddlewareで実装し、abuse（悪用）検出のための統計的プロセス制御（Statistical Process Control）として活用する提案を行っている（TLDV-02）。

> 「you got guardrails on the agent maybe you're using middleware to establish a guardrail」（TLDV-02）

この発言が示す設計パターンは以下の通りである。

```python
# ガードレールMiddlewareの概念的な実装
class GuardrailMiddleware:
    def __init__(self, rules: list[str]):
        self.rules = rules
        self.violation_counter = Counter()

    async def modify_model_request(self, request: ModelRequest) -> ModelRequest:
        # 入力チェック: ルール違反がないか検査
        for rule in self.rules:
            if self._violates_rule(request.messages, rule):
                self.violation_counter[rule] += 1
                # 統計的プロセス制御: 閾値を超えたらエスカレーション
                if self.violation_counter[rule] > THRESHOLD:
                    return self._escalate(request, rule)
                # ルール違反をシステムプロンプトに警告として注入
                request.system_prompt += f"\n\n警告: {rule} に違反しています。"
        return request
```

Middlewareとしてのガードレール実装には3つの利点がある。

**第一に、プラガブル（着脱可能）であること。**ガードレールMiddlewareは、エージェントのコアロジックから独立しているため、プロジェクトや環境に応じて自由に追加・削除できる。開発環境では緩いガードレールを、本番環境では厳格なガードレールを適用する、といった切り替えが容易である。

**第二に、合成可能（Composable）であること。**複数のガードレールMiddlewareを連鎖させることで、多層的な防御を構築できる。セキュリティガードレール → コンプライアンスガードレール → 品質ガードレール、といった順序で処理を重ねることができる。

**第三に、統計的な監視が可能であること。**Middlewareのレイヤーで違反の統計を収集できるため、どのルールが頻繁に違反されているかを可視化し、ルールの改善やエージェントの挙動分析に活用できる。

### ガードレールエージェント群アーキテクチャ

Middlewareによる実装がソフトウェアレベルのガードレールであるのに対し、より大規模なシステムレベルのガードレールとして、NVIDIAが提案するガードレールエージェント群（Guardrail Agent Ensemble）のアーキテクチャがある（TLDV-06）。

このアーキテクチャでは、4種類の専門化されたエージェントがエージェントシステム全体の安全性を監視する。

**1. グローバルセーフティエージェント（Global Safety Agent）**

システム全体のルール・方針を管理するエージェントである。すべてのエージェントの行動がグローバルポリシーに適合しているかを継続的に監視する。CAIの「憲法」をシステムレベルで体現する存在であり、ポリシーの一元的な管理と適用を担う。

**2. 攻撃エージェント（Adversarial Agent）**

いわゆるレッドチーミングを自動化するエージェントである。わざと攻撃的な入力や予期しないエッジケースを試みて、他のエージェントの弱点を探索する。ジェイルブレイクの試行、プロンプトインジェクション、権限昇格の試みなどを自動的に実行し、脆弱性を事前に発見する。

**3. 防御エージェント（Defense Agent）**

ツールの使い方や外部情報の取り込み方をチェックするエージェントである。エージェントが実行しようとしているツールコールが適切か、外部APIへのリクエストが安全か、ファイルシステムへのアクセスが許可範囲内かを検査する。入力と出力の両方を監視する「境界警備」の役割を果たす。

**4. 評価エージェント（Evaluation Agent）**

実際のエージェントの動作を分析して、安全値や成功率を数値化するエージェントである。定量的なメトリクスに基づいてシステムの健全性を評価し、異常を検出した場合にアラートを発する。

```
                    ┌─────────────────────────┐
                    │ グローバルセーフティ       │
                    │ エージェント              │
                    │ (ポリシー管理)            │
                    └──────────┬──────────────┘
                               │ 監視
          ┌────────────────────┼────────────────────┐
          │                    │                    │
┌─────────┴────────┐  ┌───────┴──────┐  ┌─────────┴─────────┐
│ 攻撃エージェント  │  │ 作業エージェント│  │ 防御エージェント   │
│ (レッドチーミング) │  │ (本体)       │  │ (境界警備)        │
└──────────────────┘  └──────────────┘  └───────────────────┘
                               │
                    ┌──────────┴──────────────┐
                    │ 評価エージェント          │
                    │ (メトリクス・異常検知)     │
                    └─────────────────────────┘
```

### リスクベースの防御配置原則

ガードレールエージェント群の研究から導出された重要な設計原則が、**リスクベースの防御配置**である（TLDV-06）。

> 「全箇所に防御を入れるのではなく、リスクが大きい部分に限定して軽量の防御を施す」（TLDV-06）

この原則は、一見すると常識的に聞こえるかもしれないが、実務ではしばしば逆のアプローチが採られる。すべてのツールコールにガードレールを設置する「全方位防御」は、安全性を最大化するように見えて、実際にはレイテンシの増大、トークン消費の増加、エージェントの行動の過度な制約といった副作用を引き起こす。

リスクベースの防御配置では、以下の基準でガードレールの配置密度を決定する。

| リスクレベル | 対象の例 | 防御密度 |
|------------|---------|---------|
| **高リスク** | 外部APIへのデータ送信、ファイル削除、本番データベースへの書き込み | 多層防御 + 人間の確認 |
| **中リスク** | ファイルの作成・編集、新しいパッケージの導入 | 自動ガードレール |
| **低リスク** | 読み取り専用のファイルアクセス、ログの出力 | 最小限の監視のみ |

[表4.1: リスクベースの防御配置マトリクス]

「重要な操作の前には人の確認を挟む」という原則は、完全自律型エージェントの理想と矛盾するように思えるかもしれない。しかし、現時点のエージェント技術においては、高リスクな操作における人間の確認（Human-in-the-Loop）は、信頼性を担保する上で欠かせない安全弁である。

### サンドボックス: ファイルシステム分離とネットワーク分離

ポリシーファイルとMiddlewareが「ソフト」なガードレールであるのに対し、サンドボックス（Sandbox）は「ハード」なガードレールである。ソフトなガードレールはLLMの応答を制御するものであり、理論上は回避される可能性がある。ハードなガードレールはOSレベルで物理的に制約を課すものであり、回避は技術的に極めて困難である。

Anthropicは、Claude Codeのサンドボックス設計について詳細な技術記事を公開している（ANT-07）。Claude Codeのサンドボックスは、2つの分離境界で構成される。

**ファイルシステム分離**

エージェントがアクセスできるファイルシステムの範囲を、プロジェクトディレクトリとその配下に限定する。Linuxでは`bubblewrap`、macOSでは`seatbelt`というOSネイティブのサンドボックス機構が使用される。

```
許可されるアクセス:
/home/user/project/        ← プロジェクトディレクトリ
/home/user/project/src/    ← 配下のサブディレクトリ
/tmp/claude-workspace/     ← 一時ファイル

禁止されるアクセス:
/home/user/.ssh/           ← SSH鍵
/home/user/.aws/           ← AWS認証情報
/etc/passwd                ← システムファイル
/home/user/other-project/  ← 他のプロジェクト
```

**ネットワーク分離**

エージェントからのネットワークアクセスを制限する。デフォルトでは外部への通信を遮断し、許可リストに記載されたエンドポイントのみへのアクセスを許可する。

これらのサンドボックス設計の成果として、**許可プロンプトが84%削減された**と報告されている（ANT-07）。サンドボックスが導入される以前は、エージェントがファイルを作成するたび、コマンドを実行するたびに、ユーザーに許可を求めるプロンプトが表示されていた。サンドボックスによって安全な境界が確立されたことで、境界内の操作は自動的に許可され、ユーザーは高リスクな操作にのみ判断を集中できるようになった。

この設計は、前述のリスクベースの防御配置原則を物理レベルで実現したものと解釈できる。サンドボックス内の操作は「低リスク」として自動許可され、サンドボックスを超える操作は「高リスク」として人間の確認を要求する。

### ドメイン特化ガードレール: 金融分野の実践例

ここまで解説したガードレールの手法は、ドメインを問わず適用可能な汎用的なものであった。しかし実務では、業界固有の規制やビジネスルールに基づく**ドメイン特化ガードレール**が不可欠となる場合がある。

金融分野はその典型例である。2025年12月のセッションで、金融業界の参加者は以下のように述べている（TLDV-05）。

> 「ガードレールというかルールとか、投資するときに必ずこういうのは当然だみたいな、ツールとか、そういうのをミスマッチとか、マッチとか、そういうのをチェックしながら」（TLDV-05）

金融分野でのドメイン特化ガードレールの具体例を以下に示す。

```markdown
# 金融ドメイン: ガードレール定義

## 投資判断ルール
- 推奨銘柄のリストは必ず承認済みリストと照合する
- 投資上限額を超える提案は自動的にフラグを立てる
- 顧客のリスク許容度プロファイルとの整合性を検証する

## コンプライアンス
- 金融商品取引法に基づく適合性原則を遵守する
- インサイダー情報に該当し得る非公開情報を出力しない
- 特定の銘柄を推奨する際は、リスク説明を必ず併記する

## データ保護
- 顧客の個人情報（氏名、口座番号等）をログに出力しない
- 取引データの外部送信は暗号化されたチャネルのみ許可する
- 顧客データへのアクセスログを監査証跡として記録する
```

金融分野のガードレールが汎用的なガードレールと異なるのは、**規制準拠が法的義務である**点である。汎用的なガードレール（たとえば「プロンプトインジェクションを防ぐ」）の違反はセキュリティインシデントであるが、金融ガードレール（たとえば「適合性原則に違反した投資推奨を行う」）の違反は法的な制裁につながり得る。

したがって、ドメイン特化ガードレールの設計では、以下の追加要件が生じる。

1. **監査証跡の保持**: すべてのガードレールの判定結果を、後から検証可能な形で記録する
2. **説明可能性**: ガードレールがなぜその判定を下したかを、人間が理解できる形で説明する
3. **規制変更への追従**: 法改正や規制変更に迅速に対応するためのポリシー更新プロセスを確立する

これらの要件は、Policy-as-Promptのアプローチと親和性が高い。ポリシー文書を更新するだけでガードレールが自動的に変更される仕組みがあれば、規制変更への対応コストを大幅に削減できる。

---

> **Column: ガードレールは「制約」か「解放」か**
>
> ガードレールについて論じると、「AIの能力を制限するもの」というネガティブな印象を持つ読者がいるかもしれない。しかし実際には、ガードレールの導入はエージェントの自律性を**拡大する**ことが多い。
>
> Claude Codeのサンドボックス導入が好例である。サンドボックスが導入される以前は、エージェントのあらゆる操作に対してユーザーの許可が必要であった。ファイルを作成するたびに「許可しますか？」、コマンドを実行するたびに「許可しますか？」と問われ、エージェントの自律的な動作は事実上不可能であった。サンドボックスの導入により、安全な範囲が明確に定義されたことで、その範囲内の操作は自動許可となり、許可プロンプトは84%削減された（ANT-07）。
>
> つまり、ガードレールは「ここまでは自由にやってよい」という境界線を引くことで、エージェントの行動可能な範囲を明確化し、結果的にその範囲内での自律性を高めるのである。第1章で紹介した馬具のアナロジーを思い出してほしい。ハーネスは馬の力を殺すのではなく、適切な方向へ導く。ガードレールもまた、エージェントの能力を制限するのではなく、信頼性高く発揮させるための基盤なのである。

## 4.5 Agentic AI Foundationとオープン標準

### AAIF: AGENTS.md + MCP + goose

2025年12月9日、Linux Foundation傘下にAgentic AI Foundation（AAIF、エージェンティックAIファウンデーション）が設立された（OAI-06, STD-01）。この設立は、AIエージェントのエコシステムにおけるオープン標準化の重要なマイルストーンである。

AAIFの設立プロジェクトとして、3つのオープン標準が寄贈された。

1. **AGENTS.md**（寄贈元: OpenAI）: AIコーディングエージェントのためのリポジトリローカルな指示ファイルの標準フォーマット
2. **MCP（Model Context Protocol）**（寄贈元: Anthropic）: AIアプリケーションとデータソース・ビジネスツール・開発環境を接続する標準プロトコル
3. **goose**（寄贈元: Block）: AIコーディングアシスタントのオープンソース実装

プラチナメンバーには、AWS、Anthropic、Block、Bloomberg、Cloudflare、Google、Microsoft、OpenAIという主要テック企業が名を連ねている（STD-01）。

AAIFの設立が意味するのは、エージェントハーネスの「ポリシーとツール」に関する標準化が、業界横断で進展しているということである。`AGENTS.md`がエージェントへの指示の標準フォーマットとなり、MCPがツール接続の標準プロトコルとなることで、特定のベンダーに依存しないエージェントエコシステムの構築が現実味を帯びてきた。

### 相互運用性の現状と展望

AAIFの設立により、エージェントハーネスの相互運用性は急速に向上しつつある。しかし、その現状を冷静に評価する必要がある。

**現状で実現されている相互運用性**

- `AGENTS.md`は60,000以上のOSSプロジェクトに採用されており、Codex、Cursor、Devin、GitHub Copilotなど主要なコーディングエージェントがサポートしている（STD-02）
- MCPは、多数のIDEやAIアシスタントに統合されており、ツール接続の事実上の標準となりつつある
- `AGENTS.md`に記述されたルールは、プラットフォームを超えて共通に適用できる

**今後の課題**

- `AGENTS.md`と`CLAUDE.md`の仕様統合: 現時点では、OpenAI系エージェントが`AGENTS.md`を、Anthropic系エージェントが`CLAUDE.md`を参照するという二重構造が存在する。AAIFの枠組みでこの統合が進む可能性がある
- ガードレールの相互運用性: ガードレールの定義フォーマットはまだ標準化されていない。各プラットフォームが独自のガードレール実装を持っており、プラットフォーム間での移植は容易ではない
- 評価基準の標準化: エージェントの品質を評価する基準（第7章で詳述）の標準化も、今後の課題である

実務的な推奨としては、**`AGENTS.md`と`CLAUDE.md`の両方を維持する**ことが現時点でのベストプラクティスである。両ファイルの内容を同期させておくことで、どのプラットフォームのエージェントを使用しても同じルールが適用される状態を保つことができる。

```
プロジェクトルート/
├── AGENTS.md       ← OpenAI系エージェント向け
├── CLAUDE.md       ← Anthropic系エージェント向け
├── .cursorrules    ← Cursor向け（内容はAGENTS.mdと同期）
└── docs/
    ├── architecture.md
    ├── coding-standards.md
    └── ...
```

将来的にAAIFのもとで仕様が統合された場合、単一のファイルに一本化できる可能性がある。しかし現時点では、複数のファイルを管理するコストよりも、すべてのプラットフォームで一貫したルールを適用できるメリットの方が大きい。

## 4.6 スキルによるプログレッシブ・ディスクロージャー

### SKILL.mdの設計原則

第4.1節で、マップファイルの肥大化問題とMCP厳選の知見を紹介した。これらの問題に対するより根本的な解決策として、Anthropicが提唱した**スキル（Agent Skills）**のアプローチがある（ANT-10）。

スキルとは、`SKILL.md`ファイルとそれに関連するファイル群で構成される、エージェントの特定能力を表現する単位である。スキルの設計における核心的な原則が、**プログレッシブ・ディスクロージャー（Progressive Disclosure）**——段階的な情報開示——である（LC-15, ANT-10）。

プログレッシブ・ディスクロージャーとは、UIデザインに由来する概念であり、「必要な情報を必要なタイミングで開示する」という設計原則である。この原則をエージェントのスキル設計に適用したのが`SKILL.md`のアプローチである。

具体的な仕組みは以下の通りである。

**ステップ1: YAML前置きのみをデフォルトロード**

エージェントがセッションを開始すると、利用可能なスキルの`SKILL.md`ファイルから**YAML前置き（front matter）のみ**がコンテキストに読み込まれる。前置きには、スキルの名前、概要、対象トリガー条件などの最小限のメタデータが記述される。

```yaml
---
name: database-migration
description: データベースのスキーマ変更とマイグレーション実行を支援する
triggers:
  - マイグレーション
  - スキーマ変更
  - ALTER TABLE
tools_required:
  - sql_executor
  - migration_generator
---
```

**ステップ2: 必要時に完全なSKILL.mdを読み込み**

エージェントがタスクの遂行中にスキルの適用が必要と判断した場合に限り、完全な`SKILL.md`（手順の詳細、コード例、注意事項など）を読み込む。

```markdown
---
name: database-migration
description: データベースのスキーマ変更とマイグレーション実行を支援する
triggers:
  - マイグレーション
  - スキーマ変更
  - ALTER TABLE
tools_required:
  - sql_executor
  - migration_generator
---

# Database Migration Skill

## 手順

1. 現在のスキーマ状態を確認する
   ```bash
   make db-status
   ```

2. マイグレーションファイルを生成する
   ```bash
   make db-generate-migration NAME=<migration_name>
   ```

3. マイグレーション内容をレビューする
   - UP（適用）とDOWN（ロールバック）の両方を確認
   - データ損失が発生しないことを検証

4. テスト環境で実行する
   ```bash
   make db-migrate-test
   ```

5. テストが通過したら本番に適用する

## 注意事項
- カラム削除は必ず2段階で実施する（まず非推奨化、次のリリースで削除）
- 大量データのマイグレーションはバッチ処理で実行する
- ロールバック手順を必ずテストしてからデプロイする
```

[図4.7: SKILL.mdの2段階ロードの仕組み]

この2段階ロードの仕組みにより、エージェントのコンテキストウィンドウにはデフォルトでYAML前置き（数十トークン）のみが占有され、必要な場合にのみ完全なスキル記述（数百〜数千トークン）が追加される。

### トークン効率性

スキルによるプログレッシブ・ディスクロージャーが解決する根本的な問題は、**トークン効率性**（Token Efficiency）である。

LangChainのLance Martin氏は、汎用エージェントが少数のツールで高い効果を発揮する背景を分析している（LC-15）。

- **Claude Code**: 約12個のツールで動作する
- **Manus**: 20未満のツールで動作する

これは一見すると少なく思えるかもしれない。MCPの普及により、数百のツールを接続することが技術的に可能であるにもかかわらず、実際に効果的なエージェントは少数のツールに絞り込んでいる。

その理由は、コンテキストウィンドウの物理的制約にある。ツールの定義はコンテキストに含まれるため、ツール数が増えるほどコンテキストの「残り枠」が減少する。さらに、多数のツールが存在すると、エージェントがどのツールを使うべきか判断するための推論コストも増加する。

Martin氏は、この問題をContext Confusion（コンテキスト混乱）と呼んでいる。多数のツールが似たような機能を提供する場合、エージェントが適切なツールを選択できず、誤ったツールを使用したり、ツール選択に過度なトークンを消費したりする問題である。

スキルのアプローチは、この問題をプログレッシブ・ディスクロージャーで解決する。

```
従来のアプローチ:
┌──────────────────────────────────────┐
│ コンテキストウィンドウ                  │
│ ┌────────────────────────────────┐  │
│ │ ツール定義（100個分）           │  │ ← 大量のトークンを占有
│ │ ■■■■■■■■■■■■■■■■■■■■■■■■■■■ │  │
│ └────────────────────────────────┘  │
│ ┌──────────┐                        │
│ │ 実作業用   │                        │ ← 残り枠が少ない
│ └──────────┘                        │
└──────────────────────────────────────┘

スキルによるアプローチ:
┌──────────────────────────────────────┐
│ コンテキストウィンドウ                  │
│ ┌──────┐                            │
│ │YAML   │ ← 前置きのみ（少量）       │
│ └──────┘                            │
│ ┌────────────────────────────────┐  │
│ │ 実作業用（大きな余裕あり）        │  │ ← 十分な残り枠
│ │                                │  │
│ │  必要時のみスキル詳細をロード     │  │
│ └────────────────────────────────┘  │
└──────────────────────────────────────┘
```

[図4.8: 従来のツールロードとスキルによるプログレッシブ・ディスクロージャーの比較]

### エージェントによる動的スキル生成

プログレッシブ・ディスクロージャーのさらに進んだ活用が、**エージェントによる動的スキル生成**である（LC-15）。

エージェントが新しいタスクに遭遇し、既存のスキルでは対応できないと判断した場合、エージェント自身が新しい`SKILL.md`を生成し、次回以降のセッションで利用可能にするというアプローチである。

```
1. エージェントが新タスクに遭遇
       │
       ▼
2. 既存スキルの検索
       │
       ├── マッチするスキルあり → そのスキルを使用
       │
       └── マッチするスキルなし
              │
              ▼
3. タスクを試行錯誤で完了
       │
       ▼
4. 完了した手順から SKILL.md を自動生成
       │
       ▼
5. 次回の同類タスクでは生成済みスキルを使用
```

この仕組みは、エージェントの学習メカニズムの一形態と捉えることができる。LLMのモデル自体は変更されないが、スキルファイルという「外部知識」が蓄積されることで、エージェントの能力が漸進的に拡張される。これは、第2章で解説したコンテキストエンジニアリングの「Write」操作戦略（外部永続化）（LC-11）の応用であり、エージェントが自らの経験を外部ファイルに書き出し、将来の自分に受け渡すという設計パターンである。

動的スキル生成においても、プログレッシブ・ディスクロージャーの原則は維持される。生成されたスキルはYAML前置き付きの`SKILL.md`フォーマットで保存され、次回以降のセッションでは前置きのみがデフォルトロードされる。

## 4.7 [Hands-on] AGENTS.md / CLAUDE.md の作成演習

### 演習の概要

本節では、ここまでに学んだ設計原則を実際のプロジェクトに適用する演習を行う。読者の手元にあるプロジェクト（または本書のGitHubリポジトリに用意されたサンプルプロジェクト）に対して、`AGENTS.md`を一から作成し、その効果を検証する。

### 準備: プロジェクトの選定

まず、`AGENTS.md`を作成する対象プロジェクトを選定する。以下の条件を満たすプロジェクトが望ましい。

- ある程度のコード量がある（数百行〜数千行以上）
- ビルド・テストの手順が確立している
- コーディング規約が（暗黙的にでも）存在する

もし手元に適切なプロジェクトがない場合は、本書のGitHubリポジトリに用意されたサンプルプロジェクト（TypeScript + Expressの簡易REST API）を使用してほしい。

### ステップ1: プロジェクトの棚卸し

`AGENTS.md`を作成する前に、プロジェクトの情報を棚卸しする。以下のチェックリストに沿って情報を収集する。

```markdown
## 棚卸しチェックリスト

### 基本情報
- [ ] 使用言語・フレームワーク
- [ ] パッケージマネージャー
- [ ] ディレクトリ構造の概要

### ビルド・テスト
- [ ] ビルドコマンド
- [ ] テスト実行コマンド
- [ ] Lint実行コマンド
- [ ] CI/CDパイプラインの有無

### コーディング規約
- [ ] フォーマッタの設定
- [ ] 命名規則
- [ ] エラーハンドリングの方針
- [ ] テスト記述のパターン

### ドメイン知識
- [ ] プロジェクトの目的
- [ ] 主要なデータモデル
- [ ] 外部サービスとの連携
- [ ] 業界固有の制約
```

### ステップ2: AGENTS.mdの骨格を作成

棚卸しの結果をもとに、`AGENTS.md`の骨格を作成する。まずは「テーブル・オブ・コンテンツ」のスタイルで、100行以内を目標に記述する。

```markdown
# AGENTS.md

## Project Overview
[言語]と[フレームワーク]で構築された[プロジェクトの目的]。

## Build & Test
- `[ビルドコマンド]` でビルド
- `[テストコマンド]` でテスト実行
- `[Lintコマンド]` でLint実行

## Code Style
- [フォーマッタ/Linter設定の要点]
- [命名規則]
- [エラーハンドリング方針]

## Architecture
→ docs/architecture.md

## Directory Structure
- [主要ディレクトリとその役割]

## Golden Principles
- [不変ルール1]
- [不変ルール2]
- [不変ルール3]
```

### ステップ3: Golden Principlesの策定

棚卸しの結果と、チームの暗黙知をもとに、Golden Principlesを策定する。以下の指針に従って策定する。

1. **機械的であること**: 判断の余地がないルールとして記述する
2. **検証可能であること**: Lintツールやテストで自動検証できるレベルの具体性を持たせる
3. **意見のあること**: 複数の選択肢がある場合、ひとつを明確に選ぶ
4. **最小限であること**: 最初は5〜10個のルールに絞る。必要に応じて追加する

### ステップ4: 効果の検証

作成した`AGENTS.md`の効果を検証する。以下の手順で検証を行う。

**検証1: Before/After比較**

1. `AGENTS.md`がない状態で、エージェントに簡単なタスク（関数の追加、バグ修正など）を依頼する
2. 生成されたコードのスタイル、テストの有無、規約への準拠を観察する
3. `AGENTS.md`を配置した状態で、同種のタスクを依頼する
4. 生成されたコードが`AGENTS.md`のルールに従っているか比較する

**検証2: 一貫性チェック**

1. `AGENTS.md`を配置した状態で、同じ種類のタスクを3回実行する
2. 3回の結果が一貫したスタイルで生成されているか確認する
3. 一貫していない箇所があれば、`AGENTS.md`のルール記述を改善する

**検証3: エッジケース**

1. `AGENTS.md`に記述されていない状況をわざと作り出す
2. エージェントがどのように振る舞うか観察する
3. 必要に応じてルールを追加する

### ステップ5: 二層構造への拡張（発展課題）

余力がある読者は、以下の発展課題に取り組んでほしい。

1. **ベースレイヤーの作成**: 組織全体で共通に適用するルールを`organization-rules.md`として作成する
2. **カスタマイズレイヤーの作成**: プロジェクト固有のルールを`project-rules.md`として作成する
3. **`AGENTS.md`からの参照**: `AGENTS.md`から両方のルールファイルを参照する構成を作る
4. **優先順位の明示**: 競合した場合の優先順位を`AGENTS.md`に明記する

```markdown
# AGENTS.md

## ルール体系
本プロジェクトのルールは以下の二層構造で管理されている。
競合する場合は、project-rules.md を優先する。

### ベースレイヤー（組織共通）
→ docs/organization-rules.md

### カスタマイズレイヤー（プロジェクト固有）
→ docs/project-rules.md
```

### 演習のポイント

本演習を通じて体得すべき最も重要なポイントは、**「完璧な`AGENTS.md`を一度で作ろうとしない」**ことである。

`AGENTS.md`は生きたドキュメントであり、プロジェクトの進展に応じて継続的に改善されるべきものである。最初のバージョンは必要最小限の内容で構わない。エージェントの実際の挙動を観察し、期待と異なる行動が見られたらルールを追加・修正する。このフィードバックループこそが、ハーネスエンジニアリングにおける「反復的洗練」（LC-13）の実践である。

---

## 本章のまとめ

- **マップファイルの設計哲学**: `AGENTS.md`/`CLAUDE.md`は「マニュアルではなくマップ」として設計する。約100行のテーブル・オブ・コンテンツとし、詳細は`docs/`ディレクトリへ誘導する。肥大化はコンテキスト圧迫の原因となるため、ファイル分割やSQLite + MCPによる動的管理で対策する（OAI-09, STD-02, ANT-13, TLDV-03）

- **Golden Principles（不変ルール）**: ルールは「機械的で意見のある」ものとして策定する。検証可能な具体性を持たせ、コードベースの一貫性とエージェント実行間の予測可能性を確保する（BLG-11）

- **二層構造の設計**: ベースレイヤー（変更不可の標準ルール）とカスタマイズレイヤー（プロジェクト固有の拡張ルール）を分離し、明確な優先順位設計で競合を解決する。カスタマイズレイヤーはベースレイヤーを上書きではなく補完・拡張する

- **ガードレールの多層防御**: Constitutional AI（ANT-17）を理論的基盤とし、Constitutional Classifiers（ANT-18）、Policy-as-Prompt（ACD-09）、Middleware（TLDV-02）、ガードレールエージェント群（TLDV-06）、サンドボックス（ANT-07）を組み合わせた多層防御を構築する。リスクベースの防御配置原則に従い、全箇所ではなくリスクが大きい部分に限定して防御を施す

- **オープン標準化の進展**: AAIF（OAI-06, STD-01）のもとで`AGENTS.md`とMCPの標準化が進行中。現時点では`AGENTS.md`と`CLAUDE.md`の両方を維持し、すべてのプラットフォームで一貫したルールを適用するのがベストプラクティスである

- **プログレッシブ・ディスクロージャー**: SKILL.mdのYAML前置きのみをデフォルトロードし、必要時にのみ完全な内容を読み込む設計により、トークン効率性を最大化する。汎用エージェントが少数のツール（Claude Code約12個、Manus 20未満）で高い効果を発揮する背景にはこの原則がある（LC-15, ANT-10）

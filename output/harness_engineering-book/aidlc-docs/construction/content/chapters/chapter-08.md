# 第8章: Agent-First開発への組織移行

> **読者レベル**: ★☆☆（入門 — 前提知識不要）
> **想定ページ数**: 約30ページ

ここまで本書は、ハーネスエンジニアリングの「なぜ（WHY）」「何を（WHAT）」「どう作るか（HOW）」を技術的な観点から解説してきた。しかし、どれほど優れたハーネスを設計しても、それを活用する組織の側が旧来の開発プロセスに縛られたままでは、Agent-First開発の真価は発揮されない。

2026年2月、OpenAIのRyan Lopopolo氏が報告した実験は、この点を鮮明に示している。7人のエンジニアが5ヶ月間にわたって一切の手書きコードなしで約100万行・1,500件のプルリクエストを生み出した事実は、技術的な偉業であると同時に、**開発プロセスそのものの再設計**が不可欠であることを証明した（OAI-01）。従来型の重量級コードレビュー、長期間のフィーチャーブランチ、手動マージの慣行をそのまま維持していたら、この生産性は実現し得なかったのである。

本章では、Agent-First開発を組織に導入するための具体的な方法論を示す。PRフローの変革、リポジトリ設計の再考、チーム体制の再編、そして段階的な導入ロードマップまでを包括的にカバーする。読者が所属する組織の規模やフェーズに関わらず、明日から着手できるアクションを提示することが本章の目標である。

## 8.1 「Humans Steer, Agents Execute」

### 人間とエージェントの新しい役割分担

Agent-First開発の核心は、**"Humans Steer, Agents Execute"**（人間が舵を取り、エージェントが実行する）という原則にある（OAI-01）。この原則は、人間とAIエージェントの役割を根本から再定義するものである。

従来のソフトウェア開発では、エンジニアの価値は「コードを書く能力」で測られてきた。アルゴリズムの選択、データ構造の設計、言語固有のイディオムの活用——これらはすべて「書く」行為に集約される。しかしAgent-First開発では、エンジニアの価値は**「意図を明確化し、エージェントが自律的に働ける環境を設計する能力」**に移行する。

この変化を理解するために、OpenAIの実験が示した具体的な役割分担を見てみよう（OAI-01）。

```
┌──────────────────────────────────────────────────────────┐
│                    人間（Steer）                           │
│                                                          │
│  ・目標の定義と分解                                        │
│  ・リポジトリの構造設計                                     │
│  ・ポリシーファイル（AGENTS.md / CLAUDE.md）の整備           │
│  ・コードレビューと品質判断                                 │
│  ・フィードバックループの構築                                │
│  ・アーキテクチャ上の意思決定                                │
├──────────────────────────────────────────────────────────┤
│                   エージェント（Execute）                   │
│                                                          │
│  ・コードの生成と実装                                       │
│  ・テストの作成と実行                                       │
│  ・リファクタリングの実施                                    │
│  ・PRの作成                                               │
│  ・ドキュメントの生成                                       │
│  ・自己検証ループの実行                                     │
└──────────────────────────────────────────────────────────┘
```

[図8.1: Agent-First開発における人間とエージェントの役割分担]

ここで重要なのは、この役割分担が「人間は何もしない」という意味ではないことである。むしろ、人間の仕事はより高度に、より戦略的になる。コードの1行1行を書く代わりに、**エージェントが100行のコードを正しく生成するための環境をどう設計するか**を考えるのが新しい仕事である。

### 深さ優先のワークフロー

Agent-First開発では、ワークフローの構造も変わる。Lopopolo氏が報告した手法は、**深さ優先（depth-first）**のアプローチである（OAI-01）。

従来の開発では、タスクを幅広く並行して進めるのが一般的であった。複数のフィーチャーブランチを同時に開発し、定期的にメインブランチへマージする。しかしAgent-First開発では、以下のフローが基本となる。

```
1. 目標を明確に定義する
   └→ 2. 目標を小さなサブタスクに分解する
       └→ 3. 各サブタスクに対してエージェントを構築・指示する
           └→ 4. エージェントの出力をレビューし、マージする
               └→ 5. 次のサブタスクへ進む
```

このフローの特徴は、**1つのタスクに深くコミットし、完了させてから次に進む**点にある。エージェントが生成したPRをレビューし、問題があればフィードバックを与えて再生成させ、品質基準を満たしてからマージする。この「深く狭く」のアプローチが、品質と速度の両立を可能にする。

なぜ深さ優先が有効なのか。理由は2つある。第一に、エージェントのコンテキストウィンドウは有限であり、複数のタスクを同時に処理すると品質が劣化する（第6章で詳述した「コンテキストロット」の問題）。1つのタスクに集中させることで、エージェントの推論能力を最大限に活用できる。第二に、小さな単位で完了・マージすることで、コンフリクトのリスクを最小化し、問題の早期発見が可能になる。

### 1人あたり3.5 PR/日の生産性

OpenAIの実験において、7人のエンジニアチームは5ヶ月間で1,500件のPRをマージした。これは1人あたり1日平均約3.5件のPRに相当する（OAI-01）。

この数字の意味を理解するために、従来の開発プロセスと比較してみよう。一般的なソフトウェア開発チームでは、1人のエンジニアが1日にマージできるPRは多くても1〜2件であり、複雑なフィーチャーであれば数日から1週間以上かかることも珍しくない。3.5 PR/日という数字は、従来比で2〜3倍以上の生産性を意味する。

しかし、この生産性はPRの性質の変化なしには理解できない。後述するように、Agent-First開発におけるPRは従来のPRとは根本的に異なるものである。短命で、焦点が絞られ、レビューコストが低い。この新しいPRのあり方を理解することが、Agent-First開発への移行の第一歩となる。

さらに注目すべきは、Lopopolo氏の報告には別の事例も含まれていることである。ある3人チームが同じくエージェント駆動の手法で100万行のコードベースを構築したケースが言及されている（OAI-01）。7人チームの事例が「少人数でも大規模な成果を出せる」ことを示しているのに対し、3人チームの事例は**「さらに少人数でも、適切なハーネスがあれば同等の規模に到達し得る」**ことを示唆している。チーム規模と成果の関係が、Agent-First開発によって根本的に変わりつつあるのである。

## 8.2 PRフローの変革

### 短命PR: 数分で生成、レビュー、マージ

Agent-First開発における最も劇的な変化の1つが、プルリクエスト（PR）のライフサイクルの短縮である。

従来の開発プロセスでは、PRは比較的「長命」であった。フィーチャーブランチで数日間にわたって開発を進め、レビューに1〜2日かけ、フィードバックへの対応にさらに時間を要する。大規模なPRほどレビューが困難になり、結果としてリードタイムが長期化するという悪循環に陥りがちである。

Agent-First開発では、このパターンが根本から変わる。PRの典型的なライフサイクルは以下のようになる。

```
┌─────────┐    ┌──────────┐    ┌─────────┐    ┌─────────┐
│ 指示作成  │ →  │ エージェント│ →  │ レビュー  │ →  │ マージ   │
│ (数分)   │    │ 実装      │    │ (数分)   │    │ (即時)   │
│          │    │ (数分〜   │    │          │    │          │
│          │    │  数十分)  │    │          │    │          │
└─────────┘    └──────────┘    └─────────┘    └─────────┘
```

[図8.2: Agent-First開発におけるPRのライフサイクル]

この「短命PR」の考え方は、OpenAIの実験から導き出された知見である。1,500件のPRを5ヶ月間で処理するということは、平均して1日あたり約15件のPRがチーム全体でマージされていたことを意味する（OAI-01）。各PRが数日間も未マージのまま放置されていたら、コンフリクトの嵐で開発は成立しなかったはずである。

短命PRが機能するための前提条件は、PRの粒度が小さいことである。1つのPRが1つの明確な変更に集中していれば、レビューは数分で完了する。「このPRは何を変更しているのか」が一目で理解でき、テストが通っていることを確認すれば、マージの判断は容易である。

### 最小限のブロッキングマージゲート

短命PRの効果を最大化するには、マージプロセスにおけるブロッキング要因を最小限に抑える必要がある。従来のゲートキーピングプロセス——2名以上の承認が必要、セキュリティレビュー必須、アーキテクチャレビュー委員会の承認——は、Agent-First開発の文脈では過剰な摩擦を生むことがある。

これは品質管理を放棄するという意味ではない。むしろ、品質管理の方法を変えるということである。ブロッキングマージゲートの代わりに、以下のアプローチが有効である。

**1. 自動化された品質チェックの強化**

人間のレビュアーが時間をかけて確認していた項目の多くは、自動化できる。リンター、型チェック、テストスイート、コーディング規約の自動検証——これらをCI（継続的インテグレーション）パイプラインに組み込むことで、人間のレビューは「自動では判断できない設計判断」に集中できる。

```yaml
# .github/workflows/agent-pr-checks.yml
# Agent-First開発向けのPRチェック設定例
name: Agent PR Checks
on: [pull_request]
jobs:
  automated-gates:
    runs-on: ubuntu-latest
    steps:
      - name: Lint Check
        run: npm run lint
      - name: Type Check
        run: npm run type-check
      - name: Unit Tests
        run: npm test
      - name: Coverage Threshold
        run: npm run coverage -- --min-coverage 80
      - name: Security Scan
        run: npm audit --production
      - name: Policy Compliance
        run: ./scripts/check-agents-md-compliance.sh
```

**2. リスクベースのレビュー戦略**

すべてのPRに同じレベルのレビューを要求するのではなく、変更のリスクに応じてレビューの深さを変える。

| リスクレベル | 対象例 | レビュー方針 |
|-------------|--------|-------------|
| 低 | ドキュメント更新、テスト追加、小規模リファクタリング | 自動チェック通過で自動マージ |
| 中 | 新機能の追加、既存ロジックの修正 | 1名のレビュー + 自動チェック |
| 高 | アーキテクチャ変更、セキュリティ関連、データモデル変更 | 2名のレビュー + 専門家確認 |

[表8.1: リスクベースのレビュー戦略]

このアプローチにより、低リスクな変更は高速にマージされ、高リスクな変更には適切な注意が払われる。Agent-First開発では、エージェントが生成するPRの大半は低〜中リスクに分類されるため、全体のスループットが大幅に向上する。

### 自動マージの条件設計

Agent-First開発の生産性を最大化するためには、一定の条件を満たしたPRを自動的にマージする仕組みが重要になる。自動マージの条件は、組織のリスク許容度に応じて設計する必要がある。

以下は、段階的に自動マージを導入する際の条件設計の例である。

**Phase 1: 保守的な自動マージ**

```
自動マージ条件:
  AND:
    - すべてのCIチェックが通過
    - テストカバレッジが既存水準を下回らない
    - 変更ファイル数が5以下
    - セキュリティ関連ファイルへの変更なし
    - 1名以上のレビュー承認
```

**Phase 2: 成熟した自動マージ**

```
自動マージ条件:
  AND:
    - すべてのCIチェックが通過
    - テストカバレッジが80%以上
    - AGENTS.md / CLAUDE.md で定義されたポリシーに準拠
    - セキュリティスキャンで脆弱性なし
  OR:
    - 変更がドキュメントのみ
    - 変更がテストの追加のみ
    - エージェント自身の自己検証ループが完了
```

自動マージの導入は、段階的に進めることが鉄則である。最初は保守的な条件で始め、運用実績を蓄積しながら条件を緩和していく。本番環境での問題発生率をモニタリングし、自動マージが品質に悪影響を与えていないことを定量的に確認することが重要である。

本番グレードのワークフロー設計に関する研究（ACD-05）では、エージェンティックAIワークフローにおいて、自動化の範囲を段階的に拡大するアプローチが推奨されている。この知見はPRフローの自動化にも直接適用可能である。

## 8.3 継続的自動リファクタリング

### バックグラウンドタスクによる逸脱スキャン

Agent-First開発の強力な利点の1つは、**リファクタリングを継続的かつ自動的に実施できる**ことである。従来の開発では、技術的負債の返済は「時間があるとき」に行うものであり、しばしば後回しにされてきた。しかしAgent-First開発では、バックグラウンドでエージェントを稼働させ、コードベースの品質を常時監視・改善することが可能になる。

Anthropicの事例では、16並列のClaudeインスタンスを稼働させ、約2,000セッションを通じてRustベースのCコンパイラを構築した（ANT-12）。この並列実行の手法は、コードの新規生成だけでなく、既存コードの継続的なリファクタリングにも応用できる。

バックグラウンドタスクによる逸脱スキャンの基本的なフローは以下の通りである。

```
┌─────────────┐     ┌──────────────┐     ┌─────────────┐
│ ポリシー定義  │ →   │ 逸脱スキャン   │ →   │ レポート生成  │
│              │     │              │     │              │
│ AGENTS.md    │     │ ・コーディング │     │ ・逸脱箇所   │
│ CLAUDE.md    │     │   規約の検証   │     │ ・修正提案   │
│ アーキテクチャ │     │ ・アーキテクチャ│     │ ・優先度     │
│ ドキュメント  │     │   パターンの   │     │              │
│              │     │   適合確認     │     │              │
│              │     │ ・依存関係の   │     │              │
│              │     │   分析         │     │              │
└─────────────┘     └──────────────┘     └─────────────┘
                           │
                           ▼
                    ┌──────────────┐
                    │ リファクタリング│
                    │ PRの自動生成   │
                    └──────────────┘
```

[図8.3: 継続的自動リファクタリングのフロー]

このフローにおいて、エージェントはポリシーファイル（`AGENTS.md`や`CLAUDE.md`、第4章で詳述）をガイドラインとして使用し、コードベース全体を走査する。ポリシーからの逸脱を検出した場合、修正を提案するPRを自動的に生成する。

### 品質グレードの自動更新

継続的リファクタリングをさらに体系化するためには、コードベースの品質を定量的に評価し、追跡する仕組みが必要である。以下のような品質グレードの体系を導入することで、リファクタリングの優先順位を客観的に決定できる。

| グレード | 基準 | 状態 |
|---------|------|------|
| A | すべてのポリシーに完全準拠、テストカバレッジ90%以上 | 理想状態 |
| B | 軽微な逸脱あり、テストカバレッジ80%以上 | 許容範囲 |
| C | 中程度の逸脱あり、テストカバレッジ60%以上 | 改善必要 |
| D | 重大な逸脱あり、テストカバレッジ60%未満 | 緊急対応 |

[表8.2: コードベースの品質グレード体系]

エージェントはバックグラウンドで定期的にコードベースをスキャンし、モジュールごとの品質グレードを自動更新する。グレードが低下した場合、改善のためのリファクタリングPRを自動生成する。

LangChainが報告したLoopDetectionMiddleware（LC-08）の考え方——繰り返しの非生産的なパターンを検出して介入する——は、このような品質監視にも応用できる。同一のポリシー違反が繰り返し検出される場合、根本原因の分析と構造的な解決策の提案にエスカレーションするのである。

### 1分以内にレビュー可能なターゲットリファクタリングPR

自動リファクタリングPRが有効に機能するためには、**人間が1分以内にレビューできる粒度**に保つことが不可欠である。

大規模なリファクタリングPRは、たとえ機械的な変更であっても、レビュアーに心理的な負担を与える。「変更量が多いので後で見よう」と後回しにされ、結局マージされないまま放置されるリスクがある。

ターゲットリファクタリングPRの設計原則は以下の通りである。

**1. 単一の変更カテゴリに限定する**

1つのPRが「変数名の統一」「未使用インポートの削除」「エラーハンドリングの追加」を同時に行うべきではない。変更カテゴリごとにPRを分割することで、レビュアーは「何が変わったか」を瞬時に理解できる。

**2. 変更の根拠を自動生成する**

エージェントがリファクタリングPRを生成する際、PR本文に以下の情報を自動的に含めることが望ましい。

```markdown
## 変更の目的
AGENTS.md セクション3.2「エラーハンドリング規約」への準拠

## 変更内容
- `src/api/handler.ts` の try-catch ブロックにカスタムエラー型を適用
- エラーメッセージにコンテキスト情報を追加

## 影響範囲
- 変更ファイル: 1
- 変更行数: +8 / -3
- テスト結果: 全パス（既存テスト12件 + 新規テスト2件）

## ポリシー準拠
- [x] AGENTS.md 3.2 準拠
- [x] テストカバレッジ低下なし
- [x] 既存APIの互換性維持
```

**3. 安全なリファクタリングから始める**

自動リファクタリングの対象は、最初は機械的で安全な変更に限定する。コードフォーマットの統一、未使用変数の削除、型アノテーションの追加など、セマンティクスを変えない変更から始め、信頼が蓄積されてから、より複雑なリファクタリングに範囲を広げていく。

この段階的アプローチは、Hugo Bowne-Anderson氏が提唱した3原則——「再アーキテクチャを受け入れる」「ビルダーにとってのより高い抽象レベル」「シンプルさは不可欠」——と整合する（BLG-06）。シンプルなリファクタリングから始め、徐々に抽象度を上げていくことで、チーム全体がAgent-First開発のリズムに慣れていくのである。

## 8.4 リポジトリ設計のベストプラクティス

### リポジトリを「記録のシステム」として設計

Agent-First開発において、リポジトリはコードの格納場所以上の意味を持つ。リポジトリは**「記録のシステム」（System of Record）**——すなわち、プロジェクトに関するすべての意思決定、設計方針、ドメイン知識が集約される唯一の信頼できる情報源——として設計されるべきである。

なぜリポジトリが記録のシステムになる必要があるのか。それは、エージェントがリポジトリを読んで動くからである。エージェントは人間のように口頭での引き継ぎを受けたり、Slackの過去ログを検索したりしない。リポジトリに書かれている情報が、エージェントにとっての世界のすべてである。したがって、リポジトリに書かれていないことは、エージェントにとって存在しないのと同じである。

OpenAIの実験（OAI-01）では、エンジニアたちがコードを一切手書きしない代わりに、リポジトリのドキュメント体系を徹底的に整備した。ドメイン知識、アーキテクチャの意思決定、コーディング規約、テスト方針——これらをすべてリポジトリ内のドキュメントとして体系化することで、エージェントは「人間が口頭で伝える必要のある知識」なしに開発を進められるようになった。

Anthropicが公開したClaude Codeのベストプラクティス（ANT-13）も、この考え方を裏付けている。`CLAUDE.md`をプロジェクトのルートに配置し、エージェントが最初に読む情報として、プロジェクトの構造、ビルドコマンド、テスト方針、コーディング規約を網羅的に記述することが推奨されている。

### docs/ ディレクトリの構造化

リポジトリを記録のシステムとして機能させるためには、`docs/`ディレクトリの構造化が不可欠である。以下は、Agent-First開発に最適化されたドキュメント構造の例である。

```
project-root/

├── CLAUDE.md                    # Claude Code固有の設定
├── docs/
│   ├── architecture/
│   │   ├── overview.md          # アーキテクチャ全体像

```

[図8.4: Agent-First開発に最適化されたリポジトリ構造]

この構造において、`AGENTS.md`（または`CLAUDE.md`）はエージェントが最初にアクセスする「マップファイル」として機能する。第4章で解説したように、マップファイルは「マニュアルではなくマップ」——すなわち、エージェントが必要な情報に到達するためのナビゲーションガイドである。

マップファイルの典型的な構造は以下の通りである。

```markdown
# AGENTS.md

## Project Overview
このプロジェクトは... (2-3行の概要)

## Architecture
詳細は docs/architecture/overview.md を参照。

## Key Directories
- src/api/       - REST APIエンドポイント
- src/core/      - ドメインロジック
- src/infra/     - インフラストラクチャ層
- tests/         - テストスイート

## Build & Test
- ビルド: `npm run build`
- テスト: `npm test`
- リント: `npm run lint`

## Coding Standards
docs/standards/ を参照。特に以下に注意:
- エラーハンドリング: docs/standards/error-handling.md
- 命名規則: docs/standards/naming-conventions.md

## Domain Knowledge
ドメイン固有の用語は docs/domain/glossary.md を参照。
```

このマップファイルは意図的に簡潔に保つ。100行程度のテーブル・オブ・コンテンツとして機能し、詳細な情報は`docs/`配下の個別ファイルに委譲する。第4章で論じたマップファイルの肥大化問題（TLDV-03）を回避するためである。

### アーキテクチャドキュメント: ドメインとパッケージ階層のマッピング

Agent-First開発で特に重要なドキュメントが、**ドメインとパッケージ階層のマッピング**である。エージェントが新しいコードを書く際、「この機能はどのパッケージに属するのか」「このドメインオブジェクトはどのモジュールで定義されているのか」を正確に判断する必要がある。

アーキテクチャドキュメント（`docs/architecture/package-map.md`）の例を示す。

```markdown
# パッケージ階層マップ

## ドメインとパッケージの対応

| ドメイン概念 | パッケージ | 責務 |
|-------------|-----------|------|
| ユーザー管理 | src/core/user/ | ユーザーの作成・認証・権限管理 |
| 注文処理 | src/core/order/ | 注文の受付・検証・処理 |
| 決済 | src/core/payment/ | 決済処理の抽象化 |
| 通知 | src/infra/notification/ | メール・Slack通知の送信 |

## 依存関係ルール

- src/core/ は src/infra/ に依存してはならない
- src/api/ は src/core/ のインターフェースのみを参照する
- src/infra/ は外部サービスとの通信を担当する

## 新しいモジュールの追加ルール

1. ドメイン概念を特定する
2. 上記マッピングに基づいて適切なパッケージを決定する
3. 既存のモジュールと同じファイル構造（index.ts, types.ts, *.test.ts）に従う
4. 依存関係ルールを遵守する
```

このようなドキュメントがリポジトリに存在することで、エージェントは新しい機能を追加する際に、既存のアーキテクチャパターンに一貫して従うことができる。ドキュメントが存在しない場合、エージェントはコードベースを探索して推測するしかなく、一貫性のない構造が生まれるリスクが高まる。

LangChainが提唱した「コードファーストのアプローチ」（LC-07）は、このリポジトリ設計の考え方とも整合する。ビジュアルワークフロービルダーのようなGUI中心のツールではなく、コードとドキュメントをリポジトリに集約し、バージョン管理・レビュー・自動テストの対象とすることで、Agent-First開発の基盤が強固になるのである。

## 8.5 チーム体制とスキル要件の変化

### 「コードを書く人」から「環境を設計する人」へ

Agent-First開発への移行は、チームメンバーに求められるスキルセットを根本的に変える。もっとも端的に言えば、エンジニアの役割は**「コードを書く人」から「環境を設計する人」**へと変化する。

この変化は、OpenAIの実験（OAI-01）で明確に観察されたものである。5ヶ月間のゼロ手書きコード実験を通じて、エンジニアたちが実際に行っていた作業は以下のようなものであった。

- **意図の明確化**: 何を実現したいのかを、エージェントが正確に理解できるレベルまで言語化する
- **環境の整備**: リポジトリの構造、ポリシーファイル、テスト方針を体系的に設計する
- **品質の判断**: エージェントが生成したコードの品質を評価し、フィードバックを与える
- **アーキテクチャの意思決定**: 全体的な設計方針を決定し、ドキュメントとして明文化する
- **フィードバックループの構築**: エージェントの出力品質を向上させるための仕組みを設計する

注目すべきは、これらの活動のどれ1つとして「コードを書く」ことを含んでいない点である。しかし、これらは従来のコーディングよりもはるかに高度な判断力と経験を要求する。

### ハーネスエンジニアの3軸スキルセット

LangChainは、エージェントエンジニアリングという新しい専門分野において、3つの必須スキルセットを定義している（LC-13）。

```
                    Product thinking
                    （プロダクト思考）
                         ▲
                        / \
                       /   \
                      /     \
                     /       \
                    /    ★    \
                   /           \
                  /             \
                 ▼               ▼
         Engineering         Data science
       （エンジニアリング）    （データサイエンス）
```

[図8.5: ハーネスエンジニアの3軸スキルセット]

**1. Product thinking（プロダクト思考）**

プロダクト思考は、エージェントが「何を」実現すべきかを定義する能力である。具体的には以下のスキルが含まれる。

- **プロンプト設計**: エージェントに対する指示を、意図が正確に伝わるように構成する能力。曖昧な指示をエージェントに与えると、曖昧な出力が返ってくる。プロンプト設計は、自分の意図を精確に言語化する訓練である
- **ユーザーニーズの理解**: エンドユーザーが何を必要としているかを深く理解し、それをエージェントの動作仕様に翻訳する能力
- **評価基準の定義**: エージェントの出力が「良い」のか「悪い」のかを判断するための基準を、定量的かつ明確に定義する能力

**2. Engineering（エンジニアリング）**

エンジニアリングは、エージェントが「どう」動作するかを設計・実装する能力である。

- **ツール構築**: エージェントが使用するツール（MCP、API統合、ファイルシステム操作など）を設計・実装する能力
- **UI/UX設計**: エージェントとのインタラクションを人間にとって理解しやすく設計する能力
- **堅牢なランタイムの構築**: エージェントが長時間安定して動作するための実行環境を設計する能力。チェックポインティング、エラーリカバリ、コンパクション（第6章で詳述）の実装が含まれる

**3. Data science（データサイエンス）**

データサイエンスは、エージェントの動作を定量的に評価・改善する能力である。

- **パフォーマンス測定**: エージェントの出力品質、レイテンシ、コストを体系的に測定する能力
- **エラー分析**: 失敗パターンを特定し、根本原因を分析する能力。第7章で解説したCLASSic次元（Cost, Latency, Accuracy, Security, Stability）に基づく評価が有効である
- **A/Bテスト**: ハーネスの変更がエージェントの性能にどう影響するかを統計的に検証する能力

従来のソフトウェアエンジニアリングでは、これら3つのスキルが1人に同時に求められることは稀であった。プロダクトマネージャーがプロダクト思考を担当し、エンジニアが実装を担当し、データサイエンティストが分析を担当するという分業が一般的であった。しかしエージェントエンジニアリングでは、**build → test → ship → observe → refine**のサイクルを高速に回すために、この3つの視点を統合的に持つことが不可欠なのである（LC-13）。

### 「本番環境こそが主要な学習環境」: ship over perfection

LangChainが強調する重要な原則が、**「本番環境こそが主要な学習環境」**（Production is the primary learning environment）であるという考え方である（LC-13）。

従来のソフトウェア開発では、「完璧に近い状態でリリースする」ことが美徳とされてきた。しかしエージェントエンジニアリングでは、エージェントの動作が本質的に非決定論的であるため、本番環境での挙動を事前に完全に予測することは不可能である。

したがって、アプローチは逆転する。

```
従来のアプローチ:
  設計 → 実装 → テスト → テスト → テスト → リリース → 運用

Agent-Firstのアプローチ:
  設計 → 実装 → 基本テスト → リリース → 観察 → 改善 → 観察 → 改善 → ...
```

「ship over perfection」（完璧さよりも出荷を優先する）という原則は、無責任に低品質なものをリリースすることを意味するのではない。基本的な品質基準を満たした上で早期にリリースし、**本番環境からのフィードバックに基づいて反復的に改善する**ことを意味する。

この考え方は、LangChainの調査結果とも整合する。57.3%の組織が既にエージェントを本番運用しているという事実（LC-05）は、業界全体が「本番での学習」を実践していることを示唆している。89%の組織がオブザーバビリティ（Observability）を実装しているという事実（LC-05）は、本番環境での動作を詳細に観察し、改善に活かす仕組みが広く認識されていることを裏付ける。

本番環境での学習を効果的に行うために必要な要素は以下の3つである。

**1. オブザーバビリティの実装**

エージェントの動作を詳細にトレースし、何がうまくいき、何が失敗しているかを可視化する。OpenTelemetryベースのトレーシング（LC-14）が業界標準として確立されつつある。

**2. 段階的なロールアウト**

全ユーザーに一度にリリースするのではなく、カナリアリリースやフィーチャーフラグを使って段階的に展開する。問題が発生した場合に即座にロールバックできる体制を整える。

**3. フィードバックループの自動化**

本番環境でのエラーパターンを自動的に収集・分類し、ハーネスの改善に反映するフィードバックループを構築する。LangChainが報告したトレーシングベースのフィードバック（LC-08）は、この自動化の具体例である。

### OpenAI報告: 3人チームで100万行の事例

OpenAIのLopopolo氏が報告したもう1つの事例として、3人チームによる100万行のコードベース構築がある（OAI-01）。この事例は、Agent-First開発がチーム規模と成果物の規模の関係を根本的に変える可能性を示している。

従来の開発における「チーム規模と生産性の関係」は、ブルックスの法則（Fred Brooks, "The Mythical Man-Month", 1975）で知られるように、人数を増やしても生産性は線形には伸びず、コミュニケーションコストの増大によりむしろ低下することがある。しかしAgent-First開発では、人間のチームメンバーが増えることによるコミュニケーションコストの問題が大幅に軽減される。エージェントはリポジトリ上のドキュメントを読んで動くため、人間同士の口頭での調整が減少するのである。

3人チームで100万行のコードベースを構築できた背景には、以下の要因がある。

- **リポジトリを記録のシステムとして徹底的に整備した**: チームメンバー間の暗黙知が最小化された
- **エージェントの並列実行を活用した**: 人間が1人でレビューに集中している間にも、別のエージェントが次のタスクを進めていた
- **短命PRのフローを確立した**: 小さな単位で頻繁にマージすることで、3人でもコンフリクトなく大規模な開発が可能になった

この事例は、スタートアップや少人数チームにとって特に励みになるものである。潤沢なエンジニアリングリソースがなくても、適切なハーネスとワークフローの設計があれば、大規模な成果を生み出せることを実証しているからである。

### スペック駆動開発（SDD）: 「人間はコード書くな」ポリシー

Agent-First開発の考え方をさらに先鋭化した実践例が、**スペック駆動開発（SDD: Spec Driven Development）**である。これは「人間はコード書くな」というポリシーを組織的に実践する手法であり、AIコーディング道場コミュニティで紹介された事例である（TLDV-04）。

SDDの基本的なフローは以下の通りである。

```
┌──────────────────────────────────────────────────────────────┐
│                 スペック駆動開発（SDD）のフロー                  │
│                                                              │
│  ┌─────────┐    ┌────────────┐    ┌──────────────────┐       │
│  │  PRD作成  │ →  │ タスク分解   │ →  │ エージェント実装    │      │
│  │ (人間)   │    │ (人間+AI)   │    │ (エージェント)     │      │
│  └─────────┘    └────────────┘    └──────────────────┘       │
│       │               │                    │                 │
│  ・要件定義        ・機能を小さな         ・コード生成          │
│  ・ユーザー          タスクに分割        ・テスト作成          │
│    ストーリー      ・優先順位の          ・PR作成             │
│  ・受入基準          決定              ・自己検証            │
│                   ・依存関係の                               │
│                     整理                                    │
│                                                              │
│  ┌──────────────────────────────────────────────────────┐     │
│  │ レビュー → フィードバック → 再生成 → マージ (人間)      │     │
│  └──────────────────────────────────────────────────────┘     │
└──────────────────────────────────────────────────────────────┘
```

[図8.6: スペック駆動開発（SDD）のフロー]

SDDの核心は、人間の仕事を**PRD（Product Requirements Document）の作成とタスク分解**に限定することにある。コードの実装はすべてエージェントに委ね、人間はレビューとフィードバックに専念する。これはまさに「Humans Steer, Agents Execute」の現場実装である。

SDDの実践で得られた重要な知見がある。

**1. PRDの品質がすべてを決める**

エージェントに曖昧な指示を与えると、曖昧な出力が返ってくる。SDDの成功は、PRDの品質に直結する。要件が精確に、かつエージェントが理解できる形で記述されていることが前提条件である。

**2. タスク分解の粒度が品質と速度のバランスを決める**

タスクが大きすぎるとエージェントのコンテキストが不足し、品質が低下する。逆にタスクが小さすぎるとオーバーヘッドが増大する。適切な粒度は、エージェントが1回のセッションで完了できる範囲——具体的には、1つのPRで完結する範囲——である。

**3. MCPの厳選が重要**

SDDの実践者から報告された重要な知見として、MCP（Model Context Protocol）の厳選がある（TLDV-04）。MCPサーバーを追加するとエージェントのツール定義がコンテキストメモリを消費するため、作業に必要なMCPのみを有効化するべきであるという知見である。これは第4章で論じたMCP厳選の原則と一致する。

SDDは、すべてのチームに即座に導入できる手法ではない。エージェントの品質がPRDの品質に強く依存するため、要件定義のスキルが成熟していないチームでは、かえって生産性が低下するリスクがある。後述する段階的導入ロードマップ（8.6節）で、SDDへの移行をどのように段階的に進めるかを解説する。

> **Column: オーバーヘッドの問題 — すべてをエージェントに任せるべきか**
>
> SDDの「人間はコード書くな」ポリシーは強力な原則であるが、万能ではない。LangChainのOpen SWEプロジェクトは、「シンプルな1行修正には最適でない」と正直に認めている（LC-17）。エージェントの起動、コンテキストの読み込み、計画の立案、テストの実行——これらのオーバーヘッドは、たとえ1行の変更であっても発生する。
>
> 経験則として、以下の判断基準が有用である。
>
> - **変更に30秒以上の思考が必要な場合**: エージェントに任せる価値がある
> - **変更が機械的で自明な場合（例: typoの修正、import文の追加）**: 手で書いた方が速い場合もある
> - **変更がアーキテクチャに関わる場合**: 必ずエージェントに設計意図を伝えてから実行させる
>
> 「Humans Steer, Agents Execute」は「人間は一切コードを触るな」という教条ではない。人間が最も価値を発揮できる活動に集中するための指針として理解すべきである。

## 8.6 段階的導入のロードマップ

Agent-First開発への移行は、一夜にして達成できるものではない。組織の規模、文化、既存のプロセスに応じて、段階的に進める必要がある。本節では、3つのフェーズからなる導入ロードマップを提示する。

### Phase 1: 個人での試行（1〜4週間）

**目標**: 1人のエンジニアがAgent-First開発の基本を体験し、有効性を実感する。

**ステップ1: `AGENTS.md` / `CLAUDE.md` の導入**

まず、既存のプロジェクトに`AGENTS.md`（OpenAI Codex向け）または`CLAUDE.md`（Claude Code向け）を導入する。最初のマップファイルは完璧である必要はない。以下の最小構成から始める。

```markdown
# AGENTS.md

## Project
[プロジェクト名]: [1-2文の概要]

## Stack
- 言語: [メイン言語]
- フレームワーク: [使用フレームワーク]

## Build & Test
- ビルド: [コマンド]
- テスト: [コマンド]

## Key Rules
- [最も重要なコーディング規約3-5個]
```

このファイルを作成し、エージェント（Codex CLI、Claude Code、Cursorなど）にコーディングタスクを依頼してみる。エージェントの出力がプロジェクトの規約に合っているかを確認し、不足している情報があればマップファイルに追記していく。

**ステップ2: 小さなタスクから始める**

最初のAgent-First開発タスクとして、以下のような低リスクなものから始める。

- テストの追加（既存機能に対するユニットテストの拡充）
- ドキュメントの生成（コードコメント、README、API仕様）
- 小規模なリファクタリング（変数名の統一、型アノテーションの追加）
- バグ修正（明確に再現手順が定義されているもの）

**ステップ3: 学びを記録する**

Phase 1での学びを体系的に記録する。特に以下の点に注意を払う。

- エージェントがうまく動いたケースと、そうでなかったケース
- マップファイルに追加が必要だった情報
- エージェントの出力に対して繰り返し同じフィードバックを与えたパターン（これはマップファイルに反映すべきルール候補である）

```
┌─────────────┐    ┌─────────────┐    ┌─────────────┐
│ マップファイル │ →  │ エージェント  │ →  │ 出力の評価   │
│ 作成・更新    │    │ によるタスク  │    │ と学びの     │
│              │    │ 実行         │    │ 記録         │
└─────────────┘    └─────────────┘    └─────────────┘
       ▲                                     │
       └─────────── フィードバック ──────────────┘
```

**成功基準**: Phase 1の終了時に、以下の状態が達成されていることを確認する。

- [ ] `AGENTS.md` / `CLAUDE.md` が整備されている
- [ ] エージェントを使って少なくとも5つのタスクを完了した
- [ ] エージェントの出力品質に対する感覚値を持っている
- [ ] Agent-First開発の有効性（またはその限界）を具体的に語れる

### Phase 2: チームでの運用（1〜3ヶ月）

**目標**: チーム全体でAgent-First開発のワークフローを確立し、共有ルール体系を整備する。

**ステップ1: 共有ルール体系の確立**

Phase 1で個人が蓄積した知見を、チーム全体で共有するルール体系に昇格させる。ここで重要なのは、第4章で解説したルールファイルの二層構造である。

```
project-root/
├── AGENTS.md                    # 全チームメンバー共通のルール
├── docs/
│   └── standards/
│       ├── coding-style.md      # コーディングスタイルの詳細
│       └── testing-policy.md    # テスト方針の詳細
└── .personal/                   # 個人設定（gitignoreに追加）
    └── agent-preferences.md     # 個人の好みや追加ルール
```

共有ルールの策定にあたっては、以下の原則を守る。

- **機械的に検証可能なルールを優先する**: 「読みやすいコードを書くこと」よりも「関数は50行以下にすること」の方がエージェントにとって明確である
- **理由を併記する**: ルールだけでなく、その背景にある理由を記述する。エージェントは理由を理解することで、ルールの適用範囲を適切に判断できる
- **例外を明示する**: ルールの例外がある場合、それも明示する。例外が暗黙的であると、エージェントはルールを杓子定規に適用して不適切な出力を生成する

**ステップ2: PRフローの見直し**

8.2節で解説したPRフローの変革を、チームレベルで実施する。

- CI/CDパイプラインにAgent-First向けの自動チェックを追加する
- リスクベースのレビュー戦略を導入する
- 低リスクな変更に対する自動マージの試行を開始する

**ステップ3: バックグラウンドタスクの導入**

8.3節で解説した継続的自動リファクタリングを、チームのワークフローに組み込む。最初は週1回のバッチ実行から始め、効果を確認しながら頻度を上げていく。

**ステップ4: チーム内の役割を再定義する**

チームメンバーの役割を、Agent-First開発に合わせて再定義する。全員が同じ速度で移行できるわけではないため、以下のような段階的な役割設計が有効である。

| 役割 | 主な責務 | 必要スキル |
|------|---------|-----------|
| ハーネスアーキテクト | ポリシーファイルの設計、ガードレールの定義、リポジトリ構造の設計 | Product thinking + Engineering |
| エージェントオペレーター | エージェントへのタスク指示、出力のレビュー、フィードバック | Product thinking |
| 品質エンジニア | 評価基準の設計、本番モニタリング、品質メトリクスの追跡 | Engineering + Data science |
| インフラエンジニア | CI/CD、サンドボックス環境、実行基盤の構築・運用 | Engineering |

[表8.3: Agent-First開発におけるチーム内役割]

これらの役割は固定的なものではなく、チームの成熟度に応じて流動的に変化させるべきである。Phase 2の初期では、ハーネスアーキテクトの役割が特に重要になる。

**成功基準**: Phase 2の終了時に、以下の状態が達成されていることを確認する。

- [ ] チーム共通のルール体系（`AGENTS.md` + `docs/standards/`）が整備されている
- [ ] CI/CDパイプラインにAgent-First向けの自動チェックが組み込まれている
- [ ] チームメンバー全員がエージェントを日常的に使用している
- [ ] チームの生産性指標（PR/日、リードタイム）が改善傾向にある
- [ ] バックグラウンドリファクタリングが定期的に実行されている

### Phase 3: 組織全体への展開（3〜12ヶ月）

**目標**: Agent-First開発を組織全体の標準プラクティスとして定着させ、品質フレームワークとガバナンス体系を確立する。

**ステップ1: 品質フレームワークの確立**

組織全体で一貫した品質基準を適用するために、品質フレームワークを確立する。このフレームワークは、第7章で解説したCLASSic次元（Cost, Latency, Accuracy, Security, Stability）を基盤とする。

```
組織レベルの品質フレームワーク:

1. 品質メトリクスの標準化
   - エージェント出力の品質スコア
   - PR自動マージ率
   - 本番インシデント発生率
   - コードベース品質グレード分布

2. 評価プロセスの標準化
   - 新規エージェント導入時の評価手順
   - 定期的なベンチマーク実行
   - 本番モニタリングの基準

3. ガバナンス
   - エージェントの利用範囲の定義
   - セキュリティポリシー
   - コンプライアンス対応
```

**ステップ2: 組織横断的なルール体系の整備**

複数のチームがAgent-First開発を実践する場合、チーム間でルールの一貫性を保つための仕組みが必要になる。以下のような二層構造が有効である。

```
org-level/
├── AGENTS.md                    # 組織共通のルール
├── standards/
│   ├── security-policy.md       # セキュリティポリシー
│   ├── data-handling.md         # データ取り扱い規約
│   └── compliance.md            # コンプライアンス要件
└── templates/
    ├── project-agents-md.md     # プロジェクト用AGENTS.mdテンプレート
    └── pr-checklist.md          # PRチェックリストテンプレート

team-level/
├── AGENTS.md                    # チーム固有のルール（org-levelを継承）
└── docs/
    └── standards/               # チーム固有の規約
```

組織レベルのルールはベースラインとして機能し、各チームはそれを継承・拡張する。この構造は、本書の冒頭で紹介したAI-DLCのルール体系（ベースレイヤー + 拡張レイヤー）と同じ考え方に基づいている。

**ステップ3: トレーニングプログラムの実施**

Agent-First開発への組織全体の移行には、計画的なトレーニングプログラムが不可欠である。

| 対象 | トレーニング内容 | 期間 |
|------|----------------|------|
| 全エンジニア | Agent-First開発の基本概念、マップファイルの書き方、エージェントの基本操作 | 1-2日 |
| チームリーダー | PRフローの設計、品質メトリクスの設計、チーム体制の再編 | 2-3日 |
| アーキテクト | ハーネス設計、ガードレール設計、リポジトリ設計 | 3-5日 |
| セキュリティ担当 | エージェントのセキュリティリスク、サンドボックス設計、コンプライアンス対応 | 2-3日 |

**ステップ4: 継続的改善の仕組み**

Phase 3では、Agent-First開発の成熟度を継続的に測定し、改善していく仕組みを構築する。

LangChainが提唱するbuild → test → ship → observe → refineのサイクル（LC-13）を、組織レベルで回すことが目標である。具体的には以下の活動を定期的に実施する。

- **月次レビュー**: エージェント利用の定量メトリクス（利用率、品質スコア、生産性指標）のレビュー

---

> **Column: 「コードを書かない」ことの副次的効果**
>
> OpenAIの実験で最も興味深い発見の1つは、「コードを書かない」という制約がもたらした副次的効果である（OAI-01）。
>
> エンジニアたちは、コードを書く代わりに、自分の意図をより精確に言語化する力を鍛えられた。「ここをこう直してほしい」という曖昧な指示では、エージェントは期待通りに動かない。何を実現したいのか、どのような制約があるのか、何を避けるべきなのか——これらを明文化する訓練が、結果としてチーム全体のコミュニケーション能力を向上させた。
>
> さらに、ドメイン知識やアーキテクチャの意思決定が必然的にドキュメント化された。口頭での「暗黙の了解」に頼ることができないため、すべての重要な判断がリポジトリに記録された。これは、新しいメンバーのオンボーディングを劇的に効率化する副次的効果をもたらした。
>
> Agent-First開発は、技術的な生産性の向上だけでなく、組織のナレッジマネジメントの質を高めるパラダイムでもあるのである。

## 本章のまとめ

- **「Humans Steer, Agents Execute」**: Agent-First開発の核心は、人間の役割を「コードを書く」ことから「意図の明確化、環境の設計、品質の判断」へ移行させることにある。OpenAIの実験では、7人チームが1人あたり3.5 PR/日の生産性を達成した（OAI-01）

- **PRフローの変革**: Agent-First開発では、PRは「短命」——数分で生成、レビュー、マージされる。リスクベースのレビュー戦略と段階的な自動マージの導入が、この速度を実現する鍵である

- **継続的自動リファクタリング**: バックグラウンドでエージェントを稼働させ、ポリシーからの逸脱を継続的にスキャンし、1分以内にレビュー可能な小粒度のリファクタリングPRを自動生成する

- **リポジトリを記録のシステムに**: エージェントはリポジトリに書かれた情報に基づいて動く。ドメイン知識、アーキテクチャの意思決定、コーディング規約をすべてリポジトリ内に体系化することが、Agent-First開発の基盤である

- **3軸スキルセット**: ハーネスエンジニアには、Product thinking、Engineering、Data scienceの3つのスキルが統合的に求められる（LC-13）。「本番環境こそが主要な学習環境」という原則のもと、完璧さよりも出荷を優先し、反復的に改善する

- **スペック駆動開発（SDD）**: 「人間はコード書くな」ポリシーを組織的に実践する手法。PRDからタスク分解、エージェント実装、レビュー・マージまでの一貫したフローが「Humans Steer, Agents Execute」の現場実装である（TLDV-04）

- **段階的導入**: Phase 1（個人での試行）→ Phase 2（チームでの運用）→ Phase 3（組織全体への展開）の3段階で進める。各フェーズの成功基準を明確に設定し、段階的に成熟度を高めていく

# 第1章: なぜ今、ハーネスエンジニアリングなのか

> **読者レベル**: ★☆☆（入門 — 前提知識不要）
> **想定ページ数**: 約20ページ

2025年はAIエージェント元年と呼ばれた。OpenAI、Anthropic、Google、Salesforceをはじめとする主要テック企業が競うようにAIエージェントプラットフォームを発表し、「AIエージェントが人間の仕事を変える」という言説が世界中のテックカンファレンスを席巻した。

しかし2026年、業界の最先端で交わされる会話のトーンは明確に変わった。話題の中心はもはや「AIエージェントそのもの」ではない。「AIエージェントを信頼性高く動作させる基盤をどう設計するか」――すなわち**ハーネスエンジニアリング**（Harness Engineering）である。

本章では、なぜこのパラダイムシフトが起きているのか、ハーネスエンジニアリングとは何か、そしてAIエージェント産業が直面する「信頼性ギャップ」をハーネスがどう解決するのかを概観する。そのうえで、本書全体のロードマップを示し、読者が自身の関心とスキルレベルに合った読書パスを選択できるようにする。

## 1.1 「モデルはコモディティ、ハーネスこそが競争優位」

### 2025年 — AIエージェント元年の光と影

2025年を振り返ると、AIエージェントの領域で驚くべき進展があった。OpenAIはCodexエージェントプラットフォームを公開し、AnthropicはClaude Codeでコーディングエージェントの実用性を実証した。GoogleはAgent Development Kit（ADK）をオープンソースとしてリリースし、SalesforceはAgentforceで企業向けエージェント市場に本格参入した。LangChainはDeep Agentsという「バッテリー込み」のエージェントハーネスを発表し、エージェント開発のあり方そのものを再定義しようとしていた。

しかし、華やかな発表の裏側で、実際にAIエージェントを本番環境に導入した企業は深刻な課題に直面していた。エージェントが途中で方向性を見失う。長時間のタスクでコンテキストが劣化し、品質が急激に低下する。ガードレールのない状態で予期しない行動をとる。これらの問題は、モデルの性能だけでは解決できないものであった。

### 2026年 — ハーネス時代の到来

2026年2月、OpenAIのRyan Lopopolo氏が「Harness engineering: leveraging Codex in an agent-first world」と題した記事を発表し、「ハーネスエンジニアリング」という概念を広く普及させた（OAI-01）。この記事が報告した実験は衝撃的であった。7人のエンジニアからなるチームが、**5ヶ月間にわたって一切の手書きコードなしで**約100万行・1,500件のプルリクエストからなるソフトウェア製品を構築したのである。

エンジニアの役割は「コードを書く」ことから「環境を設計し、意図を明確にし、フィードバックループを構築する」ことへと根本的に変化した。Lopopolo氏はこの新しい働き方を**"Humans Steer, Agents Execute"**（人間が舵を取り、エージェントが実行する）と表現した。1人のエンジニアが1日あたり平均3.5件のプルリクエストをマージするという生産性は、従来の開発プロセスでは考えられない水準である（OAI-01）。

この実験が示したのは、単にモデルの性能が向上したという事実ではない。エージェントが自律的に働ける「環境」——すなわちハーネス——を適切に設計することで、少人数のチームが大規模な成果を生み出せるということである。

### モデルのコモディティ化とハーネスの競争優位

Aakash Gupta氏は、この変化をより直截に表現した。「2025年はエージェントの年だった。2026年はエージェントハーネスの年である」と宣言し、**「モデルはコモディティ、ハーネスこそが競争優位（moat）」**であると提言した（BLG-02）。

Gupta氏の主張の核心は明快である。主要なLLMの性能差は急速に縮小しており、モデルの選択だけでは持続的な競争優位を築けない。真の差別化要因は、モデルの周囲に構築する「仕組み」——ツール連携、状態管理、コンテキスト制御、ガードレール——にある。この認識を裏付ける象徴的な事例として、AIアシスタント「Manus」の開発チームが6ヶ月間でハーネスを**5回書き直した**エピソードが紹介されている（BLG-02）。モデルは据え置きのまま、ハーネスの改善だけで製品の品質が劇的に向上したのである。

### Terminal Bench 2.0 — ハーネスの効果を数値で実証

「モデルではなくハーネスが重要」という主張は、定量的なデータによっても裏付けられている。

2026年2月、LangChainチームはコーディングエージェントのベンチマーク「Terminal Bench 2.0」における実験結果を報告した（LC-08）。Terminal Bench 2.0は、ソフトウェアエンジニアリング、バイオロジー、セキュリティ、ゲーミングの4領域にまたがる89のタスクで構成される評価基盤である。各タスクは数分から10分程度の作業時間と100回以上のツールコールを要求する、実践的な難易度を持つ（LC-16）。

LangChainチームは、使用するモデル（GPT-5.2-Codex）を**一切変更せず**、ハーネスの最適化のみを実施した。その結果、スコアは52.8から66.5へと**13.7ポイント向上**し、ランキングはTop 30からTop 5へと劇的に改善された（LC-08）。

この実験が明確に示しているのは、同じモデルであっても、ハーネスの設計次第でパフォーマンスに決定的な差が生まれるということである。LangChainチームはハーネスエンジニアリングを次のように定義している。

> ハーネスエンジニアリングとは、モデルの周囲にシステムを構築し、タスクパフォーマンス、トークン効率、レイテンシ等の目標を最適化することである。（LC-08）

具体的には、以下の3つの最適化領域が大きな効果をもたらした。

1. **Build & Self-Verification Loop**（構築と自己検証ループ）: エージェントにタスク完了前の検証パスを強制し、ハッピーパスとエッジケースの両方をテストさせる
2. **Environmental Context Delivery**（環境コンテキストの配信）: ディレクトリ構造のマッピング、利用可能ツールの発見、時間予算の警告をエージェントに注入する
3. **Reasoning Compute Allocation**（推論計算の割り当て）: 計画フェーズと検証フェーズに高い推論コストを配分し、実装フェーズでは効率を重視する「推論サンドイッチ」パターン

とりわけ興味深いのは、推論計算の割り当てに関するデータである。すべてのフェーズで最大推論を使用したアプローチでは53.9%のスコアにとどまったが、フェーズごとにバランスを取った配分では63.6%を達成した（LC-08）。「より多くの推論コストをかければ良い」という素朴な直感が、必ずしも正しくないことを示す重要な知見である。

Google DeepMindのStaff EngineerであるPhilipp Schmid氏も、「2026年におけるエージェントハーネスの重要性」と題した記事で同様の見解を示している。Schmid氏はハーネスを「AIモデルをラップして長期実行タスクを管理するインフラストラクチャ」と定義し、標準的なベンチマークでは検出できない——50ステップ後にモデルが軌道を逸れる——問題こそがハーネスで解決すべき課題であると指摘した（BLG-01）。

## 1.2 ハーネスエンジニアリングの定義と位置づけ

### エージェントハーネスとは何か

エージェントハーネス（Agent Harness）とは、AIモデルを中核に据えながら、そのモデルが長期実行タスクを信頼性高く遂行するために必要な周辺インフラストラクチャの総称である。プロンプトプリセット、ツールコールの管理、ライフサイクルフック、計画機能、サブエージェント管理など、モデルの「外側」で提供される仕組みの集合体である（BLG-01）。

「ハーネス」という言葉は、もともと馬具を意味する。馬の力を制御し、目的の方向へ安全に導くための装具——それがハーネスである。世界経済フォーラム（WEF）は2026年1月の記事で、この語源的なアナロジーをAIに適用し、「デジタルハーネス」という概念を提唱した。AIの加速する能力を人間の目的と社会的価値に整合させるフレームワーク——それがデジタルハーネスである（BLG-08）。馬具が馬の力を殺すのではなく、適切な方向へ導くのと同じように、エージェントハーネスはAIの能力を制限するのではなく、信頼性高く発揮させるための基盤なのである。

### Framework / Runtime / Harness — 3層スタック

エージェントハーネスの位置づけを理解するためには、エージェント開発スタックの全体像を把握する必要がある。LangChainのCEOであるHarrison Chase氏は、2025年11月に「Agent Frameworks, Runtimes, and Harnesses - oh my!」と題したブログ記事で、エージェントスタックを3つの層に分類した（LC-01）。

```
┌─────────────────────────────────────────┐
│         Agent Harness（ハーネス）          │ ← 最上位
│  計画、サブエージェント、ファイルシステム、    │
│  自動要約・コンパクション                    │
├─────────────────────────────────────────┤
│         Agent Runtime（ランタイム）         │ ← 中間層
│  状態管理、耐久実行、チェックポイント         │
├─────────────────────────────────────────┤
│        Agent Framework（フレームワーク）     │ ← 基盤層
│  抽象化、ツール定義、モデル統合              │
└─────────────────────────────────────────┘
```

[図1.1: エージェント開発スタックの3層構造]

- **Agent Framework（フレームワーク）**: LLMとの対話、ツールの定義、外部サービスとの統合を抽象化する層。LangChainが代表例である。主要な価値は抽象化にある
- **Agent Runtime（ランタイム）**: 状態管理、耐久実行、チェックポイントなど、エージェントの「実行基盤」を提供する層。LangGraphが代表例である。低レベルの制御を可能にする
- **Agent Harness（ハーネス）**: フレームワークとランタイムの上に構築される最上位の層。計画ツール、サブエージェント、ファイルシステムアクセス、自動要約・コンパクションなど、「バッテリー込み」の機能を提供する。DeepAgentsが代表例である

この3層分類は、LangChainの具体的な製品体系（LangChain/LangGraph/DeepAgents）に基づいているが、概念的にはプラットフォームを超えて適用可能である。OpenAIのCodexエコシステムやAnthropicのClaude Codeも、同様の層構造を内包している（詳細は第5章で比較する）。

### Harrison Chaseの直接定義

Chase氏は、LangChain VIPコミュニティコール（2025年12月9日開催）において、この3層構造をさらに具体的に説明している。以下はその直接の発言である。

> LangGraph, we think of as like a really low-level kind of like agent runtime. LangChain, we think of as like an agent framework, so the main value there is our abstractions. And then deep agents, we think of as like an **agent harness** on top of it.
>
> （LangGraphは低レベルのエージェントランタイムです。LangChainはエージェントフレームワークで、その主要な価値は抽象化にあります。そしてDeep Agentsは、その上に載る**エージェントハーネス**です。）（TLDV-01）

さらにChase氏は、Deep Agents（エージェントハーネス）が提供する4つの組み込み機能を具体的に列挙した。

> deep agents, we think of as like an agent harness. We think it's really good for like longer running, more autonomous agents. So it builds on top of LangChain. It's got like built-in planning tools, sub-agents, access to a file system, those automatic kind of like summarization and compaction.
>
> （Deep Agentsはエージェントハーネスです。長時間実行される、より自律的なエージェントに適しています。LangChainの上に構築されており、組み込みの計画ツール、サブエージェント、ファイルシステムへのアクセス、そして自動的な要約とコンパクションを備えています。）（TLDV-01）

この発言から、エージェントハーネスの核心的な機能が明らかになる。

1. **Planning tools（計画ツール）**: タスクを分解し、実行計画を立てる能力
2. **Sub-agents（サブエージェント）**: 複雑なタスクをサブタスクに分割し、専門化されたエージェントに委譲する能力
3. **File system access（ファイルシステムアクセス）**: コンテキストウィンドウの制約を超えて、外部ストレージを活用する能力
4. **Automatic summarization and compaction（自動要約・コンパクション）**: 長時間実行中にコンテキストが肥大化することを防ぎ、重要な情報を保持しながら圧縮する能力

これら4つの機能は、いずれも「モデルの外側」で提供される仕組みである。モデルの推論能力そのものを向上させるのではなく、モデルが推論能力を最大限に発揮できる「環境」を整えるのがハーネスの役割なのである。

### ハーネスエンジニアリングという用語の系譜

「ハーネス」という概念がエージェント開発の文脈で注目され始めた経緯を時系列で整理すると、以下のようになる。

| 時期 | 発信元 | 内容 |

| 2025年12月 | LangChain VIPコミュニティコール | Chase氏が「deep agents = agent harness」と直接定義。4つの組み込み機能を具体的に説明（TLDV-01） |

## 1.3 AIエージェント産業の現在地

### LangChain調査が示す本番運用の実態

ハーネスエンジニアリングの重要性を論じる前に、AIエージェント産業の現在地を客観的なデータで確認しておこう。

LangChainは2025年11月から12月にかけて、「State of Agent Engineering」と題する大規模調査を実施した。回答者は1,340名に上り、エージェント開発に従事するエンジニア、プロダクトマネージャー、テックリードが含まれる（LC-05）。この調査が明らかにしたのは、AIエージェントが既にテクノロジーの実験段階を脱し、本番環境に浸透しつつあるという事実である。

主要な調査結果は以下の通りである。

- **57.3%の組織がエージェントを本番運用中**: 過半数の組織が、もはやAIエージェントを「実験」ではなく「本番プロダクト」として運用している
- **89%の組織がオブザーバビリティを実装**: エージェントの動作を監視・追跡するための仕組みが、ほぼ標準的な要件として認識されている
- **70%がプロンプティングベースのアプローチに依存**: ファインチューニングではなく、市販モデルのプロンプティングによるカスタマイズが主流である（ACD-06）

これらのデータは、2つの重要なことを示している。第一に、AIエージェントの採用は既に臨界点を超えている。第二に、採用が進む一方で、大多数の組織がモデルのカスタマイズではなく「モデルの外側の設計」に注力しているということである。70%がプロンプティングベースのアプローチを採用しているという事実は、モデルそのものの改変よりも、モデルに渡すコンテキストやツール——すなわちハーネス——の設計が実務の中心であることを意味する。

### エージェントの信頼性ギャップ

しかし、本番運用の拡大は、同時に深刻な課題を浮き彫りにした。

Pan et al.（2025年12月発表、2026年2月改訂）による研究「Measuring Agents in Production」は、26のドメインにわたる306名の実務者を対象とした調査と20件の詳細ケーススタディを通じて、本番環境におけるAIエージェントの実態を明らかにした（ACD-06）。

その知見は、エージェント技術の現在の限界を冷静に示すものである。

- **68%のエージェントが、最大10ステップで人間の介入を必要とする**: エージェントの自律的な動作は、現時点では極めて限定的である
- **74%が主に人間による評価に依存**: 自動化された品質保証の仕組みが十分に確立されていない
- **信頼性（一貫して正しい動作を時間とともに維持すること）が、開発における最大の課題**: モデルの精度ではなく、信頼性が最優先の懸念事項である

この「信頼性ギャップ」——エージェントが実行できることと、信頼して任せられることの間に横たわる溝——こそが、ハーネスエンジニアリングが解決すべき核心的な課題である。

68%のエージェントが10ステップで人間の介入を要するという事実は、エージェントが「自律的に動く」という期待と現実の間に大きなギャップがあることを示している。しかし、これはモデルの性能不足が原因だとは限らない。適切なハーネスなしにエージェントを動かしていることが、信頼性低下の主要因である可能性が高いのである。

### ハーネスが解決する3つの課題

AIエージェントの信頼性ギャップを構成する課題を、ハーネスが解決する3つの観点から整理する。

**1. 持続性（Durability）の課題**

AIモデルは、本質的にステートレスである。1回の推論が終われば、そのセッションの文脈は失われる。長時間にわたるタスク——たとえば大規模なリファクタリングや複数日にわたるプロジェクト——を遂行するには、セッションをまたいで状態を保持し、進捗を引き継ぐ仕組みが不可欠である。ハーネスは、チェックポインティング、進捗ファイル、git統合などを通じて、この持続性を実現する。

**2. 一貫性（Consistency）の課題**

LLMの出力は非決定論的である。同じプロンプトに対して異なる回答を返すことがある。大規模なコードベースにおいてコーディングスタイルの一貫性を保つ、組織のポリシーに準拠した出力を保証する、といった要求は、モデル単体では満たせない。ハーネスは、ポリシーファイル（`AGENTS.md`、`CLAUDE.md`）、ガードレール、自己検証ループなどを通じて、この一貫性を確保する。

**3. 信頼性（Reliability）の課題**

エージェントが「暴走」する——無限ループに陥る、方向性を見失う、コンテキストの劣化により品質が急激に低下する——といったリスクは、長時間実行になるほど高まる。ハーネスは、ループ検出、コスト予算制限、コンパクション（コンテキスト圧縮）、環境コンテキストの配信などを通じて、この信頼性を担保する。

LangChainチームのTerminal Bench 2.0実験（LC-08）は、まさにこれら3つの課題に対するハーネスの有効性を実証したものである。自己検証ループ（一貫性と信頼性の向上）、環境コンテキスト配信（持続性の支援）、推論計算の適切な配分（信頼性の向上）——これらのハーネス最適化により、モデルを変更することなく13.7ポイントもの性能向上が実現された。

さらに、オープンソースのハーネス実装であるDeepAgents CLIが、Claude Sonnet 4.5使用時にTerminal Bench 2.0で平均42.65%を達成し、「Claude Code自体と同等」の性能を示した事実も注目に値する（LC-16）。これは、適切なハーネス設計があれば、オープンソースの実装であってもプロプライエタリなシステムに匹敵する成果を生み出せることを意味する。

### エージェントエンジニアリングという新しい専門分野

ハーネスエンジニアリングは、より広い文脈では「エージェントエンジニアリング」という新しい専門分野の中核に位置する。LangChainは、エージェントエンジニアリングを次のように定義している。

> 非決定論的なLLMシステムを信頼性の高い本番体験へと洗練させる反復的プロセス（LC-13）

この定義には2つの重要なポイントがある。第一に、LLMシステムが「非決定論的」であることを前提として受け入れている点。第二に、目標が「完璧なシステムの構築」ではなく、「信頼性の高い体験への反復的な洗練」である点である。

エージェントエンジニアリングには、3つの必須スキルセットが求められる（LC-13）。

1. **Product thinking（プロダクト思考）**: ユーザーニーズの理解、プロンプト設計、評価基準の定義
2. **Engineering（エンジニアリング）**: ツール構築、UI/UX設計、堅牢なランタイムの構築
3. **Data science（データサイエンス）**: パフォーマンス測定、エラー分析、A/Bテスト

従来のソフトウェアエンジニアリングでは、これら3つのスキルが1人のエンジニアに同時に求められることは稀であった。しかしエージェントエンジニアリングでは、「build → test → ship → observe → refine」のサイクルを高速に回すために、この3つの視点を統合的に持つことが不可欠である。

特に注目すべきは、LangChainが「**本番環境こそが主要な学習環境**」であると述べている点である（LC-13）。完璧なシステムを構築してからデプロイするのではなく、早期にデプロイし、本番での挙動を観察し、実世界のパターンに基づいて改善するというアプローチが推奨されている。これは、ハーネスエンジニアリングが一度きりの設計作業ではなく、継続的な改善プロセスであることを意味する。

### 各パートの概要

本書は4部構成+付録で構成される。

**第I部: パラダイムシフト（第1章・第2章）**

本章で概観したハーネスエンジニアリングの「なぜ」を、第2章でさらに掘り下げる。第2章では、プロンプトエンジニアリング → コンテキストエンジニアリング → ハーネスエンジニアリングという進化の系譜をたどり、それぞれの手法が何を解決し、何が不足していたかを明らかにする。この進化の理解は、ハーネスの設計原則を深く把握するための基盤となる。

**第II部: ハーネスの設計（第3章〜第5章）**

ハーネスの「何があるか（WHAT）」を解説するパートである。第3章ではエージェントハーネスの6つのコンポーネント（Reasoning Engine、Planning & Orchestration、Tool Registry、Memory & Context、State & Persistence、Structured I/O）を体系的に整理し、その相互関係を示す。第4章では`AGENTS.md`/`CLAUDE.md`に代表されるポリシーファイルとガードレールの設計手法を詳述する。第5章ではOpenAI、Anthropic、LangChainなど主要プラットフォームを中立的な評価軸で比較する。第3章と第4章にはHands-onセクションが含まれ、実際に手を動かして学ぶことができる。

**第III部: 実装・運用・品質保証（第6章・第7章）**

ハーネスの「どう作るか（HOW）」を解説するパートである。第6章では長期実行エージェントという最も挑戦的なユースケースを通じて、進捗ブリッジング、コンパクション、マルチエージェント並列実行、自己検証ループなどの具体的な実装パターンを示す。第7章ではエージェントの品質保証と評価の方法論を扱い、CLASSic次元（Cost, Latency, Accuracy, Security, Stability）に基づく評価フレームワークとオブザーバビリティの実装を解説する。

**第IV部: 組織導入と実践（第8章・第9章）**

ハーネスエンジニアリングの組織的な導入と将来展望を扱うパートである。第8章ではAgent-First開発への組織移行として、PRフローの変革、チーム体制の再設計、段階的導入ロードマップを提示する。第9章ではOpenAI社内プロジェクト、Manusの5回のハーネス書き直し、日本企業における導入事例などのケーススタディを紹介し、ハーネスエンジニアリングの将来展望を論じる。

**付録**

`AGENTS.md`テンプレート集、ハーネス設計チェックリスト、プラットフォーム別クイックスタートガイド、用語集、参照文献リスト、コンテキストエンジニアリングの学術的基盤など、実務ですぐに活用できるリファレンスを収録する。

> **Column: OpenAI 7人チームの5ヶ月 — 「コードを書かない」という選択**
>
> 2026年2月にOpenAIが公開した実験報告（OAI-01）は、ハーネスエンジニアリングの可能性を端的に示すものであった。
>
> 7人のエンジニアからなるチームは、5ヶ月間にわたり「手書きでコードを書かない」というルールを自らに課した。すべてのコードはCodexエージェントが生成し、エンジニアは環境設計、意図の明確化、レビューに専念した。結果として、約100万行のコードと1,500件のプルリクエストからなるソフトウェア製品が完成した。
>
> 興味深いのは、「コードを書かない」というルールがもたらした副次的効果である。エンジニアたちは、自分の意図をより精確に言語化する力が求められるようになった。「こう実装してほしい」という曖昧な指示では、エージェントは期待通りに動かない。リポジトリの構造、テストの方針、コーディング規約、ドメイン知識——これらをマップファイルやポリシーファイルとして体系的に整備することが、生産性の鍵となった。
>
> この実験は、ハーネスエンジニアリングが単なる技術的な手法論ではなく、エンジニアの「仕事の仕方」そのものを再定義するパラダイムであることを示唆している。読者の皆さんも、本書を読み進める中で、自身の開発プロセスを「ハーネスの視点」で見直してみてほしい。

## 本章のまとめ

- **モデルはコモディティ、ハーネスが競争優位**: 主要LLMの性能差が縮小する中、モデルの周囲に構築するハーネス（ツール連携、状態管理、コンテキスト制御、ガードレール）が真の差別化要因となっている（BLG-02）
- **ハーネスエンジニアリングの定量的効果**: Terminal Bench 2.0において、モデルを変更せずハーネスの最適化のみでスコアが52.8→66.5（+13.7pt）へ向上し、ランキングがTop 30→Top 5に改善された（LC-08）
- **エージェント産業の信頼性ギャップ**: 57.3%の組織がエージェントを本番運用する一方、68%のエージェントが最大10ステップで人間の介入を必要とする。この信頼性ギャップを埋めるのがハーネスの役割である（LC-05, ACD-06）
- **3層スタックモデル**: エージェント開発はFramework / Runtime / Harnessの3層で構成され、ハーネスは最上位の「バッテリー込み」レイヤーとして、計画、サブエージェント、ファイルシステムアクセス、自動コンパクションを提供する（LC-01, TLDV-01）
- **新しい専門分野の誕生**: エージェントエンジニアリングは、Product thinking、Engineering、Data scienceの3つのスキルセットを統合した反復的プロセスであり、本番環境こそが主要な学習環境である（LC-13）

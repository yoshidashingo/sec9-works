## 3.1 全体アーキテクチャ概要

### 6つのコンポーネント

Garcia Vargas氏は、本番環境で稼働するエージェントハーネスを分析し、6つの標準コンポーネントを抽出した（BLG-03）。本書ではこの分類を基盤としつつ、Anthropic（ANT-01, ANT-02）、LangChain（LC-01, LC-02）、Salesforce（PLT-01）の知見を統合して、以下の6層モデルとして再構成する。

1. **Reasoning Engine（推論エンジン）**: コアとなるLLM推論層。モデル非依存の設計を可能にする抽象化レイヤー
2. **Planning & Orchestration（計画とオーケストレーション）**: エージェントループの制御、ミドルウェアパターン、停止条件の管理を担う「指揮者」
3. **Tool Registry（ツールレジストリ）**: 外部ツールの定義・検索・呼び出しを管理するカタログ
4. **Memory & Context（メモリとコンテキスト）**: 短期・長期メモリの管理、コンテキストウィンドウの最適化を担う「作業記憶」
5. **State & Persistence（状態と永続化）**: チェックポイント、セッション間の状態引き継ぎ、ロールバックを可能にする「記録係」
6. **Structured I/O（構造化入出力）**: 外部との通信プロトコル、ストリーミング、エラーハンドリングを担う「窓口」

これら6つのコンポーネントは独立した機能単位であると同時に、密接に連携してエージェントの動作を支えている。

### コンポーネント間の相互作用

6つのコンポーネントの関係を、アーキテクチャ図として示す。

```
┌─────────────────────────────────────────────────────────┐
│                  Structured I/O                          │
│          （外部通信・プロトコル・ストリーミング）            │
├─────────────────────────────────────────────────────────┤
│                                                         │
│  ┌───────────────────────────────────────────────────┐  │
│  │          Planning & Orchestration                  │  │
│  │     （エージェントループ・ミドルウェア・停止条件）      │  │
│  │                                                    │  │
│  │  ┌──────────────┐    ┌──────────────────────────┐ │  │
│  │  │  Reasoning   │◄──►│  Memory & Context        │ │  │
│  │  │  Engine      │    │  （作業記憶・             │ │  │
│  │  │  （推論エンジン）│    │    コンテキスト管理）     │ │  │
│  │  └──────┬───────┘    └──────────┬───────────────┘ │  │
│  │         │                       │                  │  │
│  │         ▼                       ▼                  │  │
│  │  ┌──────────────┐    ┌──────────────────────────┐ │  │
│  │  │  Tool        │    │  State & Persistence     │ │  │
│  │  │  Registry    │    │  （状態管理・             │ │  │
│  │  │  （ツール管理） │    │    チェックポイント）     │ │  │
│  │  └──────────────┘    └──────────────────────────┘ │  │
│  └───────────────────────────────────────────────────┘  │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

[図3.1: エージェントハーネスの6コンポーネントアーキテクチャ]

この図における情報の流れを整理する。

1. **外部からのリクエスト**はStructured I/Oを通じて受信され、Planning & Orchestrationに渡される
2. **Planning & Orchestration**がエージェントループを制御し、Reasoning Engineに推論を依頼する。このとき、Memory & Contextから必要な文脈情報を取得してコンテキストウィンドウを構成する
3. **Reasoning Engine**が推論を実行し、次のアクション（ツール呼び出し、応答生成、計画更新など）を決定する
4. ツール呼び出しが必要な場合、**Tool Registry**から該当ツールを検索・実行し、結果をMemory & Contextに格納する
5. 各ステップの状態は**State & Persistence**によって記録され、障害発生時のロールバックやセッション間の引き継ぎを可能にする
6. 最終的な応答は**Structured I/O**を通じて外部に返される

Salesforceはエージェントハーネスを「エージェントの実行、状態、信頼性を管理するランタイム環境とインフラストラクチャ」と定義している（PLT-01）。この定義が示す通り、ハーネスは単一の機能ではなく、複数のコンポーネントが協調して動作するシステムである。

重要なのは、これら6つのコンポーネントがすべて「モデルの外側」に位置していることである。Reasoning Engineはモデルそのものではなく、モデルを「ラップ」する抽象化レイヤーである。この設計により、モデルのスワッピング（入れ替え）が可能になり、第1章で述べた「モデルはコモディティ、ハーネスが競争優位」という命題が技術的に裏付けられる。

以降のセクションでは、各コンポーネントの詳細を順に解説していく。

## 3.2 Reasoning Engine（推論エンジン）

### コアLLM推論層の役割

Reasoning Engine（推論エンジン）は、エージェントハーネスの「頭脳」に相当するコンポーネントである。具体的には、LLMに対する推論リクエストの構築、送信、レスポンスの解析を担う。

ただし、ここで重要な区別を明確にしておく必要がある。Reasoning Engineは「LLMそのもの」ではない。LLMへのアクセスを抽象化し、ハーネスの他のコンポーネントに統一的なインターフェースを提供するレイヤーである。この区別が、モデル非依存設計の基盤となる。

Reasoning Engineの主要な責務は以下の3つである。

**1. リクエストの構築**

Planning & Orchestrationから受け取ったコンテキスト（システムプロンプト、メッセージ履歴、ツール定義、モデル設定など）を、特定のLLMプロバイダーが要求するフォーマットに変換する。たとえば、OpenAIのAPIとAnthropicのAPIでは、ツール定義やメッセージの形式が異なる。Reasoning Engineはこの差異を吸収する。

**2. 推論の実行**

変換されたリクエストをLLMに送信し、レスポンスを受信する。ストリーミング対応、リトライ処理、タイムアウト管理なども、この層の責務に含まれる。

**3. レスポンスの解析**

LLMからのレスポンスを解析し、次のアクション（テキスト応答、ツール呼び出し、推論の継続など）を構造化されたデータとして抽出する。ツール呼び出しのパラメータ解析や、構造化出力（JSON）のバリデーションもここで行われる。

### モデル非依存設計の原則

エージェントハーネスにおいてReasoning Engineが果たす最も重要な役割は、**モデル非依存設計**（Model-Agnostic Design）を可能にすることである。

Garcia Vargas氏は、本番環境のハーネスにおいてモデル非依存設計が不可欠である理由を3点挙げている（BLG-03）。

1. **リスク分散**: 単一のLLMプロバイダーへの依存は、API障害、価格変更、利用規約変更などのリスクを集中させる
2. **コスト最適化**: タスクの複雑さに応じて、高性能・高コストなモデルと軽量・低コストなモデルを使い分けることで、全体のコストを最適化できる
3. **性能最適化**: モデルごとに得意なタスク領域が異なる。コーディングに強いモデル、推論に強いモデル、速度に優れるモデルを適材適所で使い分けることで、システム全体の性能を向上させられる

Salesforceもまた、エージェントハーネスの核心的な特性として「モデルスワップ可能な設計」を挙げている（PLT-01）。Salesforceの定義によれば、ハーネスは特定のモデルに結合するのではなく、モデルを「プラグイン」として扱うべきである。

### モデル選択・スワッピング戦略

モデル非依存設計を活用した具体的な戦略として、以下の3つのパターンが実務で用いられている。

**パターン1: タスク種別によるモデル選択**

タスクの性質に応じて最適なモデルを動的に選択する。たとえば、計画立案には推論能力の高いモデルを使い、コード生成にはコーディングに特化したモデルを使い、単純な応答には軽量で応答速度の速いモデルを使う。

LangChainチームがTerminal Bench 2.0で実証した「推論サンドイッチ」パターン（LC-08）は、この戦略の一形態である。計画フェーズでは推論コストを「extra-high」に設定し、実装フェーズでは「high」に下げ、検証フェーズで再び「extra-high」に戻す。このバランス型のアプローチは63.6%のスコアを達成し、すべてのフェーズで最大推論を使用した場合の53.9%を大きく上回った（LC-08）。ここでの教訓は明確である。推論エンジンの設定を一律にするのではなく、タスクのフェーズに応じて動的に切り替えることが、コスト効率とパフォーマンスの両方を最適化する鍵となる。

**パターン2: フォールバック戦略**

プライマリモデルが利用不可能な場合に、セカンダリモデルに自動的に切り替える。API障害やレート制限に対するレジリエンスを確保するための戦略である。

```python
# モデルフォールバックの概念例
reasoning_engine_config = {
    "primary": "claude-opus-4-6",
    "fallback": [
        "gpt-5.2-codex",
        "gemini-3-pro"
    ],
    "selection_criteria": {
        "timeout_ms": 30000,
        "max_retries": 2,
        "fallback_on": ["rate_limit", "server_error", "timeout"]
    }
}
```

**パターン3: コスト予算制約によるモデル切り替え**

APIコストの累積が予算の閾値に近づいた場合に、より低コストなモデルに切り替える。長期実行エージェントでは、コスト管理がハーネス設計上の重要な課題となる（この詳細は第6章で扱う）。

これら3つのパターンに共通するのは、Reasoning Engineが「モデルの能力を抽象化するレイヤー」として機能しているという点である。Planning & Orchestrationは、使用されるモデルの具体的な仕様を意識することなく、推論リクエストを発行できる。この疎結合の設計が、ハーネスの柔軟性と進化可能性を支えている。

フレームワークの3世代の進化——Chaining（2023年）からLangGraph（2024年）、そしてDeepAgents（2025年）へ（LC-14）——を通じて、Reasoning Engineの抽象化レベルは着実に向上してきた。初期のChainingでは特定のモデルAPIに直接依存する実装が一般的であったが、現在のハーネスでは、ミドルウェアを通じてモデルの種類やパラメータを動的に切り替えることが標準的な設計となっている。

## 3.3 Planning & Orchestration（計画とオーケストレーション）

### エージェントループ: ハーネスの心臓部

Planning & Orchestration（計画とオーケストレーション）は、エージェントハーネスの「指揮者」である。6つのコンポーネントの中で最も複雑であり、ハーネスの設計品質を最も大きく左右するコンポーネントでもある。

このコンポーネントの中核にあるのが**エージェントループ**（Agent Loop）である。エージェントループとは、エージェントがタスクを遂行するために繰り返す「思考→行動→観察」のサイクルのことである。

```
┌──────────────────────────────────────────┐
│            エージェントループ               │
│                                          │
│  ┌─────────────┐                         │
│  │  コンテキスト │                         │
│  │  収集        │─────┐                   │
│  └─────────────┘     │                   │
│        ▲              ▼                   │
│  ┌─────────────┐  ┌─────────────┐        │
│  │  検証        │  │  推論・      │        │
│  │  (Verify)    │◄─│  アクション   │        │
│  └──────┬──────┘  │  決定        │        │
│         │         └─────────────┘        │
│         ▼                                │
│  ┌─────────────┐                         │
│  │  停止条件    │─── 満たされた場合 ──► 終了 │
│  │  チェック    │                         │
│  └─────────────┘                         │
│                                          │
└──────────────────────────────────────────┘
```

[図3.2: エージェントループの基本構造]

各ステップの詳細は以下の通りである。

**1. コンテキスト収集（Context Gathering）**

エージェントが次のアクションを決定するために必要な情報を集める段階である。Memory & Contextコンポーネントから会話履歴や長期メモリを取得し、Tool Registryから利用可能なツールの定義を取得し、State & Persistenceから現在のタスク状態を取得する。ミドルウェア（後述）がこの段階で追加のコンテキストを注入することもある。

**2. 推論・アクション決定（Reasoning & Action Decision）**

収集されたコンテキストをReasoning Engineに渡し、LLMによる推論を実行する。LLMは次のアクションを決定する。アクションの種類には、ツールの呼び出し、ユーザーへの応答、計画の更新、サブエージェントへの委譲などがある。

**3. 検証（Verification）**

実行されたアクションの結果を検証する段階である。ツール呼び出しが成功したか、出力が期待通りか、エラーが発生していないかを確認する。LangChainチームの`PreCompletionChecklistMiddleware`（LC-08）は、この検証段階をより厳格にするための仕組みであり、タスク完了前にハッピーパスとエッジケースの両方をテストで確認することを強制する。

**4. 停止条件チェック（Termination Check）**

エージェントループを終了すべきかを判定する段階である。停止条件については後述する。条件が満たされなければ、コンテキスト収集に戻り、ループが継続される。

このサイクルが、エージェントが自律的にタスクを遂行する基本メカニズムである。Anthropicが「Building Effective Agents」で示したように、最もシンプルで効果的なエージェントは、このループを堅実に回すことに集中している（ANT-02）。

### Middlewareパターン: コンテキストエンジニアリングの実装基盤

エージェントループの各ステップにおいて、ループの振る舞いをカスタマイズするための仕組みが**ミドルウェア**（Middleware）パターンである。この概念は、LangChainチームのSydney Runkle氏がLangChain VIPコミュニティコール（2025年10月2日開催）で詳細に解説した（TLDV-02）。

ミドルウェアの中核となるのが、**modify model request**フックである。Runkle氏はこのフックについて次のように説明している。

> modify model requestは、コンテキストエンジニアリングの領域において極めて重要である。モデルリクエストの変更により、次回のLLM呼び出しに関連するほぼすべてのものを変更できる。モデル、ツール、システムプロンプト、メッセージ履歴、モデル設定がその対象である。（TLDV-02）

この発言が示す通り、ミドルウェアはコンテキストエンジニアリングをコードとして実装するための基盤である。第2章で解説した4つのコンテキスト操作戦略（Write、Select、Compress、Isolate）は、いずれもミドルウェアを通じて実装可能である。

ミドルウェアによって変更可能な項目を整理する。

| 変更対象 | 説明 | 活用例 |
|----------|------|--------|
| **モデル** | 使用するLLMの種類・バージョン | タスクフェーズに応じたモデル切り替え（推論サンドイッチ） |
| **ツール** | 利用可能なツールの集合 | タスク段階に応じたツールの追加・削除 |
| **システムプロンプト** | エージェントへの基本指示 | 環境コンテキストの動的注入 |
| **メッセージ履歴** | 会話の履歴 | コンパクション（要約・圧縮）の適用 |
| **モデル設定** | temperature、max_tokensなど | 推論コストの動的調整 |

### プリビルトミドルウェア

LangChainチームは、汎用的なユースケースに対応するプリビルトミドルウェア（Pre-built Middlewares）を4種類提供している（TLDV-02）。これらはハーネス設計の「ビルディングブロック」として機能する。

**1. Human-in-the-Loop Middleware**

特定の条件（高リスクなツール呼び出し、コスト閾値の超過など）が満たされた場合に、エージェントの実行を一時停止し、人間の承認を求める。Pan et al.の調査（ACD-06）が示すように、68%のエージェントが最大10ステップで人間の介入を必要とする現状では、この承認メカニズムをどこに挿入するかが設計上の重要な判断となる。

**2. Summarization Middleware**

会話履歴が一定の長さを超えた場合に、自動的に要約を生成してメッセージ履歴を圧縮する。コンテキストウィンドウの有限性という物理的制約に対処するための基本的な戦略であり、Memory & Contextコンポーネントと密接に連携する。

**3. Planning / Todo Middleware**

エージェントのタスク実行に計画性を付与する。タスク開始時に計画を立て、各ステップでの進捗を追跡し、必要に応じて計画を修正する。Harrison Chase氏がDeep Agentsの4つの組み込み機能として最初に挙げた「planning tools」（TLDV-01）は、このミドルウェアの進化形である。

**4. File System Middleware**

エージェントにファイルシステムへの読み書きアクセスを提供する。後述するMemory & Contextコンポーネントにおけるファイルシステムベースのコンテキストエンジニアリング（LC-12）を実現するための基盤である。

### create_react_agent から create_agent + middleware への進化

ミドルウェアパターンの導入は、エージェントフレームワークのAPIデザインにも根本的な変化をもたらした。

従来のLangChainでは、`create_react_agent`という関数がエージェントの構築に使用されていた。ReActパターン（Reasoning + Acting）に基づくこのアプローチでは、推論とアクションのループが関数内にハードコードされており、カスタマイズの余地が限定的であった。

ミドルウェアの導入により、APIは`create_agent` + middlewareの構成に進化した（TLDV-02）。`create_agent`は最小限のエージェントループを提供し、ミドルウェアがそのループの各ステップをカスタマイズする。この設計により、開発者は必要な機能だけを選択的に追加でき、エージェントの振る舞いを細粒度で制御できるようになった。

```python
# 従来のアプローチ（ReActパターンがハードコード）
agent = create_react_agent(
    model="claude-opus-4-6",
    tools=[search, calculator, file_reader]
)

# 新しいアプローチ（ミドルウェアによる構成）
agent = create_agent(
    model="claude-opus-4-6",
    tools=[search, calculator, file_reader],
    middlewares=[
        SummarizationMiddleware(max_context_tokens=100000),
        PlanningMiddleware(update_frequency="every_5_steps"),
        HumanInTheLoopMiddleware(require_approval=["delete_file"]),
        LocalContextMiddleware(project_root="./"),
    ]
)
```

この進化は、「フレームワーク → ランタイム → ハーネス」という3層スタック（LC-01）におけるハーネス層の設計思想を体現している。ハーネスは固定的な振る舞いを提供するのではなく、ミドルウェアの組み合わせによって柔軟に構成可能な「バッテリー込み」のフレームワークなのである。

### 停止条件: エージェントをいつ止めるか

エージェントループの設計において、「いつ止めるか」は「何をするか」と同じくらい重要である。適切な停止条件なしにエージェントを動かすことは、無限ループや暴走のリスクを招く。

主要な停止条件は以下の4種類に分類される。

**1. 完了条件（Completion）**

エージェントがタスクの完了を自ら宣言する。最も自然な停止条件であるが、エージェントが早すぎる完了宣言（false positive）を行うリスクがある。`PreCompletionChecklistMiddleware`（LC-08）はこのリスクに対処するために設計されたものであり、エージェントが完了を宣言する前に検証チェックリストの通過を強制する。

**2. 最大反復回数（Maximum Iterations）**

エージェントループの反復回数に上限を設定する。無限ループを防止するための最も基本的なセーフガードである。上限値の設定はタスクの複雑さに依存し、単純なタスクでは10〜20回、複雑なコーディングタスクでは100回以上が適切な場合もある。Terminal Bench 2.0の評価では、各タスクが100回以上のツールコールを要求する場合があると報告されている（LC-16）。

**3. コスト予算（Cost Budget）**

APIコストの累積額に上限を設定する。長期実行エージェントでは、コストが予想を大幅に超過するリスクがある。Nicholas Carliniのプロジェクト（16並列Claudeインスタンスで10万行のCコンパイラを構築）では、約2,000セッション、$20,000のAPIコストが費やされた（ANT-12）。このようなプロジェクトでは、セッション単位およびプロジェクト全体のコスト予算が不可欠である。

**4. エラー閾値（Error Threshold）**

連続するエラーの回数や、エラー率が閾値を超えた場合にループを停止する。`LoopDetectionMiddleware`（LC-08）は、この閾値ベースの停止条件の発展形であり、同一ファイルへのN回以上の編集を検出した場合に「破滅ループ」（Doom Loop）と判断し、エージェントにアプローチの変更を促すコンテキストを注入する。完全にループを停止するのではなく、まず「方向転換」を試みるという点が特徴的である。

これら4つの停止条件は排他的ではなく、組み合わせて使用されるのが一般的である。たとえば、「完了条件を満たすか、最大100反復に達するか、コスト予算$50を超過するか、連続エラー5回に達するか、いずれか最初に成立した条件でループを終了する」という設定が典型的である。

### デザインパターン: エージェントの構成方法

Planning & Orchestrationのもう一つの重要な責務は、複数のエージェントやタスクをどのように構成するかを定義することである。Anthropicは「Building Effective Agents」（ANT-02）において、エージェントシステムの構築に使用される主要なデザインパターンを体系的に整理している。ここでは各パターンの概要を示す（具体的な実装は第6章で扱う）。

**パターン1: プロンプトチェーン（Prompt Chaining）**

複数のLLM呼び出しを直列に連結し、各ステップの出力を次のステップの入力として使用する。タスクを明確なサブタスクに分解でき、かつ各サブタスクの実行順序が確定している場合に適する。

```
入力 → [LLM呼び出し1] → 中間結果 → [LLM呼び出し2] → 最終結果
                              ↓
                         ゲートチェック
                     （品質が基準を満たすか？）
```

[図3.3: プロンプトチェーンパターン]

**パターン2: ルーティング（Routing）**

入力の内容に応じて、異なる処理パスに振り分ける。分類タスクの後に専門化された処理を行うパイプラインに適する。たとえば、ユーザーの質問を「技術的質問」「ビジネス質問」「一般質問」に分類し、それぞれに最適化されたプロンプトやツール構成で処理する。

**パターン3: 並列化（Parallelization）**

複数のLLM呼び出しを同時に実行し、結果を集約する。同一タスクを異なるアプローチで解くセクショニング（Sectioning）と、同一タスクの独立したサブタスクを並行処理する投票（Voting）の2つのサブパターンがある。

**パターン4: オーケストレーター・ワーカー（Orchestrator-Workers）**

中央のオーケストレーターエージェントがタスクを分析・計画し、複数のワーカーエージェントにサブタスクを分配する。ワーカーの実行結果はオーケストレーターが集約・統合する。Anthropicの研究チームが構築したマルチエージェントリサーチシステムも、このパターンを採用している（ANT-06）。

```
                    ┌─────────────────┐
                    │ オーケストレーター │
                    │ （計画・統合）     │
                    └────┬──────┬─────┘
                         │      │
              ┌──────────┘      └──────────┐
              ▼                             ▼
     ┌─────────────┐               ┌─────────────┐
     │ ワーカー A    │               │ ワーカー B    │
     │ （サブタスク1）│               │ （サブタスク2）│
     └─────────────┘               └─────────────┘
```

[図3.4: オーケストレーター・ワーカーパターン]

LangChainチームは、これらのパターンをDeep Agentsの2つのプリミティブ——**サブエージェント**（Subagents）と**スキル**（Skills）——として抽象化している（LC-09）。

サブエージェントは、「コンテキストの肥大化」に対処するための作業委譲メカニズムである。親エージェントのコンテキストウィンドウがほぼ満杯になった場合、あるいは専門的なタスクが発生した場合に、独立したコンテキストを持つサブエージェントにタスクを委譲する。サブエージェントは独自のツール集、システムプロンプト、モデル設定を持つことができ、親エージェントのコンテキストを汚染することなく作業を遂行する（LC-09）。

スキルは、`SKILL.md`ファイルによって定義される「手続き的な知識」である。サブエージェントがタスクの「委譲」であるのに対し、スキルはタスクの「手順書」にあたる。繰り返し使用される手順を標準化し、複数のエージェント間で共有するのに適している（LC-09）。サブエージェントとスキルの使い分けの詳細は第4章で解説する。

Anthropicが「Building Effective Agents」で強調している原則は、これらのデザインパターンの選択に際して常に念頭に置くべきものである。

> シンプルに始め、必要な場合にのみ複雑化する。（ANT-02）

プロンプトチェーンで十分なタスクにオーケストレーター・ワーカーを持ち込むのは過剰設計であり、デバッグの困難さ、レイテンシの増大、コストの増加を招く。デザインパターンの選択は、タスクの要件に基づいて慎重に行うべきである。

### 環境コンテキスト配信: LocalContextMiddleware

LangChainチームがTerminal Bench 2.0の最適化で実証した3つの領域のうち、「Environmental Context Delivery（環境コンテキストの配信）」は、Planning & Orchestrationの設計に直接関わる（LC-08）。

**LocalContextMiddleware**は、エージェントが作業を開始する前に、作業環境に関する情報を自動的にコンテキストに注入するミドルウェアである。具体的には以下の3つの情報を配信する。

**1. ディレクトリ構造マッピング**

プロジェクトのディレクトリ構造をスキャンし、主要なファイルとディレクトリの一覧をエージェントに提供する。エージェントが「このプロジェクトの構造がわからない」状態でコーディングを開始することを防ぐ。

**2. 利用可能ツールの発見**

現在のコンテキストで使用可能なツール（テストフレームワーク、ビルドシステム、リンターなど）を検出し、その存在と使い方をエージェントに通知する。テストフレームワークの存在を知らないエージェントが、テストを書かずにコードを生成してしまうといった問題を防止する。

**3. 時間予算の警告**

タスクに設定された時間予算の残り時間をエージェントに通知する。Terminal Bench 2.0では、タスクごとに時間制限が設けられており、エージェントが時間を浪費することなく効率的に作業を進めるためには、残り時間の認識が不可欠である。

この「環境コンテキスト配信」の設計思想は、ハーネスエンジニアリングの根幹を成すものである。エージェントは、渡されたコンテキストの質に大きく影響される。優れたエンジニアが新しいプロジェクトに参加した際に、まずプロジェクトの構造を把握し、使用されているツールを確認し、締め切りを意識するのと同じように、エージェントにも適切な「オリエンテーション」を提供する必要がある。LocalContextMiddlewareは、このオリエンテーションを自動化するものである。

## 3.4 Tool Registry（ツールレジストリ）

### ツールの定義と管理

Tool Registry（ツールレジストリ）は、エージェントが利用可能な外部ツールを管理する「カタログ」である。ここでいう「ツール」とは、LLMの推論能力だけでは実現できない機能——ファイルの読み書き、Webの検索、APIの呼び出し、コードの実行、データベースへの問い合わせなど——を提供する外部関数のことである。

ツールレジストリの主要な責務は以下の3つである。

**1. ツール定義の管理**

各ツールの名前、説明文、パラメータのスキーマ、実行権限などのメタデータを管理する。LLMがツールを適切に選択・呼び出すためには、ツールの説明文が十分に明確であることが不可欠である。Anthropicが「Writing Effective Tools for AI Agents」で強調するように、ツールはLLMとの「新しいソフトウェア契約」であり、その定義の品質がエージェント全体の性能を左右する（ANT-04）。

**2. ツールの検索と選択**

エージェントが必要とするツールを、利用可能なツールの集合から効率的に見つけ出す。ツールの数が増えるにつれ、この検索の効率性が重要になる。

**3. ツールの実行と結果の返却**

選択されたツールを実行し、その結果を構造化されたフォーマットでエージェントに返す。エラーハンドリング、タイムアウト管理、権限チェックもこの層の責務である。

### Tool Search and Discovery: オンデマンドロード

エージェントが利用するツールの数が増加すると、すべてのツール定義を常にコンテキストウィンドウに含めることが非効率——あるいは不可能——になる。Anthropicは「Introducing Advanced Tool Use」（ANT-09）において、この問題に対する3つのアプローチを提案した。

**1. Tool Search and Discovery（ツール検索と発見）**

エージェントが必要なツールをオンデマンドで検索・ロードする仕組みである。すべてのツールをあらかじめコンテキストに含めるのではなく、タスクの進行に応じて必要なツールだけをロードする。これにより、コンテキストウィンドウの消費を最小化しつつ、大規模なツール集を扱うことが可能になる。

Claude Codeが約12個のツール、Manusが20個未満のツールで汎用的な機能を実現できている背景には、このオンデマンドロードの仕組みがある（LC-15）。表面上はツール数が少なく見えるが、実際にはスキル機能やMCP統合を通じて、必要に応じて膨大な数の専門ツールにアクセスできるのである。

**2. Programmatic Tool Calling（プログラマティックツール呼び出し）**

LLMが単一のツール呼び出しを逐次的に行うのではなく、コードとしてツール呼び出しのシーケンスを生成・実行する。複数のツールを組み合わせた複雑な操作を、1回のLLM推論で効率的に実行できる。Anthropicの報告では、コード実行によるMCPツール呼び出しの効率化でトークンが85%削減（約134kから約5k）されたとしている（ANT-08）。

**3. Learning from Examples（例からの学習）**

ツールの使用例を提示することで、LLMが新しいツールの使い方を迅速に学習する。ツールの説明文だけでは伝わりにくいニュアンスや使用パターンを、具体例で補完する。

Anthropicの「think」ツール（ANT-05）は、Tool Registryの設計における興味深いアプローチを示している。このツールは外部機能を提供するのではなく、LLMに「考える場所」を提供する。航空会社のカスタマーサービスでは54%の相対的改善が報告されており、ポリシーに基づく判断が必要な場面で、LLMがツール呼び出しの前に「思考」を挟むことで、判断の精度が向上する。

### MCPによるツール標準化

ツールレジストリの設計における最も重要な標準化の動きが、**MCP**（Model Context Protocol）である（ANT-15, STD-03）。

MCPは、Anthropicが2024年11月にオープンソースとして公開したプロトコルであり、AIアプリケーションと外部データソース・ツールとの接続を標準化する（ANT-15）。しばしば「AIアプリケーションのUSB-C」と表現される通り、MCPはツールの「差し口」を統一することで、任意のエージェントハーネスが任意のツールに接続できるようにする。

MCPの技術的な基盤は**JSON-RPC**（JSON Remote Procedure Call）プロトコルである（STD-03）。MCPは以下の3つのコアプリミティブを定義している。

1. **Tools（ツール）**: エージェントが呼び出し可能な外部関数
2. **Resources（リソース）**: エージェントが読み取り可能なデータソース
3. **Prompts（プロンプト）**: 再利用可能なプロンプトテンプレート

MCPの登場以前は、各エージェントフレームワークが独自のツール定義フォーマットを持っていた。LangChainのツール定義、OpenAIのfunction calling定義、Anthropicのtool use定義はそれぞれ異なる形式であり、あるフレームワーク用に作成されたツールを別のフレームワークで使用するには、定義の変換が必要であった。

MCPはこの断片化を解消する。MCPに準拠して作成されたツール（MCPサーバー）は、MCPに対応した任意のエージェントハーネス（MCPクライアント）から利用できる。2025年12月にはLinux Foundation傘下でAgentic AI Foundation（AAIF）が設立され、MCPはAGENTS.md、gooseとともに設立プロジェクトとして寄贈された（OAI-06）。OpenAI、Anthropic、Google、Microsoftなどの主要企業がプラチナメンバーとして参加しており、MCPはエージェントツールの事実上の標準プロトコルとしての地位を確立しつつある（STD-01）。

MCPの普及がハーネス設計に与える影響は大きい。Tool Registryは、独自のツール定義を管理するだけでなく、MCPサーバーの発見・接続・管理を担うゲートウェイとしての役割も果たすようになる。これにより、ハーネスの「ツールエコシステム」が飛躍的に拡大する一方で、MCP接続の管理（認証、権限制御、レート制限など）という新たな設計課題も生まれている。

MCPの詳細な仕様とプラットフォームごとの対応状況については、第5章で比較分析する。

## 3.5 Memory & Context（メモリとコンテキスト）

### 短期メモリと長期メモリ

Memory & Context（メモリとコンテキスト）は、エージェントの「作業記憶」を管理するコンポーネントである。人間のエンジニアが作業中に参照する情報——現在の作業内容、過去の決定、プロジェクトの背景知識——をエージェントにも提供するための仕組みである。

メモリは大きく2種類に分類される。

**短期メモリ（Short-term Memory）**

単一のセッション内で保持される情報である。現在の会話履歴、直前のツール呼び出しの結果、進行中のタスクの状態などが含まれる。短期メモリは主にコンテキストウィンドウ内に存在し、セッションの終了とともに失われる（明示的に永続化しない限り）。

LangGraphでは、チェックポインティング（Checkpointing）によってエージェントステップ間の状態を自動的に永続化する仕組みが提供されている（LC-11）。これにより、セッション内の任意の時点に状態を復元することが可能になる。

**長期メモリ（Long-term Memory）**

セッションをまたいで保持される情報である。過去のセッションで学んだパターン、ユーザーの好み、プロジェクト固有の知識などが含まれる。長期メモリはコンテキストウィンドウの外部に存在し、必要に応じてコンテキストに読み込まれる。

LangChainの記事によれば、ChatGPTやCursorはユーザーとのインタラクションから長期メモリを自動生成する機能を持つ（LC-11）。これらのシステムでは、ユーザーの行動パターンや好みがセッションをまたいで蓄積され、将来のセッションでのコンテキスト構築に活用される。

### コンテキストウィンドウの物理的制約: OSメタファー

エージェントのメモリ設計を理解するうえで、最も直感的で有用なアナロジーが**OSメタファー**である。LangChainはこのメタファーを次のように提示している（LC-11）。

> **LLMはCPUであり、コンテキストウィンドウはRAMである。**

コンピュータのRAMが有限であるのと同じように、LLMのコンテキストウィンドウにも物理的な上限がある。現在の最先端モデルのコンテキストウィンドウは数十万トークンに達しているが、それでも無限ではない。そして、RAMが一杯になるとコンピュータの性能が劣化するのと同じように、コンテキストウィンドウが飽和するとLLMの推論品質が劣化する。これが「コンテキストロット」（Context Rot）と呼ばれる現象である（ANT-03）。

このOSメタファーを拡張すると、ハーネスの各コンポーネントの役割がより明確になる。

| OSの概念 | LLMエージェントの対応 | 担当コンポーネント |
|----------|----------------------|-------------------|
| CPU | LLM（推論処理） | Reasoning Engine |
| RAM | コンテキストウィンドウ | Memory & Context |
| ハードディスク | 外部ストレージ（ファイル、DB） | State & Persistence |
| OS | ハーネス（リソース管理） | Planning & Orchestration |
| デバイスドライバ | ツール定義 | Tool Registry |
| I/Oポート | 通信プロトコル | Structured I/O |

[表3.2: OSメタファーとエージェントハーネスの対応]

このアナロジーから導かれる重要な設計原則がある。コンピュータのOSがRAMの管理に細心の注意を払うのと同じように、ハーネスはコンテキストウィンドウの管理を最優先の関心事として扱わなければならない。具体的には、「何をコンテキストウィンドウに入れるか」「何をコンテキストウィンドウの外に出すか」「いつコンテキストを圧縮するか」を、エージェントループの各ステップで慎重に制御する必要がある。

### 外部実行時メモリ+ポインタ方式

コンテキストウィンドウの物理的制約に対する根本的な解決策の一つが、**外部実行時メモリ+ポインタ方式**である（TLDV-06）。

この手法はIBMの研究チームによって提案された。基本的なアイデアは明快である。ツールの出力を全てコンテキストウィンドウに格納するのではなく、**外部メモリ**に保存し、コンテキストウィンドウには**ポインタ**（参照情報）だけを置く。LLMが外部メモリの情報にアクセスする必要がある場合は、ポインタを通じて取得する。

```
【従来のアプローチ】
ツール出力 → コンテキストウィンドウに直接格納
                   → コンテキストウィンドウが急速に飽和

【外部実行時メモリ+ポインタ方式】
ツール出力 → 外部メモリに保存
             → コンテキストウィンドウにはポインタのみ格納
             → 必要時にポインタ経由でアクセス
                   → コンテキストウィンドウの消費を最小化
```

[図3.5: 外部実行時メモリ+ポインタ方式の概念]

具体的には、ツールの前後にラッパー（補助的なプログラム）をかぶせ、ツール出力のインターセプト、外部メモリへの格納、ポインタの生成を自動的に行う。化学系タスクでの検証では、精度を保ちつつ実行可能であることが確認されている（TLDV-06）。

この方式は、先ほどのOSメタファーでいえば「仮想メモリ」に相当する。RAM（コンテキストウィンドウ）の物理的な上限を超えて、あたかも大きなメモリがあるかのようにエージェントを動作させる仕組みである。

### ファイルシステムによるコンテキストエンジニアリング

LangChainのNick Huang氏は、ファイルシステムをコンテキストエンジニアリングの「万能ツール」として位置づけている（LC-12）。この見方によれば、ファイルシステムは「エージェントがコンテキストを柔軟に保存・取得・更新できる単一インターフェース」である。

Huang氏は、エージェントのコンテキスト管理における4つの主要課題とファイルシステムによる解決策を整理している（LC-12）。

**課題1: トークンオーバーフロー（Token Overflow）**

取得データが必要な情報をはるかに超え、コスト増大と性能劣化を招く問題である。大量のAPI応答やデータベースクエリの結果を無加工でコンテキストに含めると、トークン消費が爆発的に増加する。

**解決策**: 大規模なツール出力を「スクラッチパッド」としてファイルシステムに書き出し、会話履歴を汚染しない。エージェントは必要な部分だけをファイルから選択的に読み込む。

**課題2: コンテキストウィンドウ不足（Insufficient Context Window）**

必要な情報がコンテキストウィンドウの容量を超過する問題である。大規模なコードベースの全体像を把握するには、コンテキストウィンドウでは不十分な場合が多い。

**解決策**: ファイルシステム上に計画書、サブエージェントの学習結果、作業指示を保存し、エージェントが必要に応じてアクセスする。コンテキストウィンドウには「現在のステップに必要な情報」だけを含める。

**課題3: 不正確な情報取得（Imprecise Information Retrieval）**

セマンティック検索がAPI仕様書やコードなどの技術文書で失敗する問題である。ベクトル類似度に基づく検索は、自然言語のテキストには有効であるが、構造化された技術文書では不正確な結果を返すことがある。

**解決策**: `ls`、`glob`、`grep`といったファイルシステムツールを、セマンティック検索の代替として使用する。構造化されたディレクトリツリーとファイル名規約を活用することで、エージェントは必要な情報を正確に見つけ出すことができる。

**課題4: 静的知識ギャップ（Static Knowledge Gaps）**

ユーザーが新しい文脈的手がかりを提供しても、エージェントが既存の知識を適応させられない問題である。LLMの訓練データに含まれない最新の情報や、ドメイン固有の知識にアクセスできない。

**解決策**: エージェントがファイルシステム上で自身の指示ファイルを動的に更新する。ユーザーとのインタラクションから学んだ知識を、ファイルとして永続化し、将来のセッションで参照する。

これら4つの課題と解決策は、ファイルシステムが単なるストレージではなく、コンテキストエンジニアリングのための能動的なツールであることを示している。Harrison Chase氏がDeep Agentsの4つの組み込み機能の一つとして「file system access」を挙げたのは（TLDV-01）、まさにこの理由からである。

### SQLiteベースMCPコンテキスト管理

ファイルシステムよりもさらに構造化されたアプローチとして、**SQLiteベースのローカルMCPによるコンテキスト管理**がある（TLDV-03）。

AIコーディング道場（2026年1月8日開催）において、参加者からClaude Memプラグインの調査報告が行われた。Claude Memは、SQLiteデータベースをローカルに持ち、MCPサーバーとして動作する仕組みである（TLDV-03）。その特徴は以下の通りである。

- エージェントとのやり取りの内容をSQLiteデータベースに自動的に書き込み蓄積する
- コンテキスト圧縮を常時適用し、コンテキストウィンドウの使用量を**50%以下に維持**する
- 通常は50回のツールコールでコンテキストウィンドウの制限に達するところを、圧縮により500トークン程度に抑え、ツールコール上限を劇的に延長する

この報告を受けた参加者の一人は、次のように述べている。

この発言は、コンテキストウィンドウの管理がエージェント開発の実務において深刻なペインポイントであり、SQLiteベースのアプローチがそれに対する有効な解決策として認識されていることを示している。

Memory & Contextコンポーネントの設計は、ハーネスの性能に最も直接的な影響を与える。コンテキストウィンドウという有限のリソースをどう管理するかは、エージェントの推論品質、コスト効率、そして長期実行能力のすべてを左右する。ここで紹介した3つのアプローチ——外部実行時メモリ+ポインタ方式、ファイルシステムベースのコンテキストエンジニアリング、SQLiteベースMCPコンテキスト管理——は、それぞれ異なるトレードオフを持つが、いずれも「コンテキストウィンドウの外側に情報を退避させ、必要に応じて取得する」という共通の設計思想に基づいている。具体的な実装パターンの選択基準と実装手順は、第6章で詳述する。

## 3.6 State & Persistence（状態と永続化）

### なぜ状態管理が必要か

State & Persistence（状態と永続化）は、エージェントの「記録係」として機能するコンポーネントである。LLMは本質的にステートレス——1回の推論が終われば、その内部状態は失われる——であるため、エージェントが複数のステップにわたるタスクを遂行するには、外部に状態を保持する仕組みが不可欠である。

OSメタファーで言えば、State & Persistenceは「ハードディスク」に相当する。RAM（コンテキストウィンドウ）は揮発性メモリであり、電源が切れれば情報は失われる。エージェントのセッションが終了した場合、あるいはコンテキストウィンドウがリセットされた場合に、それまでの進捗を失わないためには、不揮発性のストレージに情報を書き出しておく必要がある。

State & Persistenceが管理する情報には、以下のものが含まれる。

- **タスク状態**: 現在のタスクの進捗、完了したサブタスク、残りのサブタスク
- **計画**: エージェントが立てた実行計画とその更新履歴
- **成果物**: エージェントが生成したコード、ドキュメント、データ
- **メタデータ**: セッションID、タイムスタンプ、コスト累積、エラー履歴

### チェックポインティング

**チェックポインティング**（Checkpointing）とは、エージェントの状態を特定の時点でスナップショットとして保存する仕組みである。ゲームにおける「セーブポイント」に相当すると考えればわかりやすい。

チェックポインティングの目的は2つある。

**1. 障害復旧（Failure Recovery）**

エージェントの実行中にエラーが発生した場合、最後のチェックポイントから実行を再開できる。長時間のタスクにおいて、途中でAPIエラーが発生するたびにゼロからやり直すのは非効率的であり、コスト的にも許容できない。チェックポインティングにより、障害発生時の損失を最小限に抑えることができる。

**2. 探索的実行（Exploratory Execution）**

チェックポイントから分岐して、異なるアプローチを試行できる。あるアプローチが行き詰まった場合に、チェックポイントに戻って別の方法を試す。これは、人間のエンジニアが「git stash」や「git branch」を使って実験的な変更を試みるのと同じ発想である。

LangGraphでは、チェックポインティングがフレームワークレベルで組み込まれている。エージェントの各ステップの状態が自動的に永続化され、任意の時点への復元が可能である（LC-11）。

チェックポインティングの設計上の重要な判断は、「どの粒度でチェックポイントを作成するか」である。各ツール呼び出しごとにチェックポイントを作成すれば復旧の精度は高まるが、ストレージコストとI/Oオーバーヘッドが増大する。逆に、大きな区切り（サブタスクの完了など）でのみチェックポイントを作成すれば効率的であるが、障害時にやり直す範囲が大きくなる。適切な粒度はタスクの性質とコスト要件に依存する。

### git統合によるロールバック

ソフトウェア開発タスクにおいて特に強力なState & Persistenceの実装が、**git統合によるロールバック**である。

Anthropicが「Effective harnesses for long-running agents」（ANT-01）で紹介した2エージェントアーキテクチャ（Initializer Agent + Coding Agent）では、gitが状態管理の中核を担っている。Coding Agentは作業の進捗に応じてgitコミットを作成し、各コミットが事実上のチェックポイントとなる。

git統合がチェックポインティングの実装として優れている理由は3つある。

**1. 差分管理の効率性**

gitはファイルの差分のみを保存するため、フルスナップショットよりもストレージ効率が高い。大規模なコードベースにおいても、各チェックポイントのサイズは変更された部分に比例する。

**2. 分岐と統合の自然なサポート**

gitのブランチ機能を活用することで、探索的実行が自然に実現できる。エージェントが異なるアプローチを試す場合、新しいブランチを作成して試行し、成功すればマージし、失敗すればブランチを破棄する。

**3. 人間との協調**

gitは人間のエンジニアが日常的に使用するツールであるため、エージェントの作業履歴を人間が容易にレビューできる。「エージェントがどのステップで何を変更したか」がgitの履歴として可視化されることは、信頼性の確保と品質保証の両面で大きな利点となる。

Anthropicの2エージェントアーキテクチャでは、Initializer Agentがgitリポジトリのセットアップとブランチの作成を担い、Coding Agentが増分的にコミットを作成しながら作業を進める。Coding Agentのコンテキストウィンドウがリセットされた場合、新しいセッションは`claude-progress.txt`（進捗ファイル）とgit履歴を読み込むことで、前回の状態を復元する（ANT-01）。この「進捗ブリッジング」パターンの詳細な実装は第6章で解説する。

### 状態管理の設計原則

State & Persistenceコンポーネントの設計において、以下の3つの原則が重要である。

**原則1: 最小限の状態保持**

エージェントが保持すべき状態は、タスク遂行に必要な最小限にとどめる。過剰な状態保持は、ストレージコストの増大だけでなく、状態の一貫性維持の複雑さを招く。

**原則2: 冪等な復元**

チェックポイントからの復元は冪等（同じ操作を何度行っても結果が変わらない）でなければならない。復元処理自体が副作用を持つと、障害復旧がさらなる障害を引き起こすリスクがある。

**原則3: 人間可読な状態表現**

エージェントの状態は、可能な限り人間が読み取れる形式で保存する。JSONファイル、マークダウンの進捗ファイル、gitコミットメッセージなど、人間が直接確認・編集できる形式が望ましい。これは、エージェントの動作を人間がデバッグ・監査するために不可欠な設計判断である。

## 3.7 Structured I/O（構造化入出力）

### 外部との窓口

Structured I/O（構造化入出力）は、エージェントハーネスと外部世界との「窓口」を担うコンポーネントである。ユーザーからのリクエストの受信、エージェントからの応答の送信、外部システムとの通信を、構造化されたプロトコルに基づいて処理する。

Structured I/Oの設計品質は、エージェントのユーザー体験に直接的な影響を与える。どれほど優れた推論能力を持つエージェントであっても、レスポンスのフォーマットが不安定であったり、エラーメッセージが不明瞭であったりすれば、実用性は大きく損なわれる。

### JSON-RPCプロトコル

エージェントハーネスにおける通信プロトコルの標準として台頭しているのが**JSON-RPC**（JSON Remote Procedure Call）である。JSON-RPCは、JSONフォーマットに基づくリモートプロシージャコール（遠隔手続き呼び出し）のプロトコルである。

OpenAIのCodex App Serverは、JSON-RPCプロトコルベースの双方向APIとして設計されており、JetBrains、Xcode、デスクトップアプリなど複数のクライアントに統一されたハーネス層を提供している（OAI-02）。MCPもまた、JSON-RPCをトランスポート層として採用している（STD-03）。

JSON-RPCがエージェントハーネスの通信プロトコルとして選ばれる理由は以下の通りである。

1. **シンプルさ**: リクエストとレスポンスの構造が明確に定義されており、実装が容易である
2. **言語非依存**: JSONはほぼすべてのプログラミング言語でサポートされている
3. **双方向性**: クライアントからサーバーへのリクエストだけでなく、サーバーからクライアントへの通知もサポートする
4. **バッチ処理**: 複数のリクエストを1回の通信にまとめて送信できる

```json
// JSON-RPCリクエストの例（ツール呼び出し）
{
    "jsonrpc": "2.0",
    "method": "tools/call",
    "params": {
        "name": "read_file",
        "arguments": {
            "path": "/src/main.py"
        }
    },
    "id": 1
}

// JSON-RPCレスポンスの例
{
    "jsonrpc": "2.0",
    "result": {
        "content": [
            {
                "type": "text",
                "text": "# main.py\nimport sys\n..."
            }
        ]
    },
    "id": 1
}
```

### ストリーミング

長時間の推論処理やツール実行において、結果が得られるまでユーザーを待たせるのはユーザー体験として好ましくない。**ストリーミング**（Streaming）は、推論結果やツール実行結果を、生成されるにつれてリアルタイムに配信する仕組みである。

ストリーミングはStructured I/Oの設計において、以下の3つの理由で重要である。

**1. ユーザー体験の向上**

テキスト応答がトークン単位で逐次表示されることで、ユーザーはエージェントが「考えている」ことを視覚的に確認できる。長い応答を待つ間の不確実性が軽減される。

**2. 早期中断の可能性**

ストリーミング中に、エージェントの出力が明らかに方向性を誤っていることに気づいた場合、ユーザーは推論の完了を待たずに中断できる。これにより、無駄なトークン消費とAPIコストを節約できる。

**3. パイプライン処理の実現**

あるステップの出力を、完了を待たずに次のステップの入力として使用できる。たとえば、コード生成のストリーミング出力を、生成と並行して構文チェッカーに渡すといった処理が可能になる。

### エラーハンドリングとフォールバック

エージェントの実行中には、様々な種類のエラーが発生し得る。Structured I/Oはこれらのエラーを適切にハンドリングし、エージェントの耐障害性を確保する責務を担う。

エージェントハーネスで扱うべき主要なエラー類型は以下の通りである。

**LLMプロバイダーエラー**

APIのレート制限、サーバーエラー、タイムアウトなど。リトライ戦略（指数バックオフ）、フォールバックモデルへの切り替え、キューイングによる負荷分散で対処する。

**ツール実行エラー**

外部ツールの呼び出しが失敗した場合。エラーメッセージを構造化してLLMに返し、代替手段の選択を促す。ツールの一時的な障害に対しては、リトライを試みる。恒久的な障害に対しては、代替ツールへの切り替えや、ユーザーへの報告を行う。

**出力パースエラー**

LLMの出力が期待されたフォーマット（JSON、関数呼び出しの構造など）に準拠しない場合。出力の修復を試みるか、LLMに再生成を依頼する。

**権限エラー**

エージェントが許可されていない操作（ファイルの削除、ネットワークアクセスなど）を試みた場合。ガードレールコンポーネント（第4章で詳述）と連携して、操作のブロックまたは人間の承認要求を行う。

エラーハンドリングの設計原則として重要なのは、**エラーを隠蔽しないこと**である。エラーが発生した場合、それをLLMに正確に伝えることで、LLMが状況を理解し、適切な対処を選択できるようにする。エラーメッセージを丸め込んだり、無視したりすることは、エージェントの判断を歪め、さらなるエラーの連鎖を招くリスクがある。

## 3.8 [Hands-on] シンプルなエージェントハーネスの構築

本セクションでは、ここまで解説した6つのコンポーネントを実際に組み合わせて、最小構成のエージェントハーネスを構築する。目標は、6つのコンポーネントがどのように連携するかを体験的に理解することである。

### 構築するもの

ファイルの読み書きとWeb検索が可能な、シンプルなコーディングアシスタントを構築する。以下の要件を満たす。

- ユーザーの質問に対して、ファイルの内容を参照しながら回答できる
- 必要に応じてWebを検索し、最新の情報を取得できる
- 会話履歴を保持し、文脈を踏まえた応答を返す
- 一定のステップ数で自動的に停止する安全装置を備える

### 6つのコンポーネントの対応

この最小構成において、各コンポーネントがどう実装されるかを確認する。

```python
from langchain_core.tools import tool
from langgraph.checkpoint.memory import MemorySaver

# =============================================
# 1. Tool Registry（ツールレジストリ）
# =============================================
@tool
def read_file(path: str) -> str:
    """指定されたパスのファイルを読み取る"""
    try:
        with open(path, "r") as f:
            return f.read()
    except FileNotFoundError:
        return f"エラー: ファイル '{path}' が見つかりません"

@tool
def write_file(path: str, content: str) -> str:
    """指定されたパスにファイルを書き込む"""
    with open(path, "w") as f:
        f.write(content)
    return f"ファイル '{path}' に書き込みました"

@tool
def search_web(query: str) -> str:
    """Webを検索して情報を取得する"""
    # 実際の実装ではAPIを呼び出す
    return f"検索結果: '{query}' に関する情報..."

tools = [read_file, write_file, search_web]

# =============================================
# 2. Reasoning Engine（推論エンジン）
# =============================================
from langchain_anthropic import ChatAnthropic

model = ChatAnthropic(
    model="claude-sonnet-4-5",
    temperature=0
).bind_tools(tools)  # ツールをモデルにバインド

# =============================================
# 3. Memory & Context（メモリとコンテキスト）
# =============================================
# システムプロンプトでエージェントの振る舞いを定義
system_prompt = """あなたはコーディングアシスタントです。
ユーザーの質問に対して、以下のツールを活用して回答してください:
- read_file: ファイルの内容を読み取る
- write_file: ファイルに書き込む
- search_web: Webで情報を検索する

作業を始める前に計画を立て、各ステップで進捗を確認してください。"""

# =============================================
# 4. State & Persistence（状態と永続化）
# =============================================
checkpointer = MemorySaver()  # メモリベースのチェックポイント

# =============================================
# 5. Planning & Orchestration（計画とオーケストレーション）
# =============================================
from langgraph.prebuilt import create_react_agent

agent = create_react_agent(
    model=model,
    tools=tools,
    prompt=system_prompt,
    checkpointer=checkpointer,
    # 停止条件: 最大25ステップ
    max_iterations=25
)

# =============================================
# 6. Structured I/O（構造化入出力）
# =============================================
def run_agent(user_input: str, session_id: str = "default"):
    """エージェントを実行し、結果をストリーミングで返す"""
    config = {"configurable": {"thread_id": session_id}}

    try:
        result = agent.invoke(
            {"messages": [{"role": "user", "content": user_input}]},
            config=config
        )
        return result["messages"][-1].content
    except Exception as e:
        return f"エラーが発生しました: {str(e)}"

# 使用例
response = run_agent("src/main.pyの内容を読んで、改善点を提案してください")
print(response)
```

### 何が足りないか: 本格的なハーネスへの道

この最小構成は、6つのコンポーネントの連携を示すためのものである。本番環境で稼働させるためには、以下の拡張が必要である。

**ミドルウェアの追加**: Summarization Middleware（コンテキスト圧縮）、Human-in-the-Loop Middleware（危険な操作の承認）、LocalContextMiddleware（環境情報の自動注入）

**コスト管理**: APIコストの追跡と予算制限。コスト閾値に基づくモデル切り替え

**エラーハンドリングの強化**: LLMプロバイダーのレート制限対応、ツール実行エラーのリトライ戦略、フォールバックモデルの設定

**永続化の本格化**: メモリベースのチェックポインターから、ファイルシステムやデータベースベースの永続化への移行

**ガードレール**: ファイル操作の権限制御、実行時間の制限、出力のバリデーション

**オブザーバビリティ**: ログの構造化、トレーシングの導入、パフォーマンスメトリクスの収集

これらの拡張は、第4章（ポリシーとガードレール）、第6章（長期実行エージェントの実装パターン）、第7章（品質保証と評価）で順次解説していく。本セクションの目的は、6つのコンポーネントが「最小限のコードで実際に連携する」ことを体験することであり、その目的は上記のコードで達成されている。

---

> **Column: 「推論サンドイッチ」が教えてくれること --- 計算資源の配分という設計判断**
>
> LangChainチームがTerminal Bench 2.0で発見した「推論サンドイッチ」パターン（LC-08）は、ハーネスエンジニアリングの本質を端的に示すエピソードである。
>
> 素朴な直感では、「すべてのフェーズで推論コストを最大にすれば、最高の結果が得られるはずだ」と考える。しかし実際にはそうならなかった。すべてのフェーズでextra-highの推論を使用した場合のスコアは53.9%にとどまり、計画と検証にはextra-high、実装にはhighを使用するバランス型のアプローチは63.6%を達成した。
>
> なぜバランス型が優れるのか。一つの要因はタイムアウトである。すべてのフェーズで最大推論を使用すると、処理時間が長引き、タスクの時間制限に抵触するリスクが高まる。バランス型は、実装フェーズでの推論コストを抑えることで処理速度を確保し、タイムアウトによる失敗を防止していた。
>
> このエピソードから得られる教訓は2つある。第一に、「より多くのリソースを投入すれば良い」という素朴なスケーリングの発想は、ハーネス設計においては必ずしも正しくない。第二に、エージェントの各フェーズにどの程度の計算資源を配分するかという「設計判断」が、モデルの能力と同等以上にパフォーマンスを左右する。
>
> これはまさに、ハーネスエンジニアリングが「モデルの外側の設計」であることの証左である。同じモデルを使っていても、ハーネスの設計——この場合は推論コストの配分戦略——次第で、結果は10ポイント近く変動するのである。

## 本章のまとめ

- **6つのコンポーネント**: エージェントハーネスは、Reasoning Engine（推論エンジン）、Planning & Orchestration（計画とオーケストレーション）、Tool Registry（ツールレジストリ）、Memory & Context（メモリとコンテキスト）、State & Persistence（状態と永続化）、Structured I/O（構造化入出力）の6つのコンポーネントで構成される（BLG-03）
- **Middlewareパターン**: Planning & Orchestrationの中核にあるミドルウェアは、modify model requestフックを通じて、モデル・ツール・システムプロンプト・メッセージ履歴・モデル設定を動的に変更できるコンテキストエンジニアリングの実装基盤である（TLDV-02）
- **エージェントループと停止条件**: エージェントの自律的動作は「コンテキスト収集 → 推論・アクション決定 → 検証 → 停止条件チェック」のループで実現される。停止条件（完了、最大反復、コスト予算、エラー閾値）の適切な設定が暴走防止の鍵となる
- **MCPによるツール標準化**: MCPはJSON-RPCベースのプロトコルとして、エージェントハーネスとツールの接続を標準化しつつある。AAIFへの寄贈により、事実上の業界標準としての地位を確立しつつある（ANT-15, STD-03, OAI-06）
- **コンテキストウィンドウ管理の3戦略**: 外部実行時メモリ+ポインタ方式（TLDV-06）、ファイルシステムベースのコンテキストエンジニアリング（LC-12）、SQLiteベースMCPによる常時50%以下圧縮（TLDV-03）の3つのアプローチが、コンテキストウィンドウの物理的制約に対処する
- **OSメタファーの設計指針**: LLM=CPU、コンテキストウィンドウ=RAMというOSメタファー（LC-11）は、ハーネス設計の全体像を理解するための強力な枠組みであり、各コンポーネントの責務を明確化する
- **デザインパターン**: プロンプトチェーン、ルーティング、並列化、オーケストレーター・ワーカーの4つのパターンがエージェントの構成に使用される。「シンプルに始め、必要な場合にのみ複雑化する」原則が重要である（ANT-02）

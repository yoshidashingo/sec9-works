## 1. 企画趣旨

### 1.1 市場背景

2025年はAIエージェント元年と呼ばれ、OpenAI、Anthropic、Google、Salesforceなど主要テック企業がAIエージェントプラットフォームを次々と発表した。2026年に入り、業界の焦点は「AIエージェントそのもの」から「AIエージェントを信頼性高く動作させるインフラ」へと移行しつつある。

この転換を象徴する概念が**ハーネスエンジニアリング**（Harness Engineering）である。「ハーネス」という概念がエージェント開発の文脈で注目され始めた経緯は以下の通りである:

1. **2025年11月**: Anthropicのエンジニアリングチーム（Justin Young氏）が「Effective harnesses for long-running agents」を公開し、長期実行エージェントにおけるハーネス設計の重要性を示した（ANT-01）
2. **2025年11月**: LangChainのHarrison Chase氏が「Agent Frameworks, Runtimes, and Harnesses」でエージェントスタックの3層分類を提唱し、ハーネスを最上位の抽象レイヤーとして定義した（LC-01）。Chase氏はLangChain VIPコミュニティコールにおいて「deep agents, we think of as like an agent harness」と直接発言し、planning tools、sub-agents、file system access、automatic summarization/compactionの4機能を具体的に定義している（TLDV-01）
3. **2026年2月**: OpenAIのRyan Lopopolo氏が「Harness engineering: leveraging Codex in an agent-first world」を発表し、「ハーネスエンジニアリング」という用語を広く普及させた。エンジニアの役割が「コードを書くこと」から「エージェントが自律的に働ける環境を設計すること」へ変化したことを7人チーム・5ヶ月間の実験で実証した（OAI-01）

Aakash Gupta氏は「モデルはコモディティ、ハーネスこそが競争優位」と提言しており（BLG-02）、この認識は業界で広がりつつある。LangChainの調査（2025年11〜12月実施、回答者1,340名）によれば、57.3%の組織がエージェントを本番運用している（LC-05）。一方、Pan et al.（2025）の研究では、評価対象とした本番エージェントの完了率が著しく低く、68%のエージェントが最大10ステップで人間の介入を必要としたと報告されている（ACD-06）。この「エージェントの信頼性ギャップ」を埋める体系的なアプローチとして、ハーネスエンジニアリングへの注目が高まっている。

その有効性は定量的にも実証されている。LangChainチームの報告（LC-08）によれば、コーディングエージェントのベンチマーク（Terminal Bench 2.0）において、モデルを一切変更せずハーネスの最適化のみでスコアが52.8から66.5へ13.7ポイント向上し、ランキングがTop 30からTop 5に改善された。同報告はハーネスエンジニアリングを「モデルの周囲にシステムを構築し、タスクパフォーマンス、トークン効率、レイテンシ等の目標を最適化すること」と定義している（LC-08）。また、LangChainはエージェントエンジニアリングを「非決定論的なLLMシステムを信頼性の高い本番体験へと洗練させる反復的プロセス」と定義し（LC-13）、Product thinking、Engineering、Data scienceの3つの必須スキルセットを提唱している。

さらに、オープンソースのハーネス実装（DeepAgents CLI）がClaude Sonnet 4.5使用時にTerminal Bench 2.0で平均42.65%を達成し、「Claude Code自体と同等」の性能を示したことは（LC-16）、適切なハーネス設計がプロプライエタリなシステムに匹敵する成果を生み出せることを示している。

エージェントフレームワーク自体も急速に進化している。LangChainの分析（LC-14）によれば、フレームワークは3世代を経てきた: Chaining（2023年、LLMとデータ/APIの接続）→ LangGraph（2024年、耐久性と状態管理を備えた低レベルランタイム）→ DeepAgents（2025年、「バッテリー込み」のハーネス）。この進化は、エンジニアの関心が「モデルをどう呼ぶか」から「モデルの周囲をどう設計するか」へ移行していることを如実に示している。

### 1.2 なぜこの本が必要か

ハーネスエンジニアリングに関する知見は現在、英語圏のブログ記事やSubstackポスト、各社の公式ドキュメントに散在している。日本語での体系的な解説書は2026年2月時点で存在せず、以下のニーズがある:

1. **断片的知見の統合**: 101件のソース（一次ソース55件、業界記事18件、学術論文10件、書籍9件、標準化文書3件、VIPコミュニティ議事録6件）に散在する知見を、一貫した設計思想のもとで統合する書籍が求められている
2. **時間の経過に耐える原理原則の抽出**: ブログ記事やドキュメントは特定バージョンに依存するが、書籍ではプラットフォームに依存しない設計原則を体系化できる。コンテキストエンジニアリングの4操作戦略（Write/Select/Compress/Isolate）（LC-11）やMiddlewareパターン（TLDV-02）のように、実装が変わっても適用可能な原理を抽出する
3. **実践ガイド**: 散在する知見を統合し、すぐに使える設計パターン・テンプレート・チェックリスト・ハンズオンの提供。推論サンドイッチパターン（LC-08）、LoopDetectionMiddleware（LC-08）、3段階エージェント構造（LC-17）等、具体的な実装パターンを豊富に収録
4. **プラットフォーム中立の評価軸**: OpenAI、Anthropic、LangChain等を横断的に比較する評価軸を提供し、読者が最適なエコシステムを選択できる判断材料とする

## 3. 差別化ポイント

### 3.1 AI-DLCフレームワークに基づく実践ノウハウ

著者が実務で開発・運用しているAI-DLC（AI-Driven Development Life Cycle）を、ハーネスエンジニアリングの実践的フレームワークとしてケーススタディ付きで紹介。理論だけでなく、実際に動いているポリシーベースのワークフロー制御システムの内部構造を公開する。**書籍でしか提供できない体系的な設計思想**を、実務経験に基づいて記述する。

### 3.2 実務から生まれた「失敗と教訓」

各章に「失敗事例コラム」を設け、著者自身および取材先が経験した実際の失敗とそこからの教訓を紹介。成功パターンだけでなく、**アンチパターンの体系化**により、読者が同じ過ちを避けられるようにする。

### 3.3 マルチプラットフォーム横断の評価軸

特定ベンダーに偏らず、OpenAI（Codex/`AGENTS.md`）、Anthropic（Claude Code/MCP）、LangChain（DeepAgents）、Google、Salesforce、CrewAI、Microsoft等を**中立的な評価軸**で比較分析。フレームワークの3世代の進化（Chaining → LangGraph → DeepAgents）（LC-14）を踏まえ、個別プラットフォームの詳細仕様はWeb補足で常に最新版を提供しつつ、書籍では**バージョンに依存しない選択基準**を提供する。

### 3.4 ハンズオン重視の実践設計

各章に「Hands-on」セクションを設け、読者が実際に手を動かして学べる構成とする。`AGENTS.md`/`CLAUDE.md`の作成演習、簡易エージェントハーネスの構築チュートリアル、長期実行エージェントの状態管理実装など、**「読んで終わり」ではなく「手を動かして身につける」**実践性を重視。付録のテンプレート集・チェックリストに加え、GitHubリポジトリで実行可能なサンプルコードを公開する。

なお、ハーネスが重厚なアーキテクチャであるがゆえに単純なタスクにはオーバーヘッドが生じる現実（LC-17）も率直に扱い、読者がプロジェクト規模に応じた適切な粒度を選択できるよう配慮する。

### 3.5 プラットフォームに依存しない設計原則の体系化

ブログ記事や公式ドキュメントは特定のプラットフォーム・バージョンに依存するが、本書ではそれらの知見を統合し、**プラットフォームの進化に耐える設計原則**を抽出・体系化する。これは書籍という媒体でしか実現できない価値であり、2-3年後も参照価値を持つ内容を目指す。

### 3.6 一次情報へのアクセス: LangChain VIPコミュニティからの独自知見

著者はLangChain VIPコミュニティの参加者として、Harrison Chase氏やLangChainコアチーム（Sydney Runkle氏、Chester Curme氏、Eugene Yurtsev氏等）との定期的な技術議論に参加している。具体的には、Chase氏による「deep agents = agent harness」の直接定義（TLDV-01）、Runkle氏によるMiddleware概念とコンテキストエンジニアリングの関係解説（TLDV-02）など、公開ブログ記事に先行する非公開の技術議論に基づく知見を本書に収録する。ブログ記事やドキュメントだけでは得られない**開発哲学や設計判断の背景**を含むことは、他の著者には再現困難な独自の価値である。

また、著者はAIコーディング道場（TLDV-03, TLDV-04）や企業向け技術セッション（TLDV-05）の運営・参加を通じて、日本のエンジニアが実際に直面するCLAUDE.md肥大化問題（TLDV-03）やコンテキスト管理の実務課題を把握しており、日本語圏の読者に最適化された実践ガイダンスを提供できる。

## 5. 類書分析

### 5.1 直接競合

2026年2月時点で、ハーネスエンジニアリングを主題とした書籍は日本語・英語ともに確認されていない。本書の差別化の核は「先行性」ではなく、以下の3点にある:

1. **実務に裏打ちされた独自フレームワーク**: AI-DLCという、実際に運用されているポリシーベースのワークフロー制御システムを内部構造から解説できる唯一の書籍であること
2. **101件の散在するソースの統合と原理原則の抽出**: 英語圏のブログ・論文・公式ドキュメント、さらにLangChain VIPコミュニティの非公開議論に散在する知見を、プラットフォームの進化に耐える設計原則として体系化すること
3. **「読んで終わり」ではない実践設計**: 3つのHands-onセクション、付録のテンプレート集・チェックリスト、GitHubリポジトリのサンプルコードにより、読者が翌日から自プロジェクトに適用できること

### 5.2 間接競合

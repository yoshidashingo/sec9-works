## 9.1 Requirements Analysisの役割

### 9.1.1 なぜAIは質問を自動生成するのか

「ECサイトを作りたい」という一言から、優れたシステムを作れるでしょうか？

答えはNoです。「ECサイト」という言葉の中には、数百にも及ぶ設計上の決断が隠れています。

- 商品は誰が登録するのか？管理者か、出品者か、それとも外部APIから？
- 在庫はゼロになったら販売停止にするのか、バックオーダーを受け付けるのか？
- 決済はクレジットカードのみか、コンビニ払いやQRコード決済は？
- ゲスト購入は可能か、それとも会員登録必須か？

これらを人間が事前に洗い出すのは手間がかかります。AI-DLCはこの課題を逆転の発想で解決します。

> **AIが先に質問を作る。人間は答えるだけでいい。**

Claude Codeはプロジェクトの概要から、設計上の重要な決断ポイントを自動的に特定し、質問リストを生成します。あなたはその質問に答えるだけで、的確な要件定義書が生成されます。

### 9.1.2 適応型深度の概念

Requirements Analysisには「深度」があります。

| 深度 | 質問数の目安 | 適用シーン |
|------|------------|----------|
| **Minimal** | 3〜5問 | POCや個人プロジェクト、要件が自明なとき |
| **Standard** | 5〜10問 | 中規模プロジェクト（通常はこれ） |
| **Comprehensive** | 10〜15問 | 大規模プロジェクト、商業レベルの品質が必要なとき |

Claude Codeは`CLAUDE.md`の内容やプロジェクトの規模感から、深度を自動推定します。ただし、明示的に指定することもできます。

```
Requirements Analysisを「Comprehensive」深度で実行してください。
```

### 9.1.3 requirements.mdの構造

Requirements Analysisが完了すると、`requirements.md`が生成されます。主なセクションは以下の通りです。

```markdown
# 要件定義書 - BookCart

## 1. プロジェクト概要
## 2. ターゲット読者（ペルソナ）
## 3. 機能要件（FR-01〜FR-XX）
## 4. 非機能要件（NFR-01〜NFR-XX）
## 5. 制約事項
```

機能要件（FR）は `FR-01: 商品一覧表示` のように番号付きで整理され、後のCONSTRUCTIONフェーズで各ユニットへのトレーサビリティに使われます。

## 9.2 質問生成メカニズムの理解

### 9.2.1 Claude Codeはどのように質問を設計するか

Claude Codeが質問を生成するとき、以下の観点から「決断が必要なポイント」を探します。

1. **スコープの境界** — 「何を作るか」と同様に「何を作らないか」の明確化
2. **ユーザー種別** — 誰が使うのか、どんなロールがあるのか
3. **ビジネスルール** — 在庫、決済、配送など、業務固有のルール
4. **品質・制約** — パフォーマンス要件、セキュリティ、スケーラビリティ

これらを、プロジェクトの文脈に応じて取捨選択して質問ファイルに埋め込みます。

### 9.2.2 [Answer]:タグのしくみ

質問ファイルには`[Answer]:`タグが埋め込まれています。

```markdown
### Q1: ゲスト購入の可否

会員登録なしのゲスト購入を許可しますか？

A) 許可する（カゴに入れて購入まで会員登録不要）
B) 許可しない（購入には会員登録必須）
C) 許可するが、購入後に会員登録を促す

[Answer]:
```

あなたは`[Answer]:`タグの後に回答を記入するだけです。

```markdown
[Answer]:C
```

Claude Codeは回答を読み込んで、`requirements.md`を生成します。

### 9.2.3 良い回答が設計品質を決める

質問に対して曖昧な回答をすると、生成される要件定義書も曖昧になります。

| 悪い回答の例 | なぜ問題か | 改善した回答 |
|------------|---------|------------|
| 「考え中」 | AIが判断できない | 「とりあえずBで進める、後で変更可能にして」 |
| 「普通でいい」 | 「普通」の基準が人によって異なる | 「月間1万PVに耐えられる性能で」 |
| 「AとBの中間」 | 中間の実装方針が不明 | 「まずAで実装し、将来Bに切り替えられる設計にして」 |

**原則**: 回答に迷ったら「とりあえずXで進める、理由は〇〇」と決断して記入しましょう。決めないことのコストが、間違った決断のコストより高くなります。

---

> **[コラム] 要件の粒度 — どこまで詳細にすべきか**
>
> 「要件は詳細に書くほど良い」と思われがちですが、AI-DLCでは少し異なります。
>
> 詳細すぎる要件は、AIに「実装の余地」を与えません。「ボタンの色は`#3B82F6`にする」まで書いてしまうと、AIはその指示に従うだけになります。しかし「ユーザーが直感的に操作できるUI」と書けば、AIはUXのベストプラクティスを参考にして最適な実装を提案します。
>
> 要件の粒度の目安: **「何を達成したいか（What）」は詳細に、「どう実装するか（How）」はAIに任せる。**

## 9.3 ハンズオン: BookCartの要件分析

### 9.3.1 Requirements Analysisの実行

**目標**: 質問ファイルが生成される流れを観察し、各質問の意図を理解する

Chapter 7でWorkspace Detectionを完了した状態（BookCartプロジェクト）から続けます。

Claude Codeに以下のように伝えます。

```
書籍・雑貨を販売するシンプルなB2C ECサイト（BookCart）を開発したいです。
TypeScript + Next.jsで実装します。Requirements Analysisを実行してください。
```

Claude Codeは質問ファイルを生成します。

`aidlc-docs/inception/requirements/requirement-verification-questions.md`:

```markdown
# Requirements Analysis - 確認質問

### Q1: ユーザー管理

ゲスト購入（会員登録なし）を許可しますか？

A) 許可する
B) 許可しない（会員登録必須）
C) 許可するが購入後に登録を促す

[Answer]:

---

### Q2: 商品管理

商品データの登録・更新は誰が行いますか？

A) 管理者のみ（管理画面経由）
B) 外部APIから自動連携（カタログ管理ツール等）
C) 管理者 + 外部API両方

[Answer]:

---

### Q3: 在庫管理

在庫切れの場合の挙動を教えてください。

A) 購入不可にする（カートに入れられない）
B) バックオーダーを受け付ける（在庫なしでも注文可）
C) 購入はできるが在庫切れを警告表示する

[Answer]:

---
（以降、決済・配送・検索等の質問が続く）
```

各質問を読んで、その意図を確認しましょう。「この質問がなぜ必要か？」を考えながら読むことで、AIがどの視点で設計を考えているかが見えてきます。

### 9.3.2 質問への回答（サンプル回答）

**目標**: 具体的な回答を記入して、requirements.mdが生成されることを体験する

以下のサンプル回答を参考に、各`[Answer]:`タグに回答を記入してください。

```markdown
### Q1: ユーザー管理
[Answer]:C
（ゲスト購入を許可し、購入完了後に会員登録を促す。メールアドレスは必須。）

### Q2: 商品管理
[Answer]:A
（管理者のみ管理画面で登録・更新。MVP段階ではシンプルに。）

### Q3: 在庫管理
[Answer]:A
（在庫ゼロで購入不可。シンプルに管理したい。）
```

**回答のポイント**:
- 記号（A〜C等）の後に補足説明を加えると、生成される要件定義書の品質が上がる
- 「MVP段階では」「まずは」という言葉で優先度をコントロールできる

### 9.3.3 requirements.mdの生成と確認

**目標**: 回答から要件定義書が自動生成されることを確認する

すべての質問に回答したら、Claude Codeに伝えます。

```
回答しました。requirements.mdを生成してください。
```

生成される`aidlc-docs/inception/requirements/requirements.md`の例（抜粋）:

```markdown
# 要件定義書 - BookCart

## 3. 機能要件

### FR-01: ユーザー管理
- ゲスト購入を許可する（メールアドレスのみ必須）
- 購入完了後に会員登録を促すフローを設ける
- 会員登録: メールアドレス + パスワード

### FR-02: 商品管理
- 管理者のみが商品を登録・更新・削除できる
- 管理画面（/admin）を提供する
- 商品の属性: 名前、価格、在庫数、カテゴリ、画像URL

### FR-03: 在庫管理
- 在庫数が0の商品はカートに追加できない
- 注文確定時に在庫を自動デクリメントする
```

内容をレビューして、意図した要件が正しく反映されているか確認します。

## 9.4 要件の承認と次ステージへの移行

### 9.4.1 「承認」とは何か

AI-DLCでは、各ステージの成果物をユーザーが「承認」して初めて次のステージに進みます。これは単なる手続きではなく、重要な意味があります。

> **承認 = 「この方向性で進む」という合意の記録**

承認した内容は`audit.md`に記録され、将来「なぜこの設計にしたのか」を振り返ることができます。チームメンバーが後からプロジェクトに参加した場合にも、意思決定の経緯が追えます。

承認するには、Claude Codeに以下のように伝えます。

```
要件定義書を確認しました。承認します。
```

### 9.4.2 aidlc-state.mdの更新確認

承認後、`aidlc-state.md`が更新されます。

```markdown
## Stage Progress
- [x] Workspace Detection
- [x] Requirements Analysis    ← ✅ 完了
- [ ] User Stories
...
```

次のステージへの自動遷移を確認したら、Chapter 10に進みます。

## まとめ

- **Requirements Analysis** は、AIが自動生成した質問にユーザーが回答するだけで要件定義書が完成するステージ。
- **適応型深度**（Minimal/Standard/Comprehensive）でプロジェクト規模に応じた質問数をコントロールできる。
- **`[Answer]:`タグ**に回答を記入するだけで良い。曖昧な回答は避け、「とりあえずXで進める」という決断を優先する。
- 回答に補足説明を加えると、生成される`requirements.md`の品質が上がる。
- **承認** は「この方向性で進む」という合意の記録であり、`audit.md`に残ることでトレーサビリティが確保される。

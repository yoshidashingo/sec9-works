# Business Rules - Unit 3: Part III（Inceptionフェーズ・計画編）

## 1. 章フォーマットルール

### 1.1 冒頭の定型要素
- **「この章で学ぶこと」** を必ず配置（箇条書き4〜6項目）
  - 読者がこの章で得られる知識・スキルを具体的に列挙する
  - 「〜とは」「〜のしくみ」「〜のやり方」の3視点でバランスよく記述

### 1.2 末尾の定型要素
- **「まとめ」** — 章の要点を箇条書きで整理（5〜8項目）
  - 解説の要点と、ハンズオンで体験したことを混在させて記述
- **「チェックリスト」** — 読者が実施したことを確認するためのリスト
  - □ 記号を使用
  - 「〜した」という完了形で記述（確認できる具体的な行動）
  - 各章5〜8項目を目安とする

### 1.3 コラムのルール
- 各章1〜2個のコラムを配置
- コラムの見出し形式: `[コラム] タイトル`
- 本文から少し踏み込んだ実務的な視点・補足情報を提供
- 分量目安: 300〜500文字

## 3. コードサンプルルール

### 3.1 テックスタック
- すべてのコードサンプルは **TypeScript + Next.js（App Router）** で記述する
- 補助的なツール: npm, Git, GitHub CLI

### 3.2 コードブロックの書き方
- 言語タグを必ず明記する
  - TypeScriptコード: ` ```typescript `
  - ターミナルコマンド: ` ```bash `
  - ファイルの中身（YAML/JSON等）: ` ```yaml `, ` ```json `
  - Markdownファイルの例: ` ```markdown `
- コードブロックの直前に「ファイルパス」または「操作の目的」を1行で示す

### 3.3 サンプルコードの品質基準
- **実行可能であること**: コピー&ペーストして動くサンプルを提供
- **最小限であること**: 説明に必要な部分のみ記述し、無関係なコードを含めない
- **コメントを適切に入れること**: 重要な行には日本語コメントを付与

## 4. ハンズオンの構造ルール

### 4.1 ハンズオンセクションの書き方
各ハンズオンセクションは以下の構造で記述する：

```
[ハンズオンタイトル]
（目標: 〇〇することを確認する / 〇〇を体験する）

1. ステップ1の説明
   ```bash
   実行コマンド
   ```
   → 期待される出力・成果物の説明

2. ステップ2の説明
   ...

[確認ポイント]
・生成されたファイルの内容を確認する
・〇〇が正しく動作していることを確認する
```

### 4.2 ステップの粒度
- 1ステップ = 読者が1回の操作で完了できる粒度
- ターミナルコマンドは1ステップに1コマンドを原則とする（パイプ等は例外）
- 期待される出力（コマンドの結果や生成ファイル）を次のステップの前に示す

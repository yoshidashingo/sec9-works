# 詳細目次: LangChainによるAIエージェント開発の教科書

**本文書の目的**: 書籍各章の詳細構成を示し、執筆計画・内容の網羅性・分量の妥当性を確認するための文書である。

**タイトル**: 『**LangChain**によるAIエージェント開発の教科書 〜自分だけのパーソナルアシスタント/汎用業務エージェント/コーディングエージェントを育てる〜』
**想定総ページ数**: 約585〜595ページ（本文約565p＋前付け・索引等20〜30p）
**全17章構成**: 基礎から応用・カスタマイズまでを段階的にカバーし、パーソナルアシスタント/汎用業務エージェント（第III部）とコーディングエージェント（第IV部）の2つを順に構築する。第2章・第3章でLangChain/LangGraph/Deep Agentsを手厚く解説する。

**実装方針**: 全章を通じてLangChainエコシステム（LangChain/LangGraph/LangSmith）を実装の統一基盤とし、Deep Agentsパターンを本書の核心的テーマとして位置づける。Tool CallingやFunction Callingの仕組み、プロバイダー間のプロトコル差異など、エコシステムの背後にある重要概念は本文またはコラムで解説し、「なぜそう動くのか」を読者が理解できる構成とする。

### 第2章: LangChain 基礎（約45ページ）

**到達目標**: LangChain 1.0のパッケージ体系・LCEL・Message型を理解し、LangGraphのGraph API/Functional API/高度な制御機能を使って複雑なStateGraphを構築できる。

**前提**: 第1章
**活用先**: 第3章以降の全章

#### 2.1 LangChain 1.0 のパッケージ体系と設計思想
- `@langchain/core`: 基本抽象（Runnable、BaseMessage、BaseTool）
- `langchain`: 高レベルAPI（createReactAgent等）
- `@langchain/{provider}`: プロバイダー別パッケージ（`@langchain/openai`、`@langchain/anthropic`等）
- AgentExecutorの非推奨化とcreateReactAgentへの移行
- RunnableLambdaとカスタムRunnable実装の役割とLangGraphとの接続

#### 2.2 LCEL（LangChain Expression Language）
- Runnableインターフェースとパイプ演算子（|）によるチェーン構成
- RunnableParallel・RunnableBranch・RunnablePassthroughの使い分け
- ストリーミング・バッチ処理・非同期実行のサポート

#### 2.3 Message型とモデルの抽象化
- Message型の種類: HumanMessage・AIMessage・SystemMessage・ToolMessage
- BaseChatModelによるモデル抽象化とプロバイダー非依存設計
- tool_callsとAIMessageのstructured outputパターン

> **コラム: Tool Calling APIの仕組み（OpenAI / Anthropic / Google）**
> OpenAI の `tools` パラメータ・Anthropic の `tool_use` コンテンツブロック・Google の Function Declarations は、いずれも「LLMがJSON Schemaで定義されたツールを選択し呼び出し情報を返す」同一の設計思想に基づく。プロバイダー間の差異（parallel tool calling の扱い、tool_choice の制御方法、ToolMessageの返し方等）と、LangChainが提供する統一インターフェースの関係を整理する。

#### 2.4 LangGraphのGraph API（StateGraph）
- なぜLangGraphが必要か: 線形チェーンの限界とエージェンティックループの要件
- ステートマシンとしてのエージェント: StateGraph・Node・Edge・Conditional Edge
- 状態定義: `Annotation.Root()`によるスキーマ定義とリデューサー関数の設計
- ノード関数の設計原則と副作用の扱い

#### 2.5 LangGraphのFunctional API
- `entrypoint()` / `task()` 関数によるワークフロー定義
- Graph APIとFunctional APIの使い分けガイドライン
- コーディングエージェント・パーソナルアシスタント/汎用業務エージェントへの適材適所

#### 2.6 LangGraphの高度な制御機能
- Interrupt: ノード実行中のユーザー承認・入力待ち
- Command: 動的なノード遷移先の変更
- Send API: 並列ノード実行と動的なエッジ生成
- Subgraph: 複雑なエージェントの分割と合成

#### 2.7 チェックポインティングと状態永続化
- チェックポインターの種類と選定: MemorySaver・SqliteSaver・PostgresSaver
- スレッドIDによるセッション管理と状態の復元
- 長期タスクの中断・再開設計

#### 2.8 LangSmithによるトレース・評価・デバッグ
- LangSmithのセットアップとプロジェクト管理
- エージェントの実行トレースの可視化と検索
- LLM呼び出しコスト・レイテンシのモニタリング
- データセットを用いた自動評価パイプライン

#### 2.9 開発環境の構築
- Node.js/TypeScript環境とLangChain.js/LangGraph.js/LangSmithのセットアップ（概要）
- 詳細なトラブルシューティングは付録Aを参照

### 第3章: エージェンティックループとDeep Agentsパターン（約40ページ）

**到達目標**: 固定的なワークフローを超えた「エージェンティックループ（適応型ワークフロー）」の設計思想と、長期・多段階タスクを自律実行するDeep Agentsの設計パターンを習得する。Deep Agentsは本書全体を通じて実装の中心に置く概念であり、第III・IV部の具体的なエージェントもこのパターンを基盤とする。

**前提**: 第1〜2章
**活用先**: 第4章以降の全章

#### 3.1 ワークフローとエージェンティックループの違い
- 固定的なワークフロー（事前定義されたフロー）の限界
- エージェンティックループ（適応型ワークフロー）: タスクに応じて動的にフローが変化
- どちらを選ぶべきか: タスクの予測可能性と設計コストのトレードオフ

#### 3.2 基本設計パターン
- ReActパターン: LangGraphでの思考と行動の交互実行の実装
- Plan-and-Executeパターン: Plannerノードによるタスク分解と逐次実行
- Reflexionパターン: 実行結果の自己評価と再計画

#### 3.3 Deep Agentsパターン: 長期・多段階タスクへの対応
- Deep Agentsとは: 複数の推論・実行サイクルをまたいで複雑なタスクを遂行するエージェント
- LangChainエコシステムによる実装: LangGraphのSend API・Interrupt・Subgraphを組み合わせて実現
- **ディープリサーチエージェント**: 検索→収集→分析→レポート生成の多段階ループ
  - Web検索ツールと情報収集の並列実行（Send APIの活用）
  - 収集情報の統合・矛盾解消・品質評価
  - 構造化レポートの生成
- **セルフデバッグループ**: 実行失敗を自己分析して修正するエージェント
  - エラーログの解析と仮説生成
  - 修正コード生成→実行→検証の自律サイクル
- **長期タスクエージェント**: チェックポインティングを活用した中断・再開設計
  - 長時間タスクの進捗管理と状態保存
  - 障害発生時のリカバリ設計

> **コラム: なぜDeep Agentsが必要か — 単純なReActループとの違い**
> 単一ループのReActエージェントは短いタスクには十分だが、「検索→分析→検証→再検索」のように反復が多く状態が複雑なタスクでは破綻しやすい。Deep Agentsが採用するサブグラフ分割・チェックポインティング・並列実行の組み合わせにより、長期タスクの信頼性がどう向上するかを概念的に整理する。

#### 3.4 Human-in-the-Loopパターン
- LangGraphのinterrupt / resume機構の詳細
- ユーザー承認が必要な操作の判定基準
- コーディングエージェントにおける承認フロー（ファイル変更・シェル実行）
- パーソナルアシスタント/汎用業務エージェントにおける確認フロー（外部サービス操作・重要判断）

#### 3.5 ハーネスエンジニアリングの基礎
- ハーネスエンジニアリングとは: エージェントをうまく動かす環境設計の考え方
- エージェントが失敗する原因: モデル・ツール・環境の3層
- 良い環境の作り方: ツール設計・システムプロンプト・フィードバックループ
- 姉妹書『ハーネスエンジニアリング』との関係（詳細は姉妹書参照）

#### 3.6 エラーハンドリングと停止条件
- エラーの分類: LLMエラー / ツール実行エラー / 環境エラー
- リトライ戦略とフォールバック設計
- エラーからの自動回復 vs ユーザーへのエスカレーション
- 完了判定・最大反復回数・トークン予算による安全弁
- LangSmithでのコスト監視

> LangChainの`tool()`関数はZodスキーマから自動的にJSON Schemaを生成し、各プロバイダーのTool Callingフォーマットに変換してLLMに渡している。スキーマのdescriptionフィールドがそのままツール説明として入る仕組み、parallel_tool_callsのON/OFF制御、ToolMessageによる結果返却の流れを本文で詳解する。

#### 4.3 シェル実行ツール

#### 4.4 MCP（Model Context Protocol）によるツール標準化

#### 4.5 組み込みツール・サブエージェント・スキルの設計

### 第7章: エージェントの評価（Evaluations）（約35ページ）

**到達目標**: 実装したエージェントの品質をベンチマークと業務指標の両面から定量的に評価し、改善のためのインサイトを取得・活用できるようにする。

**前提**: 第3〜6章
**活用先**: 第11章（PA/汎用業務エージェントの改善）、第15章（コーディングエージェントのバグ修正）、第17章（自社特化カスタマイズ）

#### 7.1 エージェント評価の全体像
- なぜ評価が難しいか: 非決定的な出力・長いタスク・複合的な成否判定
- 評価の3層構造:
  - **ユニット評価**: 個別ツール・プロンプトの単体テスト
  - **タスク評価**: 1タスクの完了品質（正確性・コスト・レイテンシ）
  - **業務評価**: ビジネスKPIへの貢献（工数削減率・エラー率・ユーザー満足度）
- 評価→インサイト→改善→再評価のループ設計

#### 7.2 ベンチマーク評価
- 標準ベンチマークの概要:
  - **SWE-bench**: GitHub issueを自律解決するコーディングエージェント評価
  - **GAIA**: 汎用エージェント能力の多段階タスク評価
  - **AgentBench**: ツール利用・環境操作能力の評価
- 業務固有ベンチマークの設計: ゴールデンセットの作成と期待出力の定義
- LangSmithによるベンチマーク実行と結果管理
- モデル・プロンプト・ツール変更時の回帰評価パイプライン

> **コラム: SWE-benchはなぜ難しいか**
> SWE-benchのタスク（実際のGitHub issueを解決する）はマルチファイル編集・テスト実行・差分適用が連動する。公開時（2023年）にトップモデルでも解決率は数%だったが、2024〜2025年にDeep Agentsパターンを採用したシステムが50%超を達成した。この変化がDeep Agentsを本書で重視する理由の一つである。

#### 7.3 LLM-as-Judgeパターン
- LLMを評価者として使う設計: 評価プロンプトの設計と採点基準の定義
- 評価軸の例: 正確性・完全性・安全性・コスト効率・回答スタイル
- ゴールデンセット評価 vs オンライン評価（プロダクション判定）の使い分け
- バイアスの制御: 自己評価バイアス・位置バイアス・複数モデル評価による信頼性確保
- LangSmithの`evaluate()`によるLLM-as-Judgeパイプラインの実装

#### 7.4 業務評価とインサイト取得
- プロダクション環境からのデータ収集: LangSmithトレースのメトリクス集計
- 業務KPIへの接続:
  - タスク完了率・自動化率・平均処理時間
  - 人手介入率とHuman-in-the-Loopの発動頻度

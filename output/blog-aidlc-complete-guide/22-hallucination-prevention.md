# AI-DLCワークフローはハルシネーションをどう防いでいるか

[https://www.facebook.com/yoshidashingo1:title=吉田真吾] ([https://twitter.com/yoshidashingo:title=@yoshidashingo]) です。

> **これだけは覚えて帰ってね**
>
> - AI-DLCはハルシネーション防止を単一の技術ではなく、ワークフロー全体に埋め込まれた5層の構造で実現している
> - 過信防止ガイド (overconfidence-prevention.md) は実運用で発生した問題への対処として追加された「運用フィードバックの産物」である
> - 構造化された質問ファイル方式と矛盾検出が「AIに推測させない」仕組みを作る
> - 承認ゲート・監査証跡・コンテンツ検証が、ハルシネーションの「発生」ではなく「波及」を食い止める防御層として機能する
> - これらはすべてプロンプト指示であり、技術的な強制力はAIモデルの遵守能力に依存するという限界がある

## なぜ「ハルシネーション防止」を独立したテーマで語るのか

本シリーズの第7回では承認ゲートと監査証跡を、第9回ではcommonディレクトリの各ファイルを個別に解説しました。しかし、これらを「ハルシネーション防止」という横断テーマで統合的に捉えた記事はありませんでした。

ハルシネーション (Hallucination) とは、AIが事実でない情報をあたかも正しいかのように出力する現象です。チャットボットの回答であれば「間違った回答が返ってきた」で済みますが、AI-DLCのようにAIが開発ライフサイクル全体を駆動するフレームワークでは、影響範囲が根本的に異なります。

要件分析でAIが存在しない制約を「仮定」したら、その仮定に基づいて設計が進み、コードが生成され、テストが書かれます。1つのハルシネーションが後続の全ステージに伝播し、**誤った前提の上に正しく構築されたシステム**が出来上がります。バグと違って、テストでは検出できません。テスト自体が誤った前提に基づいているからです。

AI-DLCは、この問題に対して単一の技術的解決策ではなく、ワークフロー全体に分散した5つのメカニズムで対処しています。

## メカニズム1: 過信防止ガイド ― 「聞きすぎるくらい聞け」

### 運用フィードバックから生まれた対策

第9回で詳しく解説した通り、`overconfidence-prevention.md`は他のルールファイルとは性質が異なります。冒頭にこう書かれています:

```markdown
## Problem Statement
AI-DLC was exhibiting overconfidence by not asking enough clarifying questions,
even for complex project intent statements.
```

「AI-DLCが過信を示していた。複雑なプロジェクトの意図表明に対しても、十分な明確化質問を行わなかった。」

これは設計時に想定されたリスクではなく、**実運用で発生した問題**への事後対応です。根本原因は4つのステージのルールファイルにある「質問を減らす方向」の指示でした:

- Functional Design: 「Skip entire categories if not applicable」
- User Stories: 「Use categories as inspiration, NOT as mandatory checklist」
- Requirements Analysis: 質問を最小限に抑えるパターン
- NFR Requirements: 「Only if」条件による分析の抑制

### 方針の転換

修正の方向は明快です:

**旧**: 「Only ask questions if absolutely necessary (本当に必要な場合だけ質問する)」
**新**: 「When in doubt, ask the question - overconfidence leads to poor outcomes (迷ったら聞け。過信は悪い結果を生む)」

5つの新原則が定められました:

| # | 原則 | ハルシネーション防止における役割 |
|---|------|-------------------------------|
| 1 | **Default to Asking** | 曖昧さがあれば質問する → 推測による穴埋めを防止 |
| 2 | **Comprehensive Coverage** | すべてのカテゴリを評価 → 見落としによる仮定の混入を防止 |
| 3 | **Thorough Analysis** | ユーザー回答の曖昧さを精査 → 曖昧な入力からの誤解釈を防止 |
| 4 | **Mandatory Follow-up** | 不明確な回答には追加質問 → 「分かったつもり」での進行を防止 |
| 5 | **No Proceeding with Ambiguity** | 曖昧さ解消まで先に進まない → 不確実な情報の後続ステージへの伝播を防止 |

このガイドの存在自体が、プロンプト指示によるAIの行動制御には試行錯誤が伴うことを示しています。最初のルール設計では「効率」を重視して質問を減らす方向に誘導していたが、それがハルシネーションの温床になった。修正版では「正確さ」を優先し、質問を増やす方向に転換しています。

ただし、このガイドもプロンプト指示である以上、AIモデルが常にこの原則に従う技術的保証はありません。

## メカニズム2: 構造化された質問ファイルと矛盾検出

### 「チャットで質問するな」

第9回で解説した`question-format-guide.md`のルールは、ハルシネーション防止の観点から見ると別の意味を持ちます。

```markdown
**CRITICAL**: You must NEVER ask questions directly in the chat.
ALL questions must be placed in dedicated question files.
```

チャットで質問した場合、質問と回答はLLMのコンテキストウィンドウの中にしか存在しません。セッションが変わればその記録は失われ、AIは**過去に何を聞いて何を回答されたかを知らない状態**で設計を続けることになります。これはハルシネーションの直接的な原因になります。「確かユーザーはこう言っていた」という曖昧な記憶に頼る代わりに、ファイルとして永続化された質問と回答を参照できるようにする ── これが質問ファイル方式のハルシネーション防止上の意義です。

### 選択式フォーマットによる曖昧さの排除

すべての質問が選択式 (A/B/C/D + Other) で構造化されています:

```markdown
## Question [Number]
[質問文]

A) [選択肢1]
B) [選択肢2]
X) Other (please describe after [Answer]: tag below)

[Answer]:
```

自由記述ではなく選択肢を提示することで、AIがユーザーの回答を「解釈」する余地を最小化しています。「だいたいこんな感じ」という曖昧な回答が、AIの内部で「具体的にはこういうことだろう」と補完される ── この補完こそがハルシネーションの一形態です。選択式は、この補完の余地を構造的に減らします。

### 矛盾検出: 回答間の論理的整合性チェック

question-format-guide.mdの「Contradiction and Ambiguity Detection」セクションは、ハルシネーション防止の文脈では特に重要です。検出対象として以下の4種の矛盾が例示されています:

| 矛盾の種類 | 例 |
|-----------|---|
| **Scope mismatch** | 「Bug fix」なのに「Entire codebase affected」 |
| **Risk mismatch** | 「Low risk」なのに「Breaking changes」 |
| **Timeline mismatch** | 「Quick fix」なのに「Multiple subsystems」 |
| **Impact mismatch** | 「Single component」なのに「Significant architecture changes」 |

矛盾する回答をそのまま受け入れると、AIは一方の回答を無視するか、矛盾を「解釈」で解消しようとします。この「解釈」がハルシネーションです。矛盾検出は、この解釈をAIに行わせず、**ユーザーに矛盾の解消を求める**仕組みです。

矛盾が検出された場合は`{phase-name}-clarification-questions.md`で追加質問を行い、**すべての矛盾が解消されるまで先に進めません**。ただし、矛盾検出自体がAIモデルの判断に依存するため、すべての矛盾を確実に検出できる保証はありません。

## メカニズム3: 承認ゲート ― ハルシネーションの波及を食い止める

### 各ステージでの人間の検証

第7回で解説した承認ゲートは、ハルシネーション防止の文脈では「**波及防止のファイアウォール**」として機能します。

メカニズム1と2が「ハルシネーションの発生」を抑制するのに対し、承認ゲートは「**発生してしまったハルシネーションが後続ステージに伝播すること**」を防ぎます。

AI-DLCでは14ステージ中12ステージに承認ゲートが設置されており、各ステージの成果物をユーザーが検証してから次に進みます。core-workflow.mdには以下の指示が繰り返し現れます:

```markdown
**Wait for Explicit Approval**: ... DO NOT PROCEED until user confirms
```

Requirements Analysisで AIが存在しない制約を仮定したとしても、その成果物をユーザーが承認前にレビューすれば検出できます。逆に、承認ゲートなしでRequirements Analysisの成果物がそのままApplication Design → Units Generation → Code Generationに流れ込んだ場合、ハルシネーションの検出は格段に困難になります。

### 2択の標準フォーマットと創発的行動の禁止

CONSTRUCTIONフェーズの承認メッセージは2択に固定されています:

- **変更を要求する** ― レビューに基づいて修正を依頼
- **次のステージに進む** ― 承認して次へ

core-workflow.mdには「**NO EMERGENT BEHAVIOR**: Construction phases MUST use standardized 2-option completion messages」と明記されています。AIが独自に「一部だけ承認」「条件付き承認」のような第3の選択肢を生み出すことを禁止しています。

これはハルシネーション防止の観点からは、**承認プロセス自体のハルシネーション**を防ぐルールです。AIが「ユーザーは部分的に承認した」と解釈して先に進むことを構造的に防止しています。

### 承認疲れというトレードオフ

第7回で指摘した通り、12ステージの承認ゲートは「承認疲れ (Approval Fatigue)」を引き起こすリスクがあります。ユーザーが内容を精査せずに承認してしまえば、承認ゲートはハルシネーション防止のファイアウォールとして機能しなくなります。ハルシネーション防止の有効性は、最終的にユーザーの注意力に依存するという限界があります。

## メカニズム4: 監査証跡 ― 「AIが何を言ったか」の原文記録

### 要約禁止ルール

第7回で解説した`audit.md`のルールのうち、ハルシネーション防止に最も直結するのは**要約禁止ルール**です。

```markdown
- **CRITICAL**: Capture user's COMPLETE RAW INPUT exactly as provided
- **CRITICAL**: Never summarize or paraphrase user input in audit log
```

AIが要約する行為自体に、ハルシネーションのリスクが内在しています。ユーザーの発言「まあ、セキュリティはそこまで厳しくなくていいかな」をAIが「セキュリティ要件なし」と要約すれば、ニュアンスの喪失が後続の設計判断を歪めます。原文を残すことで、セッションを跨いだ際にもユーザーの発言を正確に参照でき、AIによる「記憶の再構築」 ── これもハルシネーションの一形態 ── を防止します。

### 追記専用ルール

audit.mdは追記専用 (append-only) です。既存の内容を上書きすることは禁止されています。

```markdown
- **CRITICAL**: ALWAYS append changes to EDIT audit.md file,
  NEVER use tools and commands that completely overwrite its contents
```

これは、AIが過去の記録を「修正」する形でハルシネーションを混入させることを防ぐ設計です。過去のステージでの議論を「こう言っていたはずだ」と書き換えることは、ハルシネーションの事後混入に他なりません。

ただし、audit.mdは通常のMarkdownファイルであり、技術的な改ざん耐性はありません。追記専用ルールはプロンプト指示による規定です。

## メカニズム5: コンテンツ検証とエラーハンドリング ― 出力の事前検証

### ファイル作成前のバリデーション

`content-validation.md`は、AIが生成する成果物の**出力時のバリデーション**を規定しています:

```markdown
**CRITICAL**: All generated content MUST be validated before writing to files
to prevent parsing errors.
```

Mermaid図のシンタックスチェック、ASCII図の文字幅統一、Markdownの構文検証が義務付けられています。バリデーションに失敗した場合はテキストベースのフォールバックに切り替えます。

ハルシネーション防止の文脈では、これは「**AIが構文的に不正な成果物を生成し、それが後続ステージで誤って解釈されること**」を防ぐメカニズムです。壊れたMermaid図を含むワークフロー計画がCode Generationステージに渡された場合、AIがその壊れた図を「解釈」しようとしてハルシネーションを起こすリスクがあります。

### エラーハンドリングにおける「先に進まない」ルール

`error-handling.md`は、矛盾や曖昧さを検出した際の対処として、複数の箇所で「**Do Not Proceed**」を明記しています:

| エラー種別 | 対処 |
|-----------|------|
| ユーザーが矛盾する要件を提示 | フォローアップ質問で矛盾を解消するまで先に進まない |
| 必須質問への回答が不完全 | すべての必須質問に回答されるまで先に進まない |
| ユーザーのストーリー計画回答が曖昧 | 具体例付きのフォローアップ質問を追加し、曖昧さが解消されるまで先に進まない |
| アーキテクチャ決定が不明確または矛盾 | 決定が明確かつ文書化されるまで先に進まない |
| 技術スタック選択の互換性がない | 互換性のある選択がなされるまで先に進まない |

この「Do Not Proceed」パターンは、ハルシネーション防止の最も直接的な表現です。AIが「不確実な情報を推測で補完して先に進む」こと ── まさにハルシネーションの定義そのもの ── を禁止しています。

## 5つのメカニズムの連携

これらの5つのメカニズムは独立して機能するのではなく、**多層防御 (Defense in Depth)** として連携しています。

```
ハルシネーションの発生を抑制する層:
  [メカニズム1] 過信防止 ― 「推測せず質問する」姿勢
  [メカニズム2] 構造化Q&A + 矛盾検出 ― 入力の曖昧さを排除

ハルシネーションの波及を食い止める層:
  [メカニズム3] 承認ゲート ― 各ステージの出力を人間が検証
  [メカニズム5] コンテンツ検証 ― 出力の構文的正当性を担保

ハルシネーションの事後検証を可能にする層:
  [メカニズム4] 監査証跡 ― 原文記録による検証可能性の確保
```

1つの層が突破されても、次の層で検出・修正できる設計です。Requirements AnalysisでAIが不十分な質問しかしなかったとしても (メカニズム1の失敗) 、矛盾検出で論理的整合性がチェックされ (メカニズム2) 、承認ゲートでユーザーが検証し (メカニズム3) 、audit.mdに原文が記録されているため後から検証できます (メカニズム4) 。

## 限界と留意点

### すべてはプロンプト指示である

最も根本的な限界は、5つのメカニズムのすべてがMarkdownファイルのプロンプト指示であるという点です。技術的な強制力はAIモデルのプロンプト遵守能力に依存します。「Do Not Proceed」と指示しても、AIがその指示に従う技術的保証はありません。

### 人間側の限界

承認ゲートの有効性はユーザーの注意力に依存します。12ステージの承認を繰り返す中で承認疲れが発生すれば、ハルシネーションを含む成果物がすり抜ける可能性があります。

### ハルシネーション検出の困難さ

最も厄介なのは、ハルシネーションが**もっともらしい**ことです。AIが生成した要件や設計が「間違っている」ことに気づくには、ユーザーがドメイン知識を持っている必要があります。承認ゲートは「ユーザーが検証できること」を前提としていますが、検証能力自体はフレームワークの外にあります。

### 質問過多のリスク

過信防止ガイドの「聞きすぎるくらい聞け」という方針は、質問が過多になるとユーザーの集中力を低下させ、かえって回答の質を下げるリスクがあります。Adaptive Depthが緩和策として機能しますが、質問量と回答品質のバランスは実際の運用で注意が必要です。

## まとめ

AI-DLCワークフローのハルシネーション防止メカニズムについて、横断的に整理しました。

- AI-DLCは、単一の技術ではなく**5つのメカニズムの多層防御**でハルシネーションに対処しています。過信防止と構造化Q&Aが「発生」を抑制し、承認ゲートとコンテンツ検証が「波及」を食い止め、監査証跡が「事後検証」を可能にします
- `overconfidence-prevention.md`は実運用で発覚した問題への事後対応であり、プロンプト指示によるAI制御には試行錯誤が伴うことを示す実例です
- 質問ファイル方式と矛盾検出は、AIに「推測させない」ための構造的な仕組みです。選択式フォーマットがAIの解釈余地を最小化し、矛盾検出が論理的整合性を担保します
- 承認ゲートの「NO EMERGENT BEHAVIOR」ルールは、承認プロセス自体のハルシネーションを防ぐ設計です
- 監査証跡の要約禁止・追記専用ルールは、AIによる「記憶の再構築」というハルシネーションの一形態を防止します
- これらすべてがプロンプト指示であり、技術的な強制力はAIモデルの遵守能力に依存するという根本的な限界があります。フレームワークは「ハルシネーションを不可能にする」のではなく、「ハルシネーションが発生しにくく、発生しても波及しにくく、事後に検証可能な構造」を提供しています

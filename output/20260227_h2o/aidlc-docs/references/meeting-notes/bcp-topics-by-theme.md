## 3. 技術的決定事項

### インフラアーキテクチャ

#### 復旧方式
- **採択方式: バックアップ＆リストア（コールドスタンバイ）**
- 理由: ウォームスタンバイはランニングコストが高く、H2OのコストへのこだわりからB&Rを選択
- 平時: バックアップデータのみ大阪Rに同期（EC2はAMI、RDSはSnapshot）
- 復旧時: IaC（Terraform）を使用してバックアップデータからシステムを復旧

#### 地理的配置
- **本番**: 東京リージョン（ap-northeast-1）
- **DR**: 大阪リージョン（ap-northeast-3）
- **アカウント**: 東京と大阪は別AWSアカウント（別Organization推奨）

#### バックアップ設計

| データ種別 | バックアップ方式 | 保持期間 | 変遷 |
|-----------|--------------|---------|------|
| EC2/EBS | Step Functions + AWS Backup → AMI | 90日 | 当初3日、延長 |
| RDS | RDS Backup → Snapshot → 大阪Replication | 90日 | 当初3日、延長 |
| FSx | AWS Backup CRC（実現性確認要） | 90日 | 方式確定前 |
| S3 | CRR（MDware業務データ 2バケット） | Versioning | 確定 |

#### KMS設計（重大な技術制約）
- **発見日**: 2026-02-05
- **制約**: AWSマネージドキーはクロスアカウント共有不可
- **対応**: カスタマーキー（CMK）を作成してクロスアカウント共有
- **設計方針**:
  - 東京R: `prd-dr-copy-cmk`（スナップショットコピー前の再暗号化用）
  - 大阪R: `dr-common-cmk`（全復旧リソースの暗号化用）
  - RDS自動スナップショットはクロスアカウント共有不可 → **手動スナップショット＋CMK共有**が必要

#### ネットワーク（1stステップ）
- **接続方式**: 東京RのDXGWから大阪RにAWS内ネットワークで接続（VGW Association）
- **DirectConnect切替**: 2月17日予定 → 3月5日に延期
- **2ndステップ（スコープ外）**: H2Oネットワークとの専用線接続は2ndステップ以降
- Transit Gateway: 竹谷さん（モダナイズプロジェクト）管轄、作成要否の決定が必要

#### IaCツール
- **確定: Terraform（HCL形式）**
- Remote State: S3バックエンド
- モジュール構成: network / security / backup / compute / monitoring

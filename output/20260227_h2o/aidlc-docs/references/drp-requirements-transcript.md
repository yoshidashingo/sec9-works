## 1. DRP検討の背景と狙い

### 1.1 DRP対策検討の前提となる脅威・リスク

他社で起きている甚大なセキュリティインシデントからの教訓を踏まえ、H2Oにおいて、サイバー攻撃に備えた今後のセキュリティ強化の方向性を検討する。

| 項目 | 内容 |
|------|------|
| **DRP対策検討において前提とする脅威・リスク** | ランサムウェアによるサイバーセキュリティリスク |
| ※ 備考 | 26年1月時点、DRP検討の前提とする脅威・リスクは、サイバーセキュリティとし、自然災害や外部依存リスク、大規模システム障害（AWS東京リージョン障害など）などはスコープ外とする |
| **想定される感染経路と感染方法** | - 店舗PCが標的型攻撃を受けた<br>- フィッシングサイトへのアクセスを通じて、マルウェアに感染（手動展開型ランサムウェア）<br>- マルウェアに感染した端末より侵入を許し、攻撃された |
| **想定される被害内容** | - システム停止による業務の停止<br>- 外部への情報流出（取引先ごとの商品の原価と売価、店舗ごとの売上情報などの経営情報など） |
| **他社事例からの教訓** | 以下の状況を想定した準備が必要<br>- サイバー攻撃によるシステム停止を想定した代替運用方法への切替が行えなかった<br>- 安全に復旧する環境の構築が存在しなかったために、業務再開が大幅に遅れた |

### 1.2 DRP対策の目指す姿

他社セキュリティインシデント事例を踏まえ、最低限実施すべきことは、短期間での業務再開・復旧を実現するための準備を、業務側とIT側が連携して実施することである（以下、「1STステップ」と表記）。

**1STステップで目指す姿:**

| 時間軸 | 業務側が主体となり検討すること | IT側が主体となり検討すること |
|--------|-------------------------------|------------------------------|
| **被害検知・初動** | 現場混乱を防ぐ<br>- 情報の集約化や指示命令系統などを決める<br>- **★業務の一時停止/継続判断基準を決める** | 被害拡大を止める<br>- **★被害検知後の隔離措置（ネットワーク遮断等）**<br>- 侵入システムの特定（暫定）<br>- **★経営・業務部門への一次報告（被害内容・範囲・業務影響）** |
| **状況把握・暫定継続** | 業務を継続する<br>- **★代替手段/縮退運用を決める**<br>- 優先業務への人員集中、一部業務の停止・縮小の判断<br>- 社内向け/顧客/取引先への説明方針 | 全体を把握する<br>- 被害範囲の調査、横断感染や潜在有無の確認<br>- **★バックアップの健全性確認**<br>- 外部専門家との連携（CSIRT/フォレンジック）<br>- **★復旧オプション整理（再構築/DRサイト等）** |
| **復旧判断・準備** | 通常復帰の判断準備<br>- **★通常業務復帰の条件整理**<br>- **★後追い処理（手動分）の整理計画**<br>- 復旧タイミングの合意 | 復旧準備<br>- **★復旧方針の確定（再構築 or バックアップサイトへの切替）**<br>- 復旧順序の決定<br>- クリーン環境の構築、復旧リハーサル or 検証<br>- 業務部門への復旧スケジュール提示 |
| **IT復旧・業務復旧** | 通常業務へ戻す<br>- 通常業務への切替指示<br>- 手動・暫定処理の解消<br>- 顧客・取引先への復旧連絡<br>- 影響確認 | 復旧実施<br>- **★システム再構築（バックアップサイトへの切替含む）とデータ復旧**<br>- セキュリティ検証と監視強化体制で稼働<br>- 段階的に業務へ開放 |
| **終結判断・宣言** | 終結宣言<br>- 事業継続が通常水準に回復<br>- ITからの安全性確認を受領<br>- 経営による終結宣言 | 終結条件の提示<br>- セキュリティ侵害痕跡なしの確認<br>- システムの正常稼働を確認<br>- 再発防止策の方向性確定 |

> ★ は 1STステップで実施する範囲を示す

**1STステップで目指す姿（要約）:**
- **業務側**: システムが停止した際に、どの業務を優先し、どのような代替手段で（最低限の）業務継続を叶えるのか、その判断基準と手順を予めルール化し、代替プロセスを整備する
- **IT側**: 現在は東京にしか存在しない重要システムの「予備の環境（バックアップサイト）」を、大阪に構築し、万が一に備える（※復旧時に本番サイトへ戻すことも含む）

### 1.3 DRP対策の対象となるインシデント

システムの停止が業務の停止に直結することより、システム停止の原因となる脅威・リスクを整理。1stステップにおけるDRP対策の対象となるインシデントはサイバーセキュリティリスクとする。

| 脅威・リスク | 想定事象（一例） | 対策（一例） |
|-------------|----------------|-------------|
| 自然・物理的リスク | 地震・水害・火災、広域停電、データセンター・オフィスの被災 | マルチリージョン、マルチクラウドの構築、DRサイトの構築、クラウド側で発電＋手動運用 |
| システム・運用リスク | ハードウェア/OS/ミドルウェア障害、クラウドリージョン障害、ネットワーク・回線障害、運用ミス・設定不備、バックアップ未検証 | SLAの見直し、冗長化設計、運用の標準化、バックアップデータの保護、定期的な訓練 |
| 外部依存リスク | クラウド事業者の障害、決済代行会社・SaaSの停止、物流・EDI連携先の停止、外注・委託先の業務停止 | SLAの見直し、手動運用への切替 |
| 人的・組織的リスク | キーマン不在・属人化、障害時の指示系統不明確、夜間・休日の対応体制不足、教育・訓練不足 | 標準化、定期的な訓練 |
| **サイバーセキュリティリスク（★対象）** | ランサムウェアによるシステム停止・暗号化、不正アクセス・情報漏えい、ID/認証基盤侵害、バックアップデータ破壊・改ざん、内部不正・委託先経由侵入 | EDR、隔離復旧、DRサイトの構築、セキュリティ教育、バックアップデータの保護、定期的な訓練、標準化 |

## 2. 想定インシデントと想定する被害内容・影響範囲

### 2.1 セキュリティインシデント発生のシナリオ想定

店舗/業務端末がフィッシングサイト経由でマルウェアに感染。その後、感染した端末から重要システムを攻撃するパターンを想定（Qilin同様の手動展開型ランサムウェア）。

| ステップ | 攻撃の流れ |
|---------|-----------|
| 1. **初期感染** | 任意のユーザPCにフィッシングメールが届き、誤って開封後、フィッシングサイトへアクセスし、ユーザPCが感染 |
| 2. **展開準備・内部偵察** | IPアドレス、ポートスキャン、リモート接続の調査や資格情報収集が行われる。管理サーバ、ファイルサーバ、認証サーバ、バックアップを探す（乗っ取りや破壊のため） |
| 3. **侵入情報収集** | OS/アプリケーション、VPN装置の脆弱性を悪用した侵入試行。管理者権限の奪取 |
| 4. **攻撃** | MDwareが応答しなくなる。不審なAPI呼び出し（バックアップ、スナップショット削除の試行）が増加 |
| 5. **情報窃取** | 外部への情報漏洩 |
| 6. **破壊・サービス停止（発覚）** | ファイルサーバのデータが暗号化、アプリケーションが動作不能 |
| 7. **脅迫行為** | サーバ画面に脅迫文が表示 |

### 2.2 セキュリティインシデントによる影響範囲、被害内容想定

**想定被害範囲**: 店舗/業務端末、重要システムのサーバ（EC2）を想定とする。
**前提**: 連携システムや連携するIF、バックアップのデータは感染していない前提とする。

被害対象コンポーネント:
- EC2（アプリケーションサーバ）
- RDS（データベースサーバ）
- ストレージ（OSからマウントしているストレージ）

被害対象外（正常稼働継続前提）:
- 連携システム（社外・社内）
- バックアップデータ

## 3. 復旧の考え方（DRP）※1stステップのDRP要件

### 3.1 復旧構成概要

**DRサイト**: AWS大阪リージョン（以降、大阪R）

**平時**: 東京リージョン（以降、東京R）で稼働中のシステム日次バックアップデータを大阪Rに複製する。被害発覚後、事業側からのシステム復旧指示をトリガーに、大阪Rのバックアップデータを用いてシステムを復旧。

**1stステップのゴール:**
- 定期的に東京→大阪へのバックアップデータを複製
- バックアップデータ容量とネットワーク通信量のみ発生（平時コスト最小化）
- ネットワークの冗長化は次ステップ検討

**システム復旧時:**
- バックアップデータを用いてリストア（EC2, RDS）
- EC2, RDS以外のサービスは、事前に作成する or 当日に作成する
- 運用・監視環境も復元する
- システム復旧後、手動運用期間中のデータの後追い処理を実行
- 大阪から東京へ戻す準備を開始（大阪→東京に複製）

**対象システム**: 百貨店/KFMの重要システム

### 3.2 復旧前後のアクセス方法

**#ロケーション**: 復旧先（DRサイト）は大阪RのVPCとする。

**システムアクセス経路（復旧前・通常時）**:
百貨店/KFMへのシステムアクセス経路は以下の3経路:
1. インターネット経由
2. H2Oネットワーク経由
3. H2O AWS内

**#H2OネットワークとAWS大阪Rとの接続:**

| ステップ | 内容 |
|---------|------|
| **1stステップ（26年6月末）** | 東京RのDXGWから大阪RにAWS内ネットワークを使用して接続する（大阪R側にVGWを作成しDXGWにAssociate） |
| 2ndステップ以降 | H2Oネットワークと大阪Rを専用線で接続する計画（26年2月時点、構成検討中）。1stのスコープ外 |

**#復旧前（通常時）のアクセス経路:**
- KFM MDware・百貨店POSは東京Rで稼働
- インターネット経由、H2Oネットワーク経由、H2O AWS内の3経路でアクセス

**#復旧後のアクセス経路:**
- システムを大阪Rで復旧したことに伴い、アクセス経路が変更
- 連携システムは東京Rで稼働を続けているため、重要システム（大阪R）→連携システム（東京R）へアクセスする

### 3.3 復旧方法（使用するデータ、実施方法）

**前提**: 「東京Rで稼働中の百貨店/KFMの重要システムがサイバー攻撃を受けたが、重要システムと連携するシステムは無事である」前提とし、大阪Rにて百貨店/KFMの重要システムを東京Rと全く同じ構成でシステムを復旧する。

**#システム復旧方式**: バックアップ&リストア方式

DR方式の比較（採用方式を太字で示す）:

| 方式 | 平時コスト | RTO | RPO |
|------|-----------|-----|-----|
| マルチサイト Active-Active | 最高（同スペック・同構成のランニングコストが2倍） | 最短 | 最終データ同期タイミング |
| ウォームスタンバイ | 低スペックサーバ起動状態のためランニングコスト増 | 短 | 最終データ同期タイミング |
| パイロットライト | 低スペックDBサーバ起動状態のためランニングコスト増 | 中 | 最終データ同期タイミング |
| バックアップ&リストア (Cold Standby併用) | ストレージとバックアップデータのみ（バックアップ&リストアよりはコスト高） | 長め | 最終バックアップ保存タイミング |
| **バックアップ&リストア（採用）** | **バックアップデータのみのため、ランニングコストは微増** | **最長** | **最終バックアップ保存タイミング** |

**#システム復旧の流れ:**

現行の東京Rで取得しているバックアップデータを大阪Rへ複製。事業側からのシステム復旧指示をトリガーに、IaC（Terraform使用想定）を用いてシステム復旧を行う。

| リソース | 東京R側処理 | 大阪R側処理 |
|---------|-----------|-----------|
| **EC2** | EventBridgeよりStep Functions実行→バックアップ実行前処理→AWS Backup実行（AMI取得）→バックアップ実行後処理→大阪RへReplication | 事業側からシステム復旧指示→東京RからReplication→IaCを使い、ReplicationされたAMIより復元→アプリ側の復旧作業→事業側による業務確認 |
| **RDS** | RDS Backupを実行（Snapshot取得）→大阪RへReplication | 事業側からシステム復旧指示→東京RからReplication→IaCを使い、ReplicationされたSnapshotより復元 |
| **ストレージ** | ストレージデータは東京Rを参照・書き込みに行く構成がよいか、大阪RへReplicationする必要があるかについては設計フェーズにて検討する（業務側の要件はないため、AWS側の仕様で検討） |  |
| **その他** | データを持たないサービス（NW、Security、運用管理）の構築について、大阪Rに事前にサービスを構築するか、事業側からシステム復旧指示が出た際にサービスを構築するかは設計フェーズで検討する |  |

### 3.4 復旧における役割分担と範囲

システム復旧のタスクは「インフラ復旧」「アプリ復旧」「業務確認」の3つ。復旧作業の順番は「インフラ復旧」→「アプリ復旧」→「業務確認」の順。

| 復旧作業 | 実施内容・役割 | 担当 |
|---------|--------------|------|
| **インフラ復旧** | - 事業側からシステム復旧指示が出た後、指定された目標復旧地点（RPO）のバックアップデータを用いて、システムを復元<br>- システムへのアクセス経路、セキュリティ機能、運用機能を復元する | ACN IOマルチクラウド運用チーム |
| **アプリ復旧** | - インフラ復旧完了後、アプリ側の設定変更を実施（IPアドレス/FQDN、参照先変更等）<br>- アプリの正常起動を確認する | アプリベンダー、連携システムベンダー |
| **業務確認** | - インフラおよびアプリの復旧完了後、業務再開が可能であることを判断<br>- 業務再開宣言をする | 事業部門、事業IT、アプリベンダー、連携システムベンダー |

**#復旧フロー（KFM MDwareユースケース）:**

| フェーズ | 東京側 | 大阪側 | 担当 |
|---------|--------|--------|------|
| **平時** | 日次バックアップ → バックアップデータ | 日次バックアップ完了後、大阪リージョンにコピー → バックアップ（コピー） | — |
| **インシデント発生後** | 代替運用期間 | — | — |
| **インフラ復旧（復旧指示後）** | — | 大阪リージョンにMDwareを動かすために必要なサービスをセットアップ（ロードバランサー、ネットワーク・DNS、セキュリティ機能、運用機能）→ 大阪リージョンにあるバックアップデータを用いてシステムを復元 | ACN IOマルチクラウド運用チーム |
| **アプリ復旧** | — | OSレイヤ以上の設定変更を実施<br>MDware（Web/DB、帳票、JP1、Bat）:<br>- 参照先データベースの変更<br>- プロパティファイルの再ロード<br>- サーバIP変更に伴う修正作業<br>- Tomcat再起動<br>- 後追い処理<br><br>WorkSpaces/CyberArkからの接続:<br>- 接続先変更 | アプリベンダー、連携システムベンダー |
| **業務復旧（システム復旧宣言）** | — | 業務再開に向けた動作確認:<br>- 機能が正常に動作すること<br>- 周辺システムとの連携が可能であること<br>- 業務再開可否判定<br>- システム復旧宣言 | 事業部門、事業IT、アプリベンダー、連携システムベンダー |

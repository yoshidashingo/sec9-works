## Intent Analysis

- **User Request**: H2O Retailing向けDRP基盤構築の再見積もり。RFP（DRP要件定義書v0.3、2026/2/24）に対してsec9として提案回答を本日（2026/2/27）17時までに提出する
- **Request Type**: Brownfield — 既存設計（v3.0）を活用した提案・見積もり作成
- **Scope**: MDware（EC2 27台、RDS 4DB） + 百貨店POS（EC2 45台、RDS 2DB、FSx 20.8TB）の大阪DR基盤全体

## 1. プロジェクト概要

### 1.1 背景・目的

ランサムウェアに代表されるサイバー攻撃により、百貨店/KFMの重要システムが停止するリスクに備え、安全にシステムを復旧し業務の再開を可能にするDRP基盤をAWS大阪リージョンに構築する。

### 1.2 スコープ（RFP p.30-31より）

提案依頼範囲は以下の4フェーズ：
1. **基盤設計**: 現行環境の調査 + DRP環境の設計（DRアーキテクチャ + IaC設計書）
2. **基盤テスト計画**: DRテスト方針策定 + テストシナリオ・手順・判定基準の作成
3. **基盤構築**: 設計に基づくDRP環境構築 + IaC実行環境 + IaCコード + IaCコード操作手順書
4. **基盤テスト**: DRP対象システムの復旧テスト + テスト結果報告書

**期間**: 26年3月中旬〜6月末（6月末までに基盤構築・基盤テストの完了が必須）

## 3. 機能要件（RFPレイヤ要件 p.39-45より）

### 3.1 バックアップ・データ保護

| 要件 | 内容 |
|------|------|
| バックアップ保管日数 | 90日（東京・大阪共通）← 現行3日から変更 |
| EC2バックアップ | Step Functions + AWS Backup → AMI取得 → 大阪へReplication |
| RDSバックアップ | RDS Backup → Snapshot取得 → 大阪へReplication |
| FSxバックアップ | AWS Backupのクロスリージョンコピー（実現性確認要） |
| S3 | 東京Rのサービスを継続利用するか大阪R用に作るかは基盤設計で検討 |
| EFS | 同上 |
| バックアップ実行 | Step Functions（スケジュールトリガー: EventBridge） |
| 東京Step Functions修正 | 大阪Backup VaultへのReplication設定追加 + 保管日数変更 |

### 3.2 ネットワーク（1stステップ）

| 要件 | 内容 |
|------|------|
| 大阪R接続方式 | 東京RのDXGWから大阪RにAWS内ネットワークで接続 |
| 大阪VPC | DR-PRD-VPC（MDware用）+ DR-POS-VPC（POS用）の2VPC |
| インターネット接続 | Internet GW、NAT GW（事前作成 or 復旧時作成かは基盤設計で検討） |
| 社内NW接続 | 2ndステップ以降（1stステップのスコープ外） |

### 3.3 セキュリティ

| 要件 | 内容 |
|------|------|
| クラウドセキュリティ | CloudTrail、各種ログ（VPC Flow log、ALB log、S3 log等）※ 現行設定を踏襲 |
| サーバセキュリティ | Security Group、Instance Profile ※ 現行設定を踏襲 |
| ネットワークセキュリティ | WAF、Shield、Network Firewall ※ 事前作成 or 復旧時作成かは基盤設計で検討 |
| 鍵管理 | Certificate Manager、KMS ※ KMS設計は基盤設計で検討 |
| 認証・認可 | IAM、Cognito（KFMのみ）、Managed AD（百貨店のみ、東京継続利用） |

### 3.4 運用管理

| 要件 | 内容 |
|------|------|
| ジョブ管理 | Step Functions（百貨店/KFM重要システムに関するもの） ※ 現行設定を踏襲 |
| ログ管理 | CloudWatch ※ 現行設定を踏襲。S3は基盤設計で検討 |
| 監視 | Zabbix→CloudWatch ※ 現行設定を踏襲 |
| バックアップ | AWS Backup + 大阪VaultへのReplication設定 + 保管日数変更 |

### 3.5 復旧方式（RFP p.16より）

**DR方式: バックアップ&リストア方式**
- 平時: バックアップデータのみ大阪Rに同期（AMI/RDS Snapshot）
- 復旧時: IaC（Terraform）を使用してバックアップデータからシステムを復旧
- 大阪Rの平時ランニングコストを最小化（データストレージ + データ転送のみ）

## 4. 非機能要件

### 4.1 RPO・RTO（RFP p.21-23より）

| システム | RPO | RTO（インフラ復旧） | 備考 |
|---------|-----|----------------|------|
| KFM MDware | 90日以内（事業側指定） | 12時間 | 26年1月KFMにて試算 |
| 百貨店POS | 90日以内（事業側指定） | N/A（26年4月以降に検討） | 1日復旧を希望。現実性協議予定 |

### 4.2 テスト要件（RFP p.36より）

- 東京RのAMI/Snapshotを大阪Rに複製し、大阪Rで復元できることを確認
- ネットワーク疎通（Ping等）の確認まではスコープ
- アプリ復旧はスコープ外
- データ妥当性証明は不要（現在のバックアップデータで復元できる前提）

### 4.3 コスト方針（RFP p.35より）

- 大阪Rは平時の運用コストを抑える設計
- AWSサービスの仕様・制約で大阪Rに必須のものは除く
- DRP基盤設計にて「東京Rのサービスを継続使用するもの」「大阪Rで新たに構築するもの」を定義
- 大阪Rで構築するものは「事前作成」「システム復旧時に作成」を定義

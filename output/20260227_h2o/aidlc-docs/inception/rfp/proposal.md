## 1. 提案概要

### 1.1 提案の背景と位置づけ

ランサムウェアによるサイバーインシデントを想定し、百貨店POSおよびKFM MDwareのDRP基盤を**AWS大阪リージョン**に構築するプロジェクトに対して提案します。

#### 事前調査の実績と発見事項

セクションナインは、2026年1月〜2月にわたり、H2O様のAWS東京Rアカウント（AccountID: 594597672828）に対して詳細調査を実施しました。196台のEC2・30台のRDS・4台のFSx（合計117TB）・1,189個のStep Functionsという大規模環境の全体像を把握するとともに、以下のようなDRP基盤構築上の重要課題を発見・整理しています。

- **Lambda Python3.9 EOL問題**: 百貨店POSのLambda関数27件がPython3.9を使用しており、2026/3/9に関数更新がブロックされる。Phase 3開始前にバージョンアップが必要であることを特定
- **大阪Rインスタンスタイプ非対応**: 百貨店POSのRDS（db.z1d.6xlarge）およびEC2（m5zn.6xlarge）が大阪リージョンで提供されておらず、代替インスタンスタイプの選定・性能検証が必要であることを特定
- **EC2台数の乖離**: MDwareのEC2台数がRFP記載の27台に対してCLI実測では18台であり、設計上の正確な台数確定が必要であることを確認
- **Step Functionsの分類**: 1,189個のStep Functionsからインフラジョブ（バックアップ等）とアプリジョブを分類し、DRP対象のスコープを明確化
- **FSxバックアップ方式の技術課題**: FSx Windows File Server（20.8TB）のAWS Backup Cross-Region Copy（CRC）対応可否の確認が必要であることを特定
- **Cognitoユーザー情報のレプリケーション不可**: KFM MDwareで使用するCognitoのユーザー/パスワードは大阪Rに自動レプリケーションできず、DR発動時に再設定が必要であることを確認

#### 事前調査による設計・構築のスピードアップ

これらの課題は、約2か月にわたる調査と関係者との継続的な協議を通じて発見・精緻化してきたものです。この**調査実績と課題の事前整理により、本プロジェクトでは以下のスピードアップが可能**です。

- **Phase 1（基盤設計）の工数削減**: 環境全体を把握済みのため、追加調査は差分確認（EC2台数・インスタンスタイプ詳細等）に集中でき、ゼロからの調査に比べ大幅に工期を短縮
- **リスクの事前織り込み**: Lambda EOL・インスタンスタイプ非対応・FSxバックアップ方式等の技術課題をすでに特定しているため、設計フェーズで手戻りなく対処方針を決定可能
- **Phase 3（基盤構築）の確実な実行**: Step Functions分類やバックアップパイプラインの全体像を把握済みのため、構築時の想定外事象を最小化

#### AIエージェントやAI-DLCの活用によるプロジェクト推進

さらに、当社モダナイズチームでは**AI-DLC（AI-Driven Development Life Cycle）**を全案件に適用し、調査・設計・構築・テストの各フェーズでAIを活用した開発プロセスの実績を積み上げてきました。本プロジェクトにおいても、以下のようにAIエージェントやAI-DLCを全フェーズにわたって活用し、**安全にかつ抜け漏れなく、スピード感を持ちながらもレビューや検証に十分なリソースを割り当てながらプロジェクトを推進**可能になります。

- **調査・設計フェーズ**: AIによるAWSリソースインベントリの自動分析・分類、設計パターンの最適化提案、要件カバレッジの網羅性チェック
- **IaCコード作成**: AIアシスタントによるTerraformコードの生成・レビュー・セキュリティチェック、ベストプラクティスとの自動照合
- **テスト計画・実行**: テストシナリオの網羅性検証、テスト手順書の自動生成、テスト結果の整合性チェック
- **ドキュメント品質**: 設計書・手順書の一貫性チェック、要件定義書との突合せ確認による抜け漏れ防止

本提案はRFP（DRP要件定義書v0.3）の提案依頼内容に基づき、以下の3点を提示します：
- **本書（提案書）**: 実施内容・スケジュール・体制・工数
- **コスト見積書（別紙: cost.md）**: 基盤設計・基盤テスト計画・基盤構築・基盤テストの4項目 + 合計額
- **H2O様へのご依頼事項（別紙: requests-to-h2o.md）**: プロジェクト推進に必要な環境・ご協力事項

### 1.2 設計方針・前提条件（RFP抜粋）

本提案は、RFP（DRP要件定義書v0.3）に記載の以下の前提に基づいて設計・見積を行っています。

1. **対象脅威の限定**: 本DRP基盤は**ランサムウェア攻撃を受け、百貨店/KFMの重要システムが停止した際に、安全に復旧するための場を構築する**ことを目的とします。自然災害、外部依存リスク、大規模システム障害など、ランサムウェア以外の脅威に対する復旧環境は本提案の対象外です。
2. **同一構成での復旧**: 東京Rで稼働しているDRP対象システムを、大阪Rで**全く同じ構成で復旧する**ことを前提とします。新サービスの導入は行いません。
3. **構成変更のスコープ限定**: 現行AWSサービスの設定に関し、**DRP基盤構築のための構成変更はスコープ内**としますが、それ以外の課題解決（ベストプラクティスからの逸脱是正等）は**スコープ外**とします。本プロジェクトは短期間で大阪に安全な復旧環境を構築することを主目的とします。
4. **平時運用コストの最小化**: 大阪Rはシステム復旧のための環境であるため、**平時の運用コスト（AWS月額費用）を抑える前提**で設計します。AWSサービスの仕様・制約上、大阪Rに作成が必要なもの、およびバックアップデータのストレージ使用料やNW通信料等の必須コストはこの限りではありません。
5. **RPO（目標復旧地点）**: バックアップ&リストア方式を採用するため、**RPOは前日の日次バックアップ取得タイミングとなり、最大24時間のデータ損失が生じます**。インシデント発覚時に参照するスナップショットは「感染が確認される直前のスナップショット」を事業側と協議の上で選定します（単純に最新スナップショットを使用するのではなく、感染前の安全なデータポイントまで遡ることを前提とします）。
6. **RTO（目標復旧時間）**: KFM MDwareは**12時間以内**を目標とします。百貨店POSのRTOはRFP時点で未定義のため、2026年4月以降にH2O様社内検討結果を踏まえて確定・反映します（詳細はセクション10リスク表参照）。
7. **インフラ復旧とアプリ復旧の引き渡し条件**: 本プロジェクトが担当するインフラ復旧の完了条件は以下の通りとし、これをもってアプリベンダー・連携システムベンダーへ作業を引き渡します:（a）大阪R上でEC2インスタンスが起動しOSが正常稼働すること、（b）RDSがSnapshotよりリストア完了しエンドポイントに接続できること、（c）ネットワーク疎通確認（東京R連携先・インターネット・閉域網）が完了すること、（d）IaCコード操作手順書に基づくACN IO運用チームへの引き渡しリハーサルが完了すること。

## 9. セクションナインの提案優位性

### 9.1 事前調査による即戦力

2026年1〜2月の詳細調査で把握済みの情報（EC2 196台・RDS 30台・Step Functions 1,189台等）は、他社には存在しない資産です。**現行調査工数を約50%削減**し、その分を設計・構築の精度向上に充てます。

### 9.2 技術的実現可能性の前倒し確認

RFPで「要確認」とされているFSxバックアップ方式・インスタンスタイプ大阪対応について、**Phase 1開始直後（3月中旬〜下旬）にAWS担当SAへ技術照会**を実施します。不可の場合の代替案と工数影響を**4月中旬には提示**でき、6月末完了スケジュールへの影響を最小化します。

### 9.3 AI-DLCによる設計・IaC生成の効率化

AIDLC（AI-Driven Learning Cycle）ワークフローにより、設計書・Terraform HCLコードの初稿生成を効率化します。人的工数は**設計判断・レビュー・品質保証**に集中し、定型的な記述生成の工数を削減しています。本見積の工数にはこの効率化効果を織り込み済みです。

## 1. 見積前提条件

### 1.1 確定前提

1. **DRP対象システム範囲**
   KFM MDware（EC2 27台※・RDS 4DB約20.3TB）および百貨店POS（EC2 45台・RDS 2DB約13TB・FSx 20.8TB）を大阪リージョンに構築する。
   > ※ RFP記載値27台を採用。sec9の2026年1月CLI実測値は18台。差分9台は基盤設計フェーズ開始時に読み取り専用ユーザによる追加調査で確定する。確定値が27台であれば本見積の工数内で対応。20台以上になる場合も工数内で吸収できる設計としているが、実測18台との差が生じた理由の調査工数（約3人日）が別途発生する可能性がある。

2. **AWS IAMユーザの払い出し**
   Phase 1（基盤設計）開始時（2026年3月15日）までに読み取り専用IAMユーザを払い出しいただけることを前提とする。Phase 3（基盤構築）開始時（2026年5月8日）までに、大阪DRアカウントへの構築権限および東京Rアカウントへの修正権限（Step Functions修正・Backup Vault Replication設定に必要）を付与いただけることを前提とする。遅延した場合は、遅延期間に応じてスケジュールが後ろ倒しになる。

3. **IaCツール**
   Terraform（HCL形式）を使用する。Remote Stateの保存先はS3とする。IaC実行環境として東京RにAmazon WorkSpacesでDR専用端末を作成し、平時はOFF（停止状態）とする。DR発動時に同端末を起動し、Terraformを実行する前提とする。

4. **成果物言語**
   設計書・手順書はすべて日本語で作成する。

5. **バックアップ保管日数**
   東京Rのバックアップ保管日数を現行の3日から90日に変更し、かつ大阪Rの新規Backup Vaultに90日保持でReplicationを設定する。この変更は東京Rの既存Step Functions（EventBridgeトリガー）の修正で対応する。

6. **Cognito制約（アプリ改修の可能性）**
   KFM MDware用CognitoのUserPool（ユーザー情報）は大阪Rへのレプリケーションが不可。大阪R復旧時にユーザー/パスワードの再設定が必要で、この作業はアプリベンダー（事業側）のスコープとなる。インフラ側は大阪RへのCognito UserPool再作成のみを担当。

7. **Lambda Python3.9 EOL（重要な技術制約）**
   百貨店POSのLambda関数27件がPython3.9を使用。AWSは**2026/3/9に関数更新をブロック**する予定（廃止日: 2025/12/15済み、関数作成ブロック: 2026/2/3済み）。バージョンアップ（Python3.12以上）の実施はアプリベンダースコープだが、弊社から対象Lambda関数リストを提示する。**Phase 3（5月）の構築前にH2O様側でバージョンアップが完了していることが前提**。なお、インフラジョブとして使用しているLambda関数についても、更新ブロック適用後はAWSの制約により当社での関数作成・更新が不可能となるため、**バージョンアップが期限までに完了していない場合は当該Lambda関数を構築・テスト対象から除外する**。

8. **ネットワーク接続方式（1stステップ）**
   東京RのDXGWから大阪RにAWS内ネットワークを使用して接続する。具体的には大阪R側にVGWを作成し、既存DXGWにAssociateすることで実現する。**ただし、本ネットワーク接続方式の構築自体はGICTグループにて対応しているため、当社作業の対象外とする。** 当社はDRアーキテクチャ設計においてVGW/DXGW Associationの構成を設計し、GICT様の構築結果を前提としたVPC/Subnet/Route Table等の大阪R側NW基盤構築を担当する。社内NW（専用線）との接続は2ndステップ以降のスコープ外とする。

9. **大阪リージョンのバックアップ設定範囲と東京R切り戻しの取り扱い（制約事項）**
   RFP（DRP要件定義書P5）に「復旧時に本番サイトへ戻すことも含む」と記載されている。本見積では、東京Rと同等のバックアップ体制を大阪Rに構築するまでをスコープとする。具体的には、MDwareはStep FunctionsからAWS Backupを起動する既存方式、百貨店POSはSystemWalkerからAWS Backupを起動する既存方式を、それぞれ大阪R側にも設定する。**東京Rへの切り戻し（RFP P23参照：大阪R運用開始から約1年後を想定）に関する手順化・訓練は本PJの対象外**とし、2026年4月以降のフェーズで別途検討・協議する。

10. **RPO（目標復旧地点）と復旧データポイントの選定方針**
   バックアップ&リストア方式を採用するため、**RPO（Recovery Point Objective）は前日の日次バックアップ取得タイミングであり、最大24時間分のデータ損失が生じる**。インシデント発覚時に参照するスナップショットは「最新スナップショット」を自動的に使用するのではなく、感染が疑われる期間より前の**安全なスナップショット**まで遡ることを前提とする。どの時点のスナップショットを採用するかはH2O様IT部門・ACN IO運用チームがバックアップの健全性確認後に判断し、本PJで構築するIaCコードはSnapshot IDを動的に指定できる構造とする（§2.4 #17参照）。

11. **百貨店POSのRTO未定義（制約事項）**
   RFP（DRP要件定義書P24）において、百貨店POSのRTO（目標復旧時間）は要件未定義である。本見積ではKFM MDwareのRTO（12時間以内）を基準にテスト計画・判定基準を策定するが、百貨店POSのRTOについては**2026年4月以降のH2O様社内検討結果を踏まえ、対応の詳細（テスト判定基準・復旧手順の精緻化等）を別途協議する**こととする。検討結果によりPhase 3・4の工数に影響が生じる場合は変更契約の対象とする。協議のタイムラインはrequests-to-h2o.md #10参照。

### 1.2 リスク前提（確認済みリスクと工数変動の可能性）

> **この節はRFPの透明性要件に基づき記載する。見積額はベースシナリオで計算しており、リスク顕在化時は別途協議の上、変更契約を締結する。**

#### リスク①: POSインスタンスタイプの大阪リージョン非対応
- **対象**: `db.z1d.6xlarge`（POS RDS基盤）、`m5zn.6xlarge`（EC2 Bat-AP）
- **状況**: RFP記載通り「大阪対応要確認、事業側と協議中」
- **ベースシナリオ（本見積に含む）**: 代替インスタンスタイプの選定・性能検証は事業側のスコープとする。当社はH2O様・事業側にて決定された代替インスタンスタイプを受領し、IaCコードに反映する
- **リスク顕在化時の追加工数**: 代替インスタンスタイプの決定が遅延しPhase 3開始後にIaC修正が必要となった場合、修正・再テストで**+5人日（+625,000円）** を上限として変更契約対象とする（構築フェーズ単価125,000円/人日で算定）

#### リスク②: FSxバックアップのクロスリージョンコピー方式
- **対象**: POS FSx for Windows File Server（20.8TB）
- **状況**: AWS Backupの「FSx for Windows File Server」に対するクロスリージョンコピーは、2026年2月時点でサポート対象外の可能性がある（AWS公式ドキュメントでは「FSx for Lustre」「FSx for NetApp ONTAP」等は対応しているが、Windows File Serverの対応可否は設計フェーズ開始後にAWS担当SAに確認する）
- **ベースシナリオ（本見積に含む）**: AWS Backupによるクロスリージョンコピーが利用可能であることを前提とする
- **代替シナリオ（別途変更契約）**: AWS Backupが非対応の場合、以下のいずれかの代替方式を設計フェーズで選定する
  - 案A: AWS DataSyncによる定期転送（追加設計5人日×200K + 構築10人日×125K = **+15人日、+2,250,000円**）
  - 案B: PowerShellスクリプト＋AWS CLIによる手動コピー自動化（追加設計5人日×200K + 構築・テスト15人日×125K = **+20人日、+2,875,000円**）
  - > いずれの場合も設計フェーズ（Phase 1）内でAWSに確認し、月次で最終方式を決定する

## 5. 参考: 大阪R 平時ランニングコスト試算（スコープ外/参考情報）

**RFPにて「コスト試算はスコープ外」ですが、調査・試算済みのため参考情報として提示します。**

### 主要コスト要因の説明

1stステップ（バックアップ&リストア方式）では、大阪Rの平時コストは**バックアップデータのストレージ費用とデータ転送費用のみ**です。EC2やRDSのインスタンス費用は、DRP発動時のみ発生します。

#### 主要ストレージコスト（$0.095/GB/月）

| 対象 | データ量 | 90日保持の場合の増分コスト |
|------|---------|------------------------|
| MDware RDS Snapshot | 初期~20.3TB | 変動率×日数分が増分蓄積 |
| POS RDS Snapshot | 初期~13.0TB | 同上 |
| FSx バックアップ | ~20.8TB | 増分コピー（完全解明待ち） |
| S3 CRR（MDware業務データ） | ~6TB | CRRはGB単位の転送料$0.02/GB |

#### 変動率別月次コスト試算

変動率とは「前日比のデータ増分割合」です。ランサムウェア対策の観点で重要なのは、毎日の新規スナップショット分（増分）が90日分蓄積されることです。

| 変動率 | 月次ストレージコスト目安 | リージョン間転送コスト目安 | 月次合計目安 | 年間参考（JPY/130円換算） |

| **0.5%/日** | $3,000〜$5,000 | $1,000〜$2,000 | **$4,000〜$7,000** | 624〜1,092万円/年 |
| **1.0%/日** | $6,000〜$10,000 | $1,500〜$3,000 | **$7,500〜$13,000** | 1,170〜2,028万円/年 |
